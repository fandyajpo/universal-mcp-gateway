## Goal

Implement validation checks for generated embeddings to ensure vector quality, correctness, and consistency. The validator checks vector dimensionality matches the expected model dimensions, verifies L2 normalization, detects NaN or Infinity values, checks for zero vectors, and validates that the embedding preserves semantic similarity (basic sanity check). Invalid embeddings are logged and flagged but not discarded — downstream consumers can decide how to handle them.

## Business Context

Occasionally, embedding APIs return corrupted or invalid vectors due to transient issues, input text problems, or API bugs. Validating embeddings at generation time prevents silent data corruption in the vector store. Without validation, bad embeddings could pollute the search index and produce unreliable retrieval results, undermining user trust in the RAG pipeline.

## Technical Context

The validation runs immediately after each embedding API response, before the result is cached or stored. It's a lightweight, synchronous check that adds minimal overhead (< 1 ms per vector). Validation failures are logged with the document ID, chunk ID, and specific failure reason. The same validator is used when loading embeddings from cache to detect stale or corrupted cache entries.

## Requirements

### Functional
- Validate vector dimensionality matches expected model dimensionality (3072 for text-embedding-3-large, 1536 for voyage-code-2)
- Verify vector is L2-normalized (magnitude within 0.999-1.001 range)
- Check for NaN (Not a Number) values — any NaN makes the vector unusable
- Check for Infinity or -Infinity values
- Check for zero vectors (all values exactly 0) — indicates failed embedding
- Check for near-zero vectors (magnitude < 0.001) — indicates degenerate embedding
- Check for vectors where all values are identical (e.g., [0.5, 0.5, 0.5, ...]) — indicates API error
- Return validation result: `{ valid: boolean, issues: ValidationIssue[] }` where each issue has `{ type, severity, message }`
- Severity levels: `error` (vector is unusable), `warning` (vector may degrade retrieval quality)
- Log all validation issues at appropriate log levels (error for errors, warn for warnings)
- Flag invalid vectors in the embedded chunk metadata for downstream awareness
- Support configurable validation strictness: `strict` (fail on any issue), `normal` (log issues, continue), `permissive` (only fail on NaN/Infinity)

### Non-functional
- Validation overhead: < 1 ms per vector
- No additional API calls or external dependencies

### Performance
- Validation is single-threaded and synchronous

- Invalid vectors are NOT discarded by the validator — the caller decides how to handle them
- Validation strictness is configurable via `EMBEDDING_VALIDATION_STRICTNESS` env var (default `normal`)
- In `strict` mode, validation errors throw an exception that the caller should catch

## Notes

- L2 norm check: `const magnitude = Math.sqrt(vec.reduce((s, v) => s + v * v, 0)); return magnitude >= 0.999 && magnitude <= 1.001;`
- NaN check: `vec.some(v => Number.isNaN(v))`
- Infinity check: `vec.some(v => !Number.isFinite(v))`
- Zero check: `vec.every(v => v === 0)`
- Near-zero check: `magnitude < 0.001`
- All-identical check: `vec.every(v => v === vec[0])` where `vec[0] !== 0`
- In `normal` mode, errors are logged and metadata flagged; in `strict` mode, errors throw
