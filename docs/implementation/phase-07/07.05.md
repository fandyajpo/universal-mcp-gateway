## Goal

Implement cost tracking for embedding API calls, recording token usage and estimated cost per request, per workspace, per model. Cost records are written to MongoDB for billing (Phase 14) and workspace-level cost monitoring. The tracker must handle partial token pricing (OpenRouter charges per token), track both prompt tokens (the only cost for embeddings), and provide aggregated cost reports.

## Business Context

Embedding costs are a significant operational expense in production. Without cost tracking, there is no visibility into which workspaces are consuming embedding budget, which models are most expensive, or whether costs are growing as expected. Cost data feeds into the billing system (Phase 14) for metered billing and into operational dashboards for cost management.

## Technical Context

OpenRouter's embedding response includes `usage.prompt_tokens` ‚Äî the number of tokens processed. Each model has a per-token price (e.g., `text-embedding-3-large` costs $0.00013 per 1K tokens). The cost tracker multiplies token count by model price to estimate cost, records each embedding request as a cost entry in MongoDB, and provides aggregation queries for workspace-level cost reports.

## Requirements

### Functional
- Record cost entry for each embedding API call with: `workspaceId`, `documentId`, `model`, `promptTokens`, `cost`, `timestamp`, `requestId`
- Look up model pricing from a configurable price map (defaults built-in, overridable via env)
- Store cost records in MongoDB `costs` collection
- Support cost estimation when token count is known but API response is missing usage data (fallback: estimate tokens from text length)
- Provide `getWorkspaceEmbeddingCost(workspaceId, startDate, endDate)` aggregation
- Provide `getModelCostBreakdown(startDate, endDate)` aggregation across all workspaces
- Provide `getDailyEmbeddingCost(workspaceId, days)` for trend chart
- Track cumulative cost per document to identify expensive documents
- Log cost data as structured logs for real-time cost dashboards
- Handle zero-cost models gracefully (some models may be free)

### Non-functional
- Cost record write overhead: < 10 ms per record (async write, non-blocking)
- Cost query < 500 ms for monthly aggregation on datasets up to 1M records
- Cost records are immutable ‚Äî never updated, only inserted

### Performance
- Batch cost writes: buffer records in memory and flush every 10 seconds or 100 records
- MongoDB indexes: `{ workspaceId: 1, timestamp: -1 }`, `{ model: 1, timestamp: -1 }`

### Security
- Cost records are workspace-scoped ‚Äî cross-workspace aggregation requires admin role
- No cost data in client-facing API responses without proper authorization (Phase 14 handles billing)

### Accessibility
- N/A (internal cost tracking)

## Architecture Constraints

- Cost records are write-only (append-only) ‚Äî never updated or deleted
- The batch writer uses an interval-based flush to batch MongoDB writes
- Model pricing is configured in a TypeScript map (not MongoDB) for fast lookups
- Cost data feeds into billing (Phase 14) via workspaceId and date range queries
- Cost estimation fallback: `estimatedTokens = Math.ceil(text.length / 4)` for English text

## Acceptance Criteria
