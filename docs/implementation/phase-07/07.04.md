## Goal

Implement a caching layer for embeddings that stores previously computed vectors in Redis and serves them on cache hits, avoiding redundant and costly API calls. The cache uses an LRU (Least Recently Used) eviction strategy with configurable TTL, and the cache key is derived from the SHA-256 hash of the input text concatenated with the model name.

## Business Context

Many documents contain duplicate or overlapping content (e.g., same legal disclaimer in multiple contracts, repeated headers/footers). Without caching, these duplicate chunks would be re-embedded on every upload, wasting API credits and increasing latency. The embedding cache eliminates redundant API calls for identical text chunks, reducing costs by an estimated 20-40% for typical enterprise document sets.

## Technical Context

The cache integrates with the batch processing pipeline (Step 07.02) — before sending a chunk to the embedding API, the pipeline checks the cache. On cache hit, the stored vector is returned immediately. On cache miss, the chunk is queued for API processing, and the result is stored in the cache for future use. Redis is used as the cache store because it provides sub-millisecond reads, built-in TTL, and LRU eviction.

## Requirements

### Functional
- Cache key = `SHA-256(textContent + modelName)` as hex string
- Cache value = JSON-serialized `{ vector: number[], model: string, dimensions: number, createdAt: timestamp }`
- Configurable TTL: default 30 days, configurable via `EMBEDDING_CACHE_TTL` environment variable (seconds)
- LRU eviction: Redis `allkeys-lru` policy — least recently used entries are evicted when memory is full
- Cache hit returns cached embedding within < 5 ms (including deserialization)
- Cache miss returns `null` and caller proceeds to embedding API
- After successful API embedding, store result in cache with configured TTL
- Batch cache check: accept array of chunks, return array of `{ index, vector } | null` in parallel
- TTL extension: accessing a cached entry extends its TTL by the configured duration
- Cache invalidation: support clearing cache for a specific document ID (when document is re-uploaded)
- Cache size monitoring: expose cache size, hit rate, and miss rate metrics
- Graceful degradation: if Redis is unavailable, skip cache and proceed to API directly (log warning, not error)

### Non-functional
- Cache read throughput: > 10,000 reads per second (Redis handles this easily)
- Cache write throughput: > 5,000 writes per second
- Cache hit ratio target: > 30% for typical enterprise document workloads
- Storage: each embedding is ~50 KB (3072 dimensions * 4 bytes * JSON overhead) — plan for 50K entries = ~2.5 GB

### Performance
- Batch cache check uses Redis `MGET` for efficiency (single round trip)
- Cache writes use `SET` with TTL in single operation

### Security
- Cache keys are hashes — no plain text content stored in Redis keys
- Cache values contain only vectors, model name, and timestamps — no document content
- Tenant isolation: cache key prefix includes `{workspaceId}:` to prevent cross-tenant cache hits (optional — same text from different tenants could share cache if content is identical)

### Accessibility
- N/A (internal cache)

## Architecture Constraints

- Cache key prefix: `embedding:{workspaceId}:{hash}` for tenant isolation (optional, configurable via `EMBEDDING_CACHE_TENANT_PREFIX`)
- Redis memory limit should be configured (e.g., `maxmemory 4gb`) with `allkeys-lru` policy
- The cache is write-through: embeddings are always written to cache after successful API response
- TTL is extended on every cache hit (Redis `GET` + `EXPIRE` or use `GETEX` on Redis 6.2+)
- Serialization uses `Buffer.from(JSON.stringify(value))` for Redis string storage; deserialization uses `JSON.parse()`

## Acceptance Criteria

- Cache hit returns stored embedding within < 5 ms
- Cache miss returns null and caller proceeds with API call
- After API embedding, result is stored in cache with correct TTL
- Same text + same model returns cache hit on second request
- Same text + different model returns cache miss (different key)
- Cache TTL is configurable and honored (entry expires after TTL)
- Batch cache check returns correct results for mixed hit/miss input
- Cache invalidation clears all entries for a given document
- Redis unavailability causes graceful fallback (skip cache, log warning)
- Cache metrics exposed and readable

## Notes

- Use `crypto` module's `createHash('sha256')` for key generation
- Redis `MGET` for batch checks: `await redis.mget(keys)` returns array of values or nulls
- For TTL extension on hit, use `redis.getex(key, { ex: ttl })` on Redis 6.2+ or `redis.get(key)` + `redis.expire(key, ttl)`
- Consider using Redis pipeline for batch write operations to reduce round trips
- Cache metrics can be exposed via a simple counter object: `{ hits: 0, misses: 0, sets: 0, evictions: 0 }`
