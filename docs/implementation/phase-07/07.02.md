## Goal

Implement the batch processing pipeline that takes text chunks from the PDF chunker (Step 06.04), groups them into optimally-sized batches, sends them to the embedding service (Step 07.01), and stores the resulting embeddings in the vector store (preparation for Phase 08). The pipeline handles flow control, progress tracking, and error recovery for large document collections.

## Business Context

A single document can produce thousands of chunks. Processing them one by one would be impractically slow and expensive. Batch processing groups chunks into efficient batches (matching API limits), manages concurrency, and provides visibility into progress. This pipeline is the bridge between Phase 06 (chunking) and Phase 08 (vector store).

## Technical Context

The batch pipeline receives an array of `Chunk` objects, groups them into batches of configurable size (default 32), sends each batch to the embedding service, collects the results, and returns `EmbeddedChunk` objects (chunk + vector). It respects rate limits (Step 07.03), checks the cache (Step 07.04) to avoid redundant work, and reports progress. The pipeline runs as an Inngest step function in the PDF processing workflow.

## Requirements

### Functional
- Accept array of `Chunk` objects with text content and metadata
- Group chunks into batches of configurable size (default 32, max 64)
- Check embedding cache for each chunk before calling the API
- Send cache-miss batches to embedding service in parallel (max 5 concurrent batches)
- Collect `EmbeddedChunk` results (chunk + vector + model + token count)
- Handle partial batch failures — retry failed chunks individually, then fail batch if individual retries fail
- Report progress: emit `pdf/embed/started`, `pdf/embed/completed` events per document
- Report detailed metrics: total chunks, embedded chunks, cache hits, failed chunks, tokens used, elapsed time
- Support incremental embedding: skip chunks whose hash matches an existing embedding (from cache check)
- Return `EmbeddedChunk[]` sorted in original chunk order

### Non-functional
- Pipeline throughput: minimum 1000 chunks per minute for cache-miss batches
- Cache miss rate: first-time documents = 100% miss, re-processed = ~0% miss (if chunks unchanged)
- Backpressure: if embedding service queues grow > 500 chunks, pause and wait

### Performance
- Memory: hold maximum 1000 chunks in memory at once (process in pages of 1000 if larger)
- Batch parallelism: 5 concurrent embedding requests

### Security
- Chunk content is embedded — never logged or exposed
- Cache keys are content hashes — no content leakage via cache

### Accessibility
- N/A (internal processing pipeline)

## Architecture Constraints

- Batch size must be configurable via environment variable `EMBEDDING_BATCH_SIZE` (default 32)
- The pipeline must handle empty input (return empty array immediately)
- Chunks are processed in document order (sorted by page number, then position)
- If a batch fails entirely after individual retries, the pipeline stops and propagates the error
- Progress is reported per-document, not per-batch, to UI (Phase 06 progress tracking)

## Acceptance Criteria

- 1000 chunks are processed through batch pipeline in < 1 minute
- Cache hits skip API calls and return stored embeddings immediately (< 5 ms per hit)
- Partial batch failure retries individual chunks and succeeds on retry
- Full batch failure after retries propagates error
- Embed progress events emitted with correct document ID and counts
- Incremental embedding skips chunks with unchanged content
- Embedded chunks returned in original sort order
- Pipeline handles empty chunk array gracefully

## Notes

- Use `p-limit` or a simple semaphore pattern to limit concurrency to 5 batches
- Cache check for each chunk: `cache.get(hash(chunk.text + model))` — if exists, skip embedding
- Progress estimation: `currentChunks / totalChunks * 100` updated every 50 chunks
- Batch splitting: simple array chunking (`for (let i = 0; i < chunks.length; i += batchSize)`)
- For incremental embedding, compute chunk content hash and compare with stored hash in vector store
