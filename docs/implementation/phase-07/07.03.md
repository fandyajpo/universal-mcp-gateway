## Goal

Implement robust rate limit handling for the OpenRouter embedding API. The rate limiter must track token and request quotas, respect `Retry-After` headers, implement exponential backoff with jitter, queue pending requests during rate limit windows, and prevent the embedding service from exceeding API rate limits.

## Business Context

OpenRouter imposes rate limits per API key. Exceeding them results in 429 errors and temporary bans. The rate limiter ensures the embedding service stays within allowed limits while maximizing throughput. Without it, the embedding pipeline would be unreliable and frequently interrupted by rate limit errors.

## Technical Context

Rate limiting operates at two levels: (1) a local token-bucket algorithm that pre-emptively limits request rates to avoid hitting API limits, and (2) reactive handling of 429 responses with `Retry-After` header parsing. The rate limiter uses Redis (via `@repo/cache`) for distributed rate limit state across multiple workers. Configuration includes requests per minute (RPM) and tokens per minute (TPM) limits.

## Requirements

### Functional
- Implement token bucket algorithm with configurable capacity and refill rate
- Track two limits simultaneously: requests per minute (RPM) and tokens per minute (TPM)
- Read rate limit configuration from environment variables: `OPENROUTER_RPM` (default 3500), `OPENROUTER_TPM` (default 10M)
- Parse `Retry-After` header from 429 responses (supports both seconds integer and HTTP-date)
- Implement exponential backoff: base delay 1s, multiplier 2, max delay 60s, with jitter (±25%)
- Queue requests when rate limit is exceeded — process them as capacity becomes available
- Support priority queueing: embedding for interactive queries higher priority than batch processing
- Expose current rate limit state: remaining requests, remaining tokens, next reset time
- Log rate limit events: when throttled, when reset, backoff duration
- Circuit breaker: after 3 consecutive 429 responses within 5 minutes, pause all embedding for 60 seconds
- Support per-model rate limits (different models may have different limits)

### Non-functional
- Rate limiter overhead: < 1 ms per check (Redis is fast)
- Queue wait time: interactive priority requests wait < 100 ms before processing
- False 429 avoidance: stay within 80% of the API's rate limit (configured limit * 0.8)

### Performance
- Redis operations: 2 operations per embedding request (check + consume)
- Batch checking: check rate limit once per batch, not per individual request

### Security
- Rate limit keys are scoped to API key, not to any customer-specific identifier
- No leakage of rate limit state between tenants

### Accessibility
- N/A (internal rate limiting)

## Architecture Constraints

- Rate limit state stored in Redis with keys: `ratelimit:{model}:tokens`, `ratelimit:{model}:requests`, `ratelimit:{model}:reset`
- Token bucket refills at configured rate every second (not every request)
- The circuit breaker state is stored in Redis with a 5-minute TTL
- Priority queue: interactive (priority 1) vs batch (priority 2); interactive requests always processed first
- Rate limit configuration is per-environment, not per-tenant (multi-tenancy handled at workspace level)

## Acceptance Criteria

- Rate limiter blocks requests that would exceed RPM or TPM limits
- 429 response triggers backoff and retry with correct `Retry-After` parsing
- Queue drains pending requests as capacity becomes available
- Interactive priority requests are processed before batch requests
- Circuit breaker opens after 3 consecutive 429s and pauses for 60 seconds
- Rate limit state is correctly initialized from environment variables
- Overhead of rate limit check is < 1 ms per batch request
- Multiple workers share rate limit state via Redis (distributed)

## Notes

- Token bucket: `capacity = RPM / 60` tokens, refill `capacity / 60` tokens per second
- For Retry-After parsing, handle both formats: `Retry-After: 30` (seconds) and `Retry-After: Wed, 21 Oct 2024 07:28:00 GMT` (HTTP-date)
- Jitter formula: `delay * (0.75 + Math.random() * 0.5)` — random value between 75% and 125% of delay
- Circuit breaker state: `{ open: boolean, lastFailure: timestamp, failureCount: number }` stored in Redis
- For per-model limits, key prefix includes model name (e.g., `ratelimit:text-embedding-3-large:requests`)
