## Goal

Implement the core embedding service that generates vector embeddings from text chunks using OpenRouter's embedding API. The service wraps OpenRouter's embeddings endpoint, handles authentication, constructs requests, parses responses, and returns normalized vectors. It supports multiple embedding models with configurable dimensions and provides a unified interface for the rest of the platform.

## Business Context

Embeddings are the foundation of semantic search and RAG. Every text chunk from Phase 06 must be converted to a vector representation. The embedding service abstracts away the API details so that the rest of the platform (chunking, vector store, retrieval) can work with a clean interface: `text in → vector out`.

## Technical Context

OpenRouter provides a unified API for multiple embedding models. The primary model is OpenAI's `text-embedding-3-large` (3072 dimensions). The service must handle authentication via `OPENROUTER_API_KEY`, construct proper request bodies with model name and input text, handle response parsing including model metadata, and return normalized vectors. The service lives in `@repo/rag` but uses the OpenRouter client infrastructure from `@repo/ai` (or directly, since embedding is a separate API surface).

## Requirements

### Functional
- Generate embedding vector from a single text string (up to 8192 tokens)
- Generate embedding vectors from a batch of text strings (up to 32 texts per request)
- Support primary model: `text-embedding-3-large` (3072 dimensions)
- Support fallback model: `voyage-code-2` (1536 dimensions)
- Support customizable dimensions (via `dimensions` parameter) for primary model
- Return vectors as `Float64Array` (standard JS typed array)
- Normalize vectors to unit length (L2 normalization)
- Include metadata in response: model used, token count, dimensions, latency
- Handle API errors: 400 (invalid input), 401 (auth), 429 (rate limit), 500 (server error)
- Implement retry with exponential backoff: max 3 retries, base delay 1s, max delay 30s
- Implement request timeout: 30 seconds per request
- Truncate input texts that exceed the model's maximum token limit
- Log all API calls with text length, token count, latency, and status

### Non-functional
- Response time: single embedding < 500 ms p95 (text < 512 tokens)
- Response time: batch embedding (32 texts) < 3 seconds p95
- Availability: handle OpenRouter API downtime gracefully (queue and retry)

### Performance
- Maximum concurrency: 5 in-flight embedding requests per worker
- Batch sizing: optimize batch size based on text length (shorter texts → larger batches)

### Security
- API key managed via environment variable (never logged)
- Input texts logged at TRACE level only (not INFO or higher)
- No customer data in error messages returned to the caller

### Accessibility
- N/A (internal service)

## Architecture Constraints

- All embedding requests go through OpenRouter — never call OpenAI directly
- The `dimensions` parameter is only supported for `text-embedding-3-*` models; for fallback models, use native dimensions
- Input texts longer than model max tokens are truncated from the end with a warning log
- Vectors are always L2-normalized before return (unit length)
- The service is stateless — all state (rate limits, cache) lives in Redis

## Acceptance Criteria

- `embedText("Hello world")` returns a normalized vector of correct dimensionality
- `embedBatch(["text1", "text2", "text3"])` returns an array of vectors
- Batch size of 32 texts processes correctly
- Fallback model activated when primary model returns 4xx or 5xx
- Input text truncation works for texts exceeding token limit
- Retry logic executes on 429 and 5xx responses
- Request timeout kills hanging requests after 30 seconds
- Response metadata includes correct model, token count, and latency

## Notes

- OpenRouter embeddings endpoint: `POST https://openrouter.ai/api/v1/embeddings`
- Request body: `{ model: "text-embedding-3-large", input: "text" | string[], dimensions?: number }`
- Response: `{ data: [{ embedding: number[], index: number }], model: string, usage: { prompt_tokens: number } }`
- The `dimensions` parameter is unique to OpenAI models and is passed through OpenRouter
- For L2 normalization: `const norm = Math.sqrt(vec.reduce((s, v) => s + v * v, 0)); return vec.map(v => v / norm);`
