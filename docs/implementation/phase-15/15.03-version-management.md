## Goal

Implement version management for marketplace listings. Support semantic versioning, changelog tracking, version-specific manifests, compatibility declarations, update notifications, and rollback capability.

---

## Business Context

Connectors and tools evolve over time. New features are added, bugs are fixed, and API changes require version bumps. Users need to know when updates are available, what changed, and whether the update is compatible with their platform version. Publishers need to manage multiple versions, deprecate old versions, and force updates for security fixes.

---

## Technical Context

Each version is stored as a sub-document in the listing document or as a separate collection for version history. Versions follow strict semantic versioning (MAJOR.MINOR.PATCH). Each version includes: the version string, connector manifest (specific to this version), changelog (markdown), compatibility (minimum platform version, maximum platform version), release date, deprecation status, and security/hotfix flag.

## Requirements

### Functional requirements

- Create new version for a published listing with updated manifest and changelog
- Version format must enforce semantic versioning: MAJOR.MINOR.PATCH
- Validate that new version is greater than the previous version (no downgrade)
- Store full manifest per version (each version can have different tools)
- Track changelog per version (markdown, rendered in marketplace UI)
- Declare platform compatibility per version (min/max platform version)
- Support deprecation of specific versions with reason and alternative version recommendation
- Support security/hotfix flag — versions marked as security fixes are force-updated for all users
- Implement automatic update checking: API that returns latest compatible version for a given platform version
- Support version pinning: users can pin to a specific version (ignore updates)
- Provide version history view in marketplace UI and developer dashboard
- Support rolling back to previous version (creates a new version entry with reverted manifest)

### Non-functional requirements

- Version comparison must use proper semver comparison (not string comparison)
- Update checking must complete in under 50ms
- Version history loading must be paginated for listings with many versions

### Performance requirements

- Version creation: < 2s
- Update check: < 50ms
- Version history load: < 500ms for 100 versions

### Security requirements

- Only listing authors and marketplace admins can create versions
- Version manifests must be validated independently per version
- Force-update for security fixes must not break existing installations
- Deprecation reason must not include sensitive information

### Accessibility requirements

- Version changelog must be rendered with proper heading hierarchy
- Version comparison must be accessible via keyboard

## Architecture Constraints

- Versions are stored in a separate `marketplace_versions` collection (not embedded in the listing) to support large version histories
- Each version references the parent listing by `listingId`
- The listing document has a `latestVersion` field that is updated on each new version creation
- Semantic version comparison must use a dedicated semver library (e.g., `semver` npm package)
- Platform compatibility is expressed as semver ranges: `>=1.0.0 <2.0.0`

## Notes

The version creation flow should be simple enough that publishers can push updates quickly but structured enough to maintain a clear version history. Consider implementing an auto-increment suggestion for PATCH versions when the manifest changes but the tool interfaces remain the same. MAJOR version bumps should require explicit confirmation as they indicate breaking changes. The update checker should be called on workspace login and periodically to check for connector updates.
