## Goal

Implement the complete connector publishing flow: developer submits a connector for marketplace listing, the submission goes through a review process, and approved listings are published and made available for installation. The flow integrates with the connector registry from step 11.13.

---

## Business Context

The publishing flow is the primary developer experience for the marketplace. Developers need to submit their connectors with all required metadata, track the review status, respond to reviewer feedback, and ultimately publish their integration. The flow must be clear, guided, and transparent about the review process and timeline.

---

## Technical Context

The publishing flow consists of several stages: (1) developer submits connector metadata and manifest through a form, (2) the submission is stored as a draft, (3) developer submits for review, (4) marketplace admins review and either approve or reject, (5) approved listings are published and appear in the marketplace, (6) published connectors are installable through the connector registry.

The connector manifest (from step 11.01) includes the tool definitions, authentication requirements, and lifecycle hooks. The publishing flow validates the manifest against the connector-sdk specification and extracts metadata for the marketplace listing.

## Requirements

### Functional requirements

- Developer dashboard: list all their submissions with status, date, and actions
- Create listing: guided form with connector manifest upload, metadata entry, screenshot uploads, pricing configuration
- Upload connector manifest (JSON) — validate against connector-sdk schema, extract tools, auth methods, and metadata
- Submit for review: transition listing from draft to pending_review, notify admin queue
- Edit listing while in draft or pending_review status
- Admin review queue: list all pending submissions with connector details, manifest content, and developer info
- Admin approve: publish listing, make connector available for installation
- Admin reject: provide rejection reason, notify developer, allow resubmission
- Developer notification on status changes (approved, rejected, revision requested)
- Request revision from developer with specific feedback
- Connector registry integration: on approval, register the connector in the connector registry
- Unpublish listing: remove from public view, keep existing installations working
- Archive listing: completely retire listing, prevent new installations

### Non-functional requirements

- Manifest validation must complete in under 2 seconds
- Screenshot uploads must be processed asynchronously (resize, optimize)
- Admin review notification must be sent within 5 minutes of submission

### Performance requirements

- Listing creation: < 2s
- Manifest validation: < 2s
- Admin review queue: < 1s to load
- Status change notification: < 5 minutes

### Security requirements

- Only the listing author can edit their draft listing
- Manifest must be validated against connector-sdk schema to prevent malicious tool definitions
- Screenshot uploads must be scanned for malware and validated as actual images
- Admin review actions require `admin:marketplace` permission
- Developer identity must be verified before submission (email verified, GitHub linked)

### Accessibility requirements

- Submission form must have proper field labels and error announcements
- Status indicators must include text and icons
- File upload area must be keyboard accessible

---

## Architecture Constraints

- Listing status transitions must follow a state machine: `draft ↔ pending_review → approved → published ↔ archived`, with `rejected` and `revision_requested` as terminal states from `pending_review`
- The connector manifest must be stored both in the listing document and registered in the connector registry
- Screenshots are stored at `r2://marketplace/{listingId}/screenshots/{index}.{ext}`
- Admin review notifications are sent via in-app notification and email
- On approval, the connector is registered in the connector registry with `source: marketplace` and the listing ID

## Notes

The manifest validation is the most critical part of this step. A malicious or malformed manifest could introduce security vulnerabilities. Validate the manifest against the latest connector-sdk schema, check tool names for conflicts, verify that the manifest does not request excessive permissions, and validate all URLs in the manifest against a blocklist. Consider implementing an automated manifest analysis that flags suspicious patterns for admin review.
