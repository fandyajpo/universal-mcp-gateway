## Goal

Design and implement the marketplace listing schema, including connector metadata, categorization, pricing models, screenshots, author information, and search indexing. This establishes the data model for the entire marketplace.

---

## Business Context

The marketplace is where third-party developers publish their connectors and tools, and where users discover and install them. The listing schema is the foundation of the marketplace experience — it determines what information is available for browsing, searching, and evaluating connectors. A well-designed schema enables effective search, clear presentation, and informed installation decisions.

---

## Technical Context

Listing data is stored in a `marketplace_listings` MongoDB collection. Each listing represents a published connector or tool with all its metadata. The schema must support multiple languages, pricing models (free, paid, subscription), categorization (tags, categories), and rich media (icons, screenshots, documentation). Search is powered by MongoDB text indexes on name, description, and tags.

## Requirements

### Functional requirements

- Define listing schema with: name (unique), slug, short description, full description (markdown), category, tags, icon URL, screenshots (up to 5), author (name, email, website, GitHub profile), pricing model (free, one-time, monthly, usage-based), price (cents), featured image, documentation URL, support URL, privacy policy URL, status (draft, pending_review, approved, rejected, published, archived), compatibility (platform versions), permissions required, and creation/update timestamps
- Support multiple categories: chat, productivity, development, data, analytics, communication, marketing, sales, support, custom
- Support search by name, description, tags, and author
- Support filtering by category, pricing model, status, and compatibility
- Support sorting by name, date, rating, popularity, and price
- Provide listing statistics: view count, install count, active users, average rating, review count
- Schema must be extensible for future fields without breaking existing listings

### Non-functional requirements

- Search queries must complete in under 200ms
- Listing detail queries must complete in under 50ms
- Icon and screenshot URLs must support CDN caching

### Performance requirements

- Search query: < 200ms for 10k listings
- Listing detail: < 50ms
- Category filter query: < 100ms

### Security requirements

- Listing status must be validated on every public query — draft and archived listings are not returned
- Pricing information must be validated to prevent under-priced listings
- Author identity must be verified (verified email, GitHub account link)
- URLs in listing data must be validated against allowed patterns (no XSS vectors)

### Accessibility requirements

- Icon and screenshot images must have alt text fields in the schema
- Description markdown must support accessible heading hierarchy

## Architecture Constraints

- The `slug` field must be unique and immutable after listing creation
- MongoDB text index must cover: `name`, `shortDescription`, `fullDescription`, `tags`, `author.name`
- Listing statistics must be updated asynchronously (event-driven via Inngest) — never on the read path
- Pricing is stored in cents (integer) — never use floating-point for prices
- Screenshots and icons are stored in R2 — the listing stores URLs, not the files themselves

## Notes

The slug should be auto-generated from the name but editable during the draft stage. Once published, the slug becomes immutable to prevent broken installation references. The `status` field drives the entire publishing workflow — design the state transitions carefully. Consider using a state machine pattern for status transitions to enforce valid paths (e.g., draft → pending_review → approved → published, never published → draft).
