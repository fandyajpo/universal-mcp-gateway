## Goal

Implement `@repo/validation` — the centralized validation layer built on Zod. This package provides validation schemas for every domain entity (matching the types in `@repo/types`), reusable validation primitives, request validation helpers for Next.js Route Handlers and Server Actions, and TypeScript type inference helpers. Every user-facing input in the platform must be validated through this package before it reaches business logic.

---

## Business Context

Input validation is the first line of defense against malformed data, injection attacks, and logic errors. In a multi-tenant AI platform, a single invalid input can corrupt data across workspaces or trigger unexpected AI behavior. Centralizing all validation schemas in one package ensures consistency: the same schema that validates an API request also generates the TypeScript type and documents the expected shape. This "single source of truth" approach eliminates drift between API contracts and type definitions.

---

## Technical Context

The package scaffold has `package.json` with dependencies on `@repo/types` and `zod`. The `@repo/types` package defines all domain model interfaces; the validation package creates Zod schemas that mirror those interfaces. Where possible, types should be inferred from Zod schemas using `z.infer` rather than duplicating type definitions.

## Requirements

### Functional requirements

1. Create validation schemas for every domain entity defined in `@repo/types`:
   - Auth: `userSchema`, `sessionSchema`, `workspaceSchema`, `workspaceMemberSchema`, `workspaceRoleSchema`
   - Document: `documentSchema`, `documentChunkSchema`, `documentMetadataSchema`, `documentSourceSchema`
   - Connector: `connectorSchema`, `connectorConfigSchema`, `connectorStatusSchema`, `connectorTypeSchema`
   - Agent: `agentSchema`, `agentConfigSchema`, `agentStatusSchema`, `agentToolSchema`
   - Chat: `chatMessageSchema`, `chatThreadSchema`, `chatRoleSchema`, `chatConfigSchema`
   - Embedding: `embeddingSchema`, `embeddingConfigSchema`, `embeddingModelSchema`
   - RAG: `ragResultSchema`, `ragConfigSchema`, `retrievalStrategySchema`, `chunkStrategySchema`
   - MCP: `mcpToolSchema`, `mcpRequestSchema`, `mcpResponseSchema`, `mcpErrorSchema`, `mcpCapabilitySchema`
   - Platform: `apiKeySchema`, `auditLogSchema`, `billingPlanSchema`, `subscriptionSchema`, `featureFlagSchema`, `invitationSchema`, `notificationSchema`, `permissionSchema`, `rateLimitSchema`, `webhookSchema`
2. Create reusable validation primitives:
   - `slugSchema` — validates workspace/URL slugs (lowercase alphanumeric + hyphens, 3-63 chars)
   - `emailSchema` — validates email addresses
   - `urlSchema` — validates URLs with optional protocol enforcement
   - `uuidSchema` — validates UUID v4 format
   - `brandedIdSchema` — validates branded ID format
   - `passwordSchema` — validates password strength (min 8 chars, 1 uppercase, 1 lowercase, 1 number)
   - `hexColorSchema` — validates hex color codes
3. Create request validation helpers:
   - `validateBody<T>(schema, data)` — validates request body against a Zod schema
   - `validateQuery<T>(schema, searchParams)` — validates query parameters
   - `validateParams<T>(schema, params)` — validates route parameters
   - `validateAction<T>(schema, formData)` — validates Server Action form data
   - Each returns a typed result: `{ success: true, data: T } | { success: false, error: ZodError }`
4. Create a `validationMiddleware` helper for Next.js Route Handlers that wraps the handler with automatic body/query/param validation.
5. Export inferred types for every schema using `z.infer`.

### Non-functional requirements

1. All schemas must be tree-shakeable — importing one schema should not pull in all schemas.
2. Validation error messages must be human-readable and localized to English.
3. The `validateBody` family must strip unknown fields by default (`z.strip()` behavior).

### Performance requirements

1. Schema parsing must complete in under 5ms for typical payloads.
2. Schema objects should be created once at module level, not on every request.

### Security requirements

1. String schemas must have `.max()` limits to prevent billion-laughs-type attacks.
2. All user-provided strings must be trimmed of leading/trailing whitespace via `.trim()`.
3. The `passwordSchema` must enforce minimum complexity requirements.
4. HTML tags must be stripped or rejected in name/display fields via `.refine()`.

### Accessibility requirements

N/A — validation package has no UI.

## Package Boundaries

`@repo/validation` exports:
- All domain entity Zod schemas
- All reusable primitives (`slugSchema`, `emailSchema`, `passwordSchema`, etc.)
- All request validation helpers (`validateBody`, `validateQuery`, `validateParams`, `validateAction`, `validationMiddleware`)
- All inferred types from schemas

`@repo/validation` does NOT export:
- Raw `z.object()` calls without wrapping in a named schema
- Internal utility functions used only within the package
- Types that duplicate those already in `@repo/types`

---

## Architecture Constraints

1. Schemas must mirror the types in `@repo/types` — there must be a Zod schema for every model interface.
2. Where a type exists in `@repo/types` that is purely derived from a Zod schema, the schema is the source of truth.
3. Schemas must NOT import from any other `@repo/` package except `@repo/types` for type-only imports.
4. Validation helpers must be agnostic to HTTP framework — they accept plain objects, not Next.js-specific types.

---

## Acceptance Criteria

1. `userSchema.parse({...})` accepts valid user data and rejects invalid data.
2. `validateBody(userSchema, req)` returns `{ success: true, data }` for valid JSON payloads.
3. `passwordSchema.parse("weak")` returns a ZodError with messages about password requirements.
4. `slugSchema.parse("My Workspace")` fails because slugs must be lowercase with hyphens.
5. All schema types can be inferred with `z.infer<typeof userSchema>` and match the `@repo/types` interfaces.
6. `pnpm typecheck` and `pnpm lint` pass.

## Notes

- Use `z.object({...}).strip()` as the default (unknown field stripping). Opt into `.passthrough()` only when explicitly needed.
- For the `validationMiddleware`, accept a config object: `{ body?: ZodSchema, query?: ZodSchema, params?: ZodSchema }`.
- The middleware should return `NextResponse.json({ error: ... }, { status: 400 })` on validation failure.
- Keep primitives general-purpose; domain-specific refinements belong in domain schemas.
- Consider using `z.brand()` for branded ID validation to match `@repo/types` branded types.
