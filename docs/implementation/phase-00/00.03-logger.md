## Goal

Implement `@repo/logger` — a Pino-based structured logging package that provides context-aware logging, correlation ID propagation, configurable log levels, and a Sentry transport for error reporting. Every package and app in the monorepo will use this logger exclusively.

---

## Business Context

Observability is critical for an enterprise AI platform. Every AI model call, tool execution, API request, and background job must produce structured logs that can be ingested into the organization's observability pipeline (e.g., Grafana, Datadog, or Splunk). The logger must support correlation IDs so that a single user request spanning multiple services can be traced end-to-end. The Sentry transport ensures that errors are automatically captured and surfaced to the engineering team.

---

## Technical Context

The package scaffold already exists with `package.json` (depends on `@repo/types`, `pino`, `pino-pretty`) and a partially implemented `src/index.ts` that exports `createLogger`. The current implementation supports `level`, `pretty` mode, and `redact` paths. It lacks correlation ID support, child loggers, Sentry transport, and proper type definitions. The `src/types.ts` file referenced in imports does not exist — it must be created.

## Requirements

### Functional requirements

1. Define `LoggerConfig` interface with fields: `name`, `level` (union of Pino levels), `pretty` (boolean), `redact` (string array), `sentryDsn` (optional string).
2. Define `LogLevel` type as `"trace" | "debug" | "info" | "warn" | "error" | "fatal"`.
3. Update `createLogger(name, config?)` to accept the full `LoggerConfig` and return a Pino logger instance.
4. Implement `createChildLogger(parent, bindings)` that creates a child logger with additional context bindings (e.g., `{ requestId, userId, tenantId }`).
5. Implement Sentry transport integration: when `sentryDsn` is provided, errors logged at `error` or `fatal` level are automatically sent to Sentry.
6. Implement correlation ID support: the logger should accept a `correlationId` in context bindings and include it in every log line.
7. Implement `redact` paths merging: user-provided redact paths merge with the default sensitive field list.
8. Export the `Logger` return type from `createLogger`.

### Non-functional requirements

1. Zero overhead when no logger is created — Pino is imported lazily.
2. Sentry transport must not block the main thread — fire-and-forget async delivery.
3. Child logger creation must be O(1) — Pino's native child logger is used.

### Performance requirements

1. Pino is chosen for its low overhead (~50% faster than Winston in benchmarks).
2. Pretty-printing must be disabled in production — detected by `NODE_ENV !== "production"` unless explicitly overridden.

### Security requirements

1. Default redact list must include: `"password"`, `"secret"`, `"token"`, `"authorization"`, `"cookie"`, `"set-cookie"`, `"apiKey"`, `"accessToken"`, `"refreshToken"`.
2. Sentry transport must redact the same fields before sending.
3. Correlation IDs must not contain PII.

### Accessibility requirements

N/A — logger package has no UI.

## Package Boundaries

`@repo/logger` exports:
- `createLogger(name, config?)` — create a root logger
- `createChildLogger(logger, bindings)` — create a child logger with context
- `Logger` type — return type of `createLogger`
- `LoggerConfig` interface
- `LogLevel` type

`@repo/logger` does NOT export:
- Raw Pino instance creation
- Internal transport implementations
- Sentry SDK directly (it's an implementation detail)

---

## Architecture Constraints

1. Must NOT depend on any app-level package.
2. `@repo/types` dependency must only be used for type imports — no runtime import from types.
3. Sentry must be an optional integration — the package works without a DSN.
4. Pino must be the only logging library — no fallback to `console.log`.

---

## Acceptance Criteria

1. `createLogger("test").info("hello")` produces a structured JSON log line.
2. `createChildLogger(parent, { requestId: "abc" }).info("child")` includes `requestId` in output.
3. Redact paths successfully mask `password` and `authorization` fields in log output.
4. Providing a `sentryDsn` sends `error` level logs to Sentry (verified with test DSN).
5. `pnpm typecheck` and `pnpm lint` pass for the package.
6. Unit tests achieve ≥80% coverage.

## Notes

- The Sentry transport should use `@sentry/core`'s `captureException` and `captureMessage` rather than the full `@sentry/node` SDK to keep the bundle small.
- For the Sentry transport, if `@sentry/core` is not already a dependency, add it to `package.json` as an optional peer dependency.
- Pino's `level` should be read from `LOG_LEVEL` env var by default, falling back to `"info"`.
- The `createChildLogger` function is a thin wrapper around Pino's `child({...bindings})` method.
