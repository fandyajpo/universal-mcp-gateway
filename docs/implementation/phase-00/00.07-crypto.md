## Goal

Implement `@repo/crypto` — a security-critical package providing encryption, hashing, key derivation, and secure random generation. This package wraps Node.js built-in `crypto` module with a ergonomic, type-safe API. It will be used by the auth package (password hashing), API key management (key hashing), and any future feature that needs field-level encryption (e.g., connector credentials, OAuth tokens).

---

## Business Context

Enterprise AI platforms handle sensitive data: user passwords, API keys, OAuth tokens, connector credentials. Improper handling of this data is a compliance violation (SOC2, GDPR) and a security incident waiting to happen. Centralizing all cryptographic operations in a single, audited package ensures that encryption standards are consistent (AES-256-GCM, bcrypt cost 12), key management is controlled, and the attack surface is minimized.

---

## Technical Context

The package scaffold has `package.json` with a dependency on `@repo/types`. The package must use Node.js built-in `crypto` module exclusively — no third-party crypto libraries. The `bcrypt` algorithm is used for password hashing (via the `bcrypt` npm package, which wraps the native implementation). All encryption uses AES-256-GCM with random IVs.

## Requirements

### Functional requirements

1. **Password hashing** (`src/password.ts`):
   - `hashPassword(password)` — hashes a password using bcrypt with cost factor 12
   - `verifyPassword(password, hash)` — verifies a password against a bcrypt hash
   - `needsRehash(hash, cost?)` — checks if a password hash needs upgrading to a higher cost factor

2. **Symmetric encryption** (`src/encryption.ts`):
   - `encrypt(plaintext, key)` — encrypts a string using AES-256-GCM, returns base64-encoded ciphertext with IV and auth tag
   - `decrypt(ciphertext, key)` — decrypts a base64-encoded AES-256-GCM ciphertext
   - `generateEncryptionKey()` — generates a random 256-bit key and returns it as base64
   - `encryptObject(obj, key)` — serializes an object to JSON and encrypts
   - `decryptObject<T>(ciphertext, key)` — decrypts and parses JSON back to object

3. **Hashing utilities** (`src/hash.ts`):
   - `hashString(input, algorithm?)` — creates a SHA-256/SHA-512 hash of a string
   - `hashEquals(a, b)` — constant-time string comparison to prevent timing attacks
   - `hmac(input, key)` — creates an HMAC-SHA256 signature

4. **Token generation** (`src/token.ts`):
   - `generateApiKey()` — generates a formatted API key (e.g., `umgw_` + 48 random chars)
   - `generateResetToken()` — generates a password reset token (64 random hex chars)
   - `generateVerificationCode(length?)` — generates a numeric verification code (for MFA/email)
   - `generateSecret()` — generates a base64-encoded random secret (for TOTP)

### Non-functional requirements

1. All functions must use async/await where the underlying operation supports it (bcrypt).
2. Constant-time comparison must be used for all hash/secret comparisons.
3. Encryption keys must never be logged or exposed in error messages.

### Performance requirements

1. Password hashing (bcrypt cost 12) must complete in ~250ms — acceptable for auth operations.
2. Symmetric encryption/decryption must complete in under 5ms for typical payloads (<10KB).

### Security requirements

1. AES-256-GCM with random 96-bit IV (12 bytes) — never use static IVs.
2. Bcrypt cost factor of 12 minimum — never lower.
3. All functions must accept `Buffer | string` for keys.
4. Encryption keys must be provided by the caller (stored in env vars), not generated at runtime.
5. API key prefix (`umgw_`) must be configurable.
6. Decryption must authenticate the auth tag before returning plaintext — GCM provides this.

### Accessibility requirements

N/A — crypto package has no UI.

## Acceptance Criteria

1. `hashPassword("myPassword")` returns a bcrypt hash string starting with `$2b$12$`.
2. `verifyPassword("myPassword", hash)` returns `true` for the correct password.
3. `encrypt("secret", key)` returns a base64 string that `decrypt` can recover.
4. `decrypt` with wrong key or tampered ciphertext throws an error (auth tag verification fails).
5. `hashEquals(a, b)` returns `true` for equal strings and `false` for different strings.
6. `generateApiKey()` returns a string matching `/^umgw_[A-Za-z0-9_-]{48}$/`.
7. `generateEncryptionKey()` returns a valid base64-encoded 256-bit key.
8. `pnpm typecheck` and `pnpm lint` pass.

## Notes

- Add `bcrypt` (not `bcryptjs`) to dependencies — it uses native bindings for performance.
- The bcrypt cost factor of 12 is a baseline. In production with high auth traffic, consider offloading to a worker thread.
- For `generateApiKey`, use `crypto.randomBytes(36)` and encode as base64url, then prepend the prefix. 36 bytes = 48 base64 characters.
- The encryption key should be 32 bytes (256 bits) encoded as base64. Store it in the `ENCRYPTION_KEY` env var (not yet in `.env.example` — add it in this step).
