## Goal

Implement the `@repo/database` package's core infrastructure: the MongoDB/Mongoose connection manager and the generic `BaseRepository` class. The connection manager handles connection lifecycle, health checks, auto-reconnect, and pool configuration. The `BaseRepository` provides a type-safe, tenant-aware generic CRUD interface that every entity-specific repository will extend.

---

## Business Context

The database layer is the backbone of the platform. Every feature — auth, workspace, chat, RAG, billing — depends on reliable database access. A robust connection manager that handles transient failures (network blips, replica set elections, maintenance windows) is essential for production uptime. The `BaseRepository` pattern enforces tenant isolation at the data access level: it is structurally impossible for a repository to query data from another tenant's workspace without explicitly bypassing the abstraction.

---

## Technical Context

The package scaffold exists with `package.json`. Mongoose is the ODM for MongoDB Atlas. The database is multi-tenant with tenant isolation enforced by a `tenantId` field on every document. The connection string is provided by `@repo/config`. The repository pattern follows the architecture documented in `ARCHITECTURE.md`.

## Requirements

### Functional requirements

1. **Connection Manager** (`src/connection.ts`):
   - `connect()` — establishes Mongoose connection using `DATABASE_URL` from config
   - `disconnect()` — gracefully closes the Mongoose connection
   - `getConnection()` — returns the current Mongoose connection (throws if not connected)
   - `isConnected()` — returns boolean indicating connection state
   - `healthCheck()` — pings the database and returns `{ ok: boolean, latency: number, error?: string }`
   - Event listeners for `connected`, `disconnected`, `reconnected`, `error` events
   - Configurable pool size (min/max connections, default: min=5, max=20)
   - Auto-reconnect with exponential backoff (initial delay: 1s, max delay: 30s)
   - Connection timeout (default: 10s)

2. **BaseRepository** (`src/repositories/base.ts`):
   - Generic class `BaseRepository<T, TDocument>` where `T` is the domain entity type and `TDocument` is the Mongoose document type
   - `findById(id)` — find by MongoDB `_id`
   - `findOne(filter)` — find one document with filter
   - `findMany(filter, options?)` — find multiple documents with pagination (`skip`, `limit`), sorting, field selection
   - `create(data)` — create a single document
   - `createMany(data)` — bulk insert
   - `updateById(id, data)` — update by `_id` with partial update
   - `updateMany(filter, data)` — bulk update
   - `deleteById(id)` — soft delete (sets `deletedAt` timestamp)
   - `hardDeleteById(id)` — permanent deletion (restricted to admin operations)
   - `count(filter)` — count documents matching filter
   - `exists(filter)` — check if any document matches filter
   - Every query method automatically injects `tenantId` filter when operating in tenant context

3. **Tenant-aware behavior** (`src/repositories/tenant-aware.ts`):
   - `TenantAwareRepository<T>` extends `BaseRepository<T>` with mandatory `tenantId` scoping
   - Constructor requires `tenantId` — all queries are scoped to that tenant
   - `create` automatically sets `tenantId` on the new document
   - Override `withoutTenantScope()` method for admin operations that cross tenant boundaries (requires explicit `isAdmin` flag)

4. **Schema utilities** (`src/schema.ts`):
   - `baseSchema` — common fields for all Mongoose schemas: `_id`, `tenantId`, `createdAt`, `updatedAt`, `deletedAt`
   - `timestampsPlugin` — Mongoose plugin that auto-manages `createdAt`/`updatedAt`
   - `softDeletePlugin` — Mongoose plugin that filters out soft-deleted documents by default
   - `toJSONTransform` — standard toJSON transform that removes `__v` and maps `_id` to `id`

### Non-functional requirements

1. Connection manager must be a singleton — only one connection per process.
2. Repository operations must use Mongoose lean queries where possible for read performance.
3. Soft-delete must be the default for all deletion operations — hard delete requires explicit intent.

### Performance requirements

1. Connection pool must support at least 20 concurrent operations.
2. `findById` must complete in under 10ms on a warm connection.
3. Bulk insert of 100 documents must complete in under 200ms.

### Security requirements

1. `tenantId` must be injected automatically — it cannot be passed from user input in queries.
2. Soft-deleted documents must be excluded from all query results by default.
3. Connection string must never be logged or exposed in error messages.

### Accessibility requirements

N/A — database package has no UI.

`@repo/database` does NOT export:
- Raw Mongoose instance
- Internal connection state
- Mongoose driver directly

---

## Architecture Constraints

1. All entity repositories must extend `BaseRepository` or `TenantAwareRepository` — never use raw Mongoose models.
2. The connection manager must use `@repo/config` for connection string — never read `process.env` directly.
3. Soft delete is the only delete operation available by default — hard delete must be explicitly called.
4. Every schema must include `tenantId`, `createdAt`, `updatedAt`, and `deletedAt` fields.

---

## Acceptance Criteria

2. `healthCheck()` returns `{ ok: true, latency: <number> }` on a healthy connection.
3. `BaseRepository.findById()` returns the correct document.
4. `TenantAwareRepository.findMany()` only returns documents matching the scoped `tenantId`.
5. `TenantAwareRepository.create()` automatically sets `tenantId` on the new document.
6. Soft-deleted documents are excluded from normal queries.
7. `disconnect()` cleanly closes the connection.
8. `pnpm typecheck` and `pnpm lint` pass.

## Notes

- Use `mongodb-memory-server` for integration tests — it downloads and runs a real mongod binary in-memory.
- Keep the Mongoose schema definition separate from the TypeScript type — use `@repo/types` for types and Mongoose `Schema` for runtime validation.
- The `baseSchema` should define: `{ tenantId: { type: String, required: true, index: true }, createdAt: Date, updatedAt: Date, deletedAt: { type: Date, default: null } }`.
- For `softDeletePlugin`, add a `pre('find')` hook that filters `{ deletedAt: null }` automatically.
- Connection pooling: Mongoose default is `min: 5, max: 20`. For production, tune based on workload.
- The `mongodb-memory-server` must be a `devDependency` only — it is NOT used in production.
