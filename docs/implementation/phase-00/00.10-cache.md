## Goal

Implement `@repo/cache` — a distributed caching layer built on Upstash Redis. This package provides a Redis client wrapper, multiple cache strategies (TTL-based, LRU, sliding window), a rate limiter implementation, and distributed lock support. Every package that needs caching (AI Gateway for rate limiting, RAG Engine for embedding cache, Auth for session store) will use this package.

---

## Business Context

Caching is critical for both performance and cost control. The AI Gateway needs rate limiting (Redis-based token bucket) to prevent abuse and control costs. The RAG Engine needs embedding caching to avoid redundant API calls to OpenRouter (each embedding call costs money and adds latency). Session caching in Redis is faster and more scalable than database lookups. A centralized cache package with consistent key naming and TTL management prevents cache-related bugs (stale data, cache stampedes, key collisions across tenants).

---

## Technical Context

The package scaffold exists with `package.json` (depends on `@repo/types`, `@repo/logger`, `@upstash/redis`). Upstash Redis is a serverless Redis provider accessed via HTTP/REST (not the Redis protocol). The `@upstash/redis` SDK handles the HTTP connection. The cache must be tenant-aware — all keys are prefixed with `tenant:{tenantId}:`.

## Requirements

### Functional requirements

1. **Redis client wrapper** (`src/client.ts`):
   - `createCacheClient(config?)` — creates an Upstash Redis client with default configuration
   - `get(key)` — gets a value by key (auto-deserializes JSON)
   - `set(key, value, ttl?)` — sets a value with optional TTL (auto-serializes JSON)
   - `del(key)` — deletes a key
   - `exists(key)` — checks if a key exists
   - `healthCheck()` — pings Redis and returns `{ ok: boolean, latency: number }`
   - `flush()` — clears all keys (restricted to admin operations)

2. **Cache strategies** (`src/strategies.ts`):
   - `TTLCache` — basic TTL-based cache with configurable default TTL
   - `SlidingWindowCache` — cache that resets TTL on every read (for hot data)
   - `LRUCache` — least-recently-used eviction (uses Redis sorted sets for tracking)
   - Each strategy implements `CacheStrategy` interface: `get<T>(key)`, `set<T>(key, value, ttl?)`, `del(key)`, `has(key)`, `clear()`

3. **Rate limiter** (`src/rate-limiter.ts`):
   - `RateLimiter` class implementing token bucket algorithm using Redis
   - `check(key, limit, window)` — checks if the request is within limits
   - `increment(key, window)` — increments the counter for a sliding window
   - `getRemaining(key, limit, window)` — returns remaining capacity
   - `reset(key)` — resets the rate limit counter
   - Support for multiple tiers: global, tenant, user, API key
   - Uses Redis sorted sets for sliding window implementation

4. **Distributed lock** (`src/lock.ts`):
   - `acquireLock(key, ttl)` — acquires a distributed lock with Redis SET NX + TTL
   - `releaseLock(key, token)` — releases the lock only if owned by the caller (Lua script for atomicity)
   - `withLock<T>(key, ttl, fn)` — convenience function that acquires lock, executes fn, releases lock
   - Lock timeout: default 10s — must be less than the operation's expected duration

5. **Tenant-aware cache** (`src/tenant-cache.ts`):
   - `TenantCache` — wraps any `CacheStrategy` with automatic tenant key prefixing
   - Constructor takes `tenantId` and a `CacheStrategy` instance
   - All methods transparently prefix keys with `tenant:{tenantId}:`

### Non-functional requirements

1. All operations must be async — Upstash Redis is HTTP-based.
2. Serialization uses `JSON.stringify`/`JSON.parse` for objects, raw string for strings.
3. Cache misses must return `undefined` — never throw.
4. Rate limiter must be statistically accurate (sliding window, not fixed window).

### Performance requirements

1. `get`/`set` operations must complete in under 50ms (Upstash Redis global latency).
2. Rate limiter `check` must complete in under 30ms.
3. Distributed lock acquire must complete in under 50ms.

### Security requirements

1. Cache keys must never contain PII.
2. Rate limiter keys must be prefixed by tenant ID to prevent cross-tenant rate limit exhaustion.
3. Lock tokens must be cryptographically random (use `crypto.randomUUID()`).

### Accessibility requirements

N/A — cache package has no UI.

---

## Architecture Constraints

1. Must use `@upstash/redis` REST client — not the Node.js `redis` package (io-redis).
2. All keys must follow the pattern: `tenant:{tenantId}:{domain}:{entity}:{id}:{field}`.
3. The `RateLimiter` must use a sliding window implementation (sorted set with timestamps), not a fixed window counter.
4. Distributed locks must use the Redlock-like algorithm with Lua scripting for atomicity.
5. Must not have any dependency on `@repo/config` — the caller provides Redis URL.

## Notes

- Mock `@upstash/redis` for unit tests. Create `src/__mocks__/@upstash/redis.ts` if Vitest manual mocks are used.
- The `RateLimiter` sliding window implementation: use `ZADD` to add a timestamped entry, `ZREMRANGEBYSCORE` to remove entries outside the window, `ZCARD` to count remaining entries. This is atomic with a Lua script.
- For the distributed lock Lua script: `if redis.call("GET", KEYS[1]) == ARGV[1] then return redis.call("DEL", KEYS[1]) else return 0 end`.
- Default TTL for caches should be 300 seconds (5 minutes) unless otherwise specified.
- The `LRUCache` is approximate — Redis does not have native LRU on individual keys. Use a sorted set with access timestamps and periodically evict.
