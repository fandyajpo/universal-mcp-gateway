## Goal

Implement `@repo/config` — a centralized, validated configuration loader that reads environment variables and produces a typed, immutable configuration object. Every app and server-side package will source its configuration from this package, ensuring consistent env var names, type safety, and early validation at startup.

---

## Business Context

Enterprise deployments require configuration that varies by environment (development, staging, production). Hard-coded defaults, inconsistent env var names, and missing validation lead to runtime failures that are difficult to debug. A single config package that validates all required variables on startup eliminates an entire class of production incidents. It also provides a single source of truth for documentation — developers can read the config schema to understand what environment variables are needed.

---

## Technical Context

The scaffold `package.json` declares dependencies on `@repo/types` and `zod`. The package does not yet have any source files. The implementation must read from `process.env`, validate with Zod, and export a frozen config object. Configuration is loaded once at startup and cached for the lifetime of the process.

## Requirements

### Functional requirements

1. Define a Zod schema for the complete app configuration, covering: `NODE_ENV`, `APP_URL`, `APP_NAME`, `DATABASE_URL`, `REDIS_URL`, `R2_*` (5 vars), `OPENROUTER_*` (3 vars), `BETTER_AUTH_*` (2 vars), `SENTRY_*` (2 vars), `INNGEST_*` (3 vars), `LOG_LEVEL`, `LOG_PRETTY`.
2. Implement `loadConfig()` function that reads `process.env`, validates against the schema, and returns a typed, deeply frozen config object.
3. Implement `getConfig()` function that returns the cached config object (singleton pattern — loads on first call).
4. Implement `validateConfig()` function that can be called at startup to fail fast if env vars are missing or invalid.
5. Implement `isProduction()`, `isDevelopment()`, `isTest()` helper functions.
6. Each config section must be a nested object: `config.app.url`, `config.database.url`, `config.redis.url`, etc.
7. Default values must be provided for optional vars (e.g., `LOG_LEVEL` defaults to `"info"`, `LOG_PRETTY` defaults to `false`).

### Non-functional requirements

1. Config is loaded once and cached — subsequent calls to `getConfig()` return the frozen singleton.
2. Zod validation errors must produce human-readable messages that include the env var name and expected format.
3. The config object must be immutable — `Object.freeze` applied recursively.

### Performance requirements

1. Config loading must complete in under 50ms — only string parsing and Zod validation.
2. `getConfig()` must be O(1) after first load — simply returns the cached object.

### Security requirements

1. The config object must NOT expose raw env var values in error messages — only the var name and validation error.
2. Connection strings containing credentials must NOT be logged.

### Accessibility requirements

N/A — config package has no UI.

## Package Boundaries

`@repo/config` exports:
- `getConfig()` — returns the typed, frozen config object
- `loadConfig(overrides?)` — loads config with optional override object (for testing)
- `validateConfig()` — validates env vars and throws on failure
- `isProduction()`, `isDevelopment()`, `isTest()` — environment checks
- `Config` type — inferred from Zod schema via `z.infer`
- `AppConfig`, `DatabaseConfig`, `RedisConfig`, `R2Config`, `OpenRouterConfig`, `AuthConfig`, `SentryConfig`, `InngestConfig`, `LoggingConfig` — section types

---

## Architecture Constraints

1. Must only depend on `@repo/types` (for type imports) and `zod`.
2. Must NOT import from `@repo/logger` — config is needed before logger initialization.
3. Config must be loadable without any other package initialized.
4. The config object must be frozen to prevent runtime mutations.

---

## Acceptance Criteria

1. `getConfig().app.url` returns `"http://localhost:3000"` by default when `APP_URL` is unset.
2. Calling `loadConfig()` without `DATABASE_URL` set throws a Zod validation error mentioning `DATABASE_URL`.
3. `isProduction()` returns `false` when `NODE_ENV=development`.
4. The returned config object is deeply frozen — `Object.isFrozen(config)` is `true`.
5. `pnpm typecheck` and `pnpm lint` pass.

## Notes

- Use `z.string().url()` for URL fields, `z.string().min(1)` for required non-URL strings.
- The `NODE_ENV` field should use `z.enum(["development", "production", "test"])`.
- For the `loadConfig(overrides?)` parameter, accept a `Partial<Record<string, string>>` to allow test overrides without touching `process.env`.
- Section naming convention: `app`, `database`, `redis`, `r2`, `openrouter`, `auth`, `sentry`, `inngest`, `logging`.
