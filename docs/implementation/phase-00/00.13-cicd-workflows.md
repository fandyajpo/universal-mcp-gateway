## Goal

Create and harden GitHub Actions workflows for continuous integration and continuous deployment. The CI workflow must run on every push and pull request, executing lint → typecheck → test → build in sequence. The security workflow must run dependency scanning and SAST. A deploy workflow must be ready for manual dispatch. All workflows must complete within reasonable time limits (CI < 10 minutes) and provide clear failure notifications.

---

## Business Context

A reliable CI pipeline is the gatekeeper of code quality. In a 15-package monorepo, a broken build in one package can cascade to dependent packages. The CI pipeline must catch issues early, provide fast feedback, and prevent broken code from reaching production. Automated security scanning (dependency audit, SAST) is required for SOC 2 compliance. Deploy workflows must be reproducible and auditable.

---

## Technical Context

The `.github/workflows/` directory already has three workflow files: `ci.yml`, `security.yml`, and `dependency-review.yml`. These are scaffold stubs that need to be replaced with complete, working configurations. The build system is Turborepo, which handles task orchestration and caching. The workflows must use `pnpm` for package management and configure the Node.js runtime (20.x).

## Requirements

### Functional requirements

1. **CI workflow** (`.github/workflows/ci.yml`):
   - Trigger: `push` to `main`, `pull_request` to `main`
   - Job 1 — Lint: runs `pnpm lint` across all packages
   - Job 2 — TypeCheck: runs `pnpm typecheck` across all packages (depends on lint)
   - Job 3 — Test: runs `pnpm test` across all packages (depends on typecheck)
   - Job 4 — Build: runs `pnpm build` (depends on test, runs in parallel for all apps)
   - Job 5 — Bundle Analysis: runs on main branch only, uploads bundle stats
   - Caching: Turborepo remote caching if configured, otherwise local `node_modules` caching
   - Timeout: 15 minutes
   - Concurrency: cancel in-progress runs for the same branch
   - Notification: post failure comment on PR

2. **Security workflow** (`.github/workflows/security.yml`):
   - Trigger: `push` and `pull_request` to `main`, weekly schedule
   - Job 1 — Dependency Audit: `pnpm audit` with fail on critical/high
   - Job 2 — SAST: `semgrep` or `eslint-plugin-security` scan
   - Job 3 — Secrets Scan: `trufflehog` or `gitguardian` scan for committed secrets
   - Job 4 — License Check: verify dependency licenses are compatible (MIT, Apache-2.0, BSD)

3. **Deploy workflow** (`.github/workflows/deploy.yml`):
   - Trigger: `workflow_dispatch` with environment selection (dev, staging, production)
   - Job 1 — Build: builds all apps
   - Job 2 — Deploy: deploys to the selected environment
   - Environment-specific secrets and approval gates
   - Slack notification on success/failure

4. **Dependency review workflow** (`.github/workflows/dependency-review.yml`):
   - Trigger: `pull_request` to `main`
   - Uses `actions/dependency-review-action@v4`
   - Fails on high/critical vulnerabilities in new dependencies
   - Comments on PR with summary

### Non-functional requirements

1. CI must complete in under 10 minutes (cached).
2. Failures must be visible within 2 minutes of push.
3. Cache hit rate must be >80% for consecutive runs.

### Performance requirements

1. Node module caching: `actions/setup-node` with `cache: pnpm`.
2. Turborepo cache: persist `.turbo/` directory between runs.
3. Build only changed packages: Turborepo handles this automatically.

### Security requirements

1. All secrets must be stored as GitHub Actions secrets, not in workflow files.
2. Deploy workflow must require approval for production environment.
3. `GITHUB_TOKEN` must have minimal permissions (read: contents, write: checks).
4. `pnpm audit` must fail on critical and high severity vulnerabilities.

### Accessibility requirements

N/A — CI/CD workflows have no UI.

## Notes

- Use `actions/checkout@v4` with `fetch-depth: 0` for all workflows.
- For Turborepo caching between runs, persist `.turbo/` directory: `actions/cache` with path `.turbo` and key based on lockfile hash.
- For pnpm caching: `actions/setup-node` with `cache: 'pnpm'` and `cache-dependency-path: 'pnpm-lock.yaml'`.
- The deploy workflow should have environment-specific variables: `dev`, `staging`, `production`, each with different secrets.
- Consider adding a matrix strategy for the build job if app builds can be parallelized.
- Add `.github/dependabot.yml` for automated weekly dependency updates.
