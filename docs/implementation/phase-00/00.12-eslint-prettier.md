## Goal

Create shared, consistent ESLint and Prettier configurations that enforce code quality standards across all packages and apps in the monorepo. The ESLint config must be an extensible flat config (`eslint.config.js`) that every package and app can extend. It must integrate with `typescript-eslint`, enforce the coding conventions documented in `AI_INSTRUCTIONS.md`, and be compatible with Prettier (no conflicting rules).

---

## Business Context

Code quality automation is non-negotiable in a monorepo with 15+ packages and 4 apps. Manual code review cannot catch every stylistic inconsistency, unused import, or type error. Shared lint and format configurations ensure that every contributor produces code that looks like it was written by one person. This reduces review friction, prevents CI failures due to formatting, and enforces security-sensitive rules (no `any`, no `console.log`, no direct `process.env` access).

---

## Technical Context

ESLint v9 uses the flat config format (`eslint.config.js`). The project already has Prettier configured at the root (`.prettierrc`). The ESLint config must be a sharable flat config that is installed as an npm package or referenced by all packages/apps. The `typescript-eslint` plugin provides TypeScript-aware rules. Plugins needed: `@eslint/js`, `typescript-eslint`, `eslint-plugin-react`, `eslint-plugin-react-hooks`, `eslint-plugin-import`, `eslint-plugin-perfectionist` (for import ordering), `eslint-config-prettier` (to disable formatting rules).

## Requirements

### Functional requirements

1. **ESLint flat config** (`packages/config/eslint/index.ts`):
   - Extensible base config that sets up `typescript-eslint` with type-aware rules
   - Config object for each language/tech: TypeScript base, React, Node.js, imports
   - Rules that enforce naming conventions (PascalCase for types/interfaces, camelCase for functions/variables)
   - Import ordering rule (external → `@repo/*` → internal)
   - Restriction on `console.log` (only `console.error` and `console.warn` allowed)
   - Restriction on `any` type — prefer `unknown`
   - Restriction on `process.env` direct access — must use `@repo/config`
   - React rules: prefer named exports, no default exports, hooks rules
   - `eslint-config-prettier` must be the last config to disable formatting rules

2. **Package-specific configs** — each package and app should have a minimal `eslint.config.js`:
   ```typescript
   import { baseConfig } from "@repo/config-eslint";
   export default baseConfig;
   ```

3. **Prettier** — the existing `.prettierrc` at root is the single source of truth. No per-package Prettier configs.

4. **Scripts** — root-level `pnpm lint` runs ESLint across all packages and apps. Each package has its own `lint` script.

### Non-functional requirements

1. ESLint must complete in under 30 seconds for the entire monorepo on modern hardware.
2. The base config must be the only place where rules are defined — every package/app inherits from it.

### Performance requirements

1. ESLint caching must be enabled with `--cache` flag in scripts.
2. Type-aware rules must use project `tsconfig.json` for performance.

### Security requirements

1. Rule against `innerHTML` and `dangerouslySetInnerHTML` must be enabled.
2. Rule against `eval()` must be enabled.
3. Rule requiring CSP headers must be advisory.
4. No secrets in lint output — disable verbose logging by default.

### Accessibility requirements

N/A — config package has no UI.

## Architecture Constraints

1. Must use ESLint flat config (v9) format — NOT the legacy `.eslintrc` format.
2. Must use `typescript-eslint` v8+ with type-aware rules.
3. Must NOT have any ESLint plugin that conflicts with Prettier.
4. The root `package.json` `lint` script must use `turbo lint` which invokes each package's script.

---

## Acceptance Criteria

1. `pnpm lint` runs without errors across all packages and apps.
2. `console.log("test")` triggers a lint error (must use `@repo/logger`).
3. `const x: any = 5` triggers a lint error.
4. `process.env.SOME_VAR` triggers a lint error (must use `@repo/config`).
5. Import order: external packages first, then `@repo/`, then relative imports.
6. No formatting conflicts: `pnpm lint` passes alongside `pnpm format:check`.

## Notes

- The ESLint config package name should be `@repo/config-eslint` in its `package.json`.
- Use `typescript-eslint`'s `configs.recommendedTypeChecked` as the base, then add custom rules.
- For the `no-restricted-syntax` rule, target `CallExpression[callee.object.name='console'][callee.property.name='log']`.
- For `process.env` restriction, use `no-restricted-syntax` targeting `MemberExpression[object.name='process'][property.name='env']`.
- Import ordering: use `eslint-plugin-perfectionist` or `eslint-plugin-simple-import-sort`.
- Each package's `eslint.config.js` should be minimal: just import from `@repo/config-eslint` and export.
