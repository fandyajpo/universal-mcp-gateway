## Goal

Integrate Sentry for error monitoring and performance tracing across the entire platform. The integration must capture unhandled exceptions, log error-level events from the logger package, and provide release tracking. Sentry must be configured with source maps for readable stack traces and environment-aware DSN switching.

---

## Business Context

Production error monitoring is essential for an enterprise SaaS platform. Sentry provides real-time error alerts, performance tracing, and release health monitoring. Integration at the foundation level ensures that every package and app automatically benefits from error tracking — no per-feature Sentry setup needed. Source map upload ensures that production stack traces are readable. Environment-aware configuration ensures development errors don't pollute production monitoring.

---

## Technical Context

The Sentry DSN is provided via environment variable (`SENTRY_DSN`). The base logger package (step 00.03) conditionally integrates with Sentry when a DSN is present. This step sets up the Sentry SDK in the monorepo, creates the initialization helpers, configures source map upload in the build pipeline, and creates the Sentry wrappers for Next.js applications. The `@sentry/nextjs` SDK is used for Next.js apps, while `@sentry/core` is used for packages.

## Requirements

### Functional requirements

1. **Sentry initialization** (`packages/config/sentry/index.ts`):
   - `initSentry()` — initializes the Sentry SDK with DSN, environment, release, and sample rate
   - Reads config from `@repo/config` (SENTRY_DSN, SENTRY_ENVIRONMENT)
   - Sets release from `process.env.SENTRY_RELEASE` or git commit SHA
   - Configures default integrations (InboundFilters, FunctionToString, Breadcrumbs, Deduplicate)
   - Sets `tracesSampleRate` based on environment (1.0 for dev, 0.2 for production)

2. **Next.js integration** — wrappers for each Next.js app:
   - `sentry.client.config.ts` — client-side Sentry configuration
   - `sentry.server.config.ts` — server-side Sentry configuration
   - `sentry.edge.config.ts` — edge runtime Sentry configuration
   - These files are created in each app that has Next.js (web, admin, docs, landing)

3. **Source map upload** — configure the build pipeline to upload source maps to Sentry:
   - `sentry.properties` — Sentry CLI configuration for source map upload (org, project, URL)
   - Build script integration: `sentry sourcemaps upload` after build
   - `cleanArtifacts: true` to remove source maps from production build output

4. **Performance monitoring**:
   - `Sentry.startSpan` for critical operations (database queries, AI gateway calls, MCP tool execution)
   - Transaction naming for API routes and Server Actions
   - Route change tracking for client-side navigation

5. **Error boundary** — shared Sentry ErrorBoundary component in `@repo/ui`:
   - Catches React rendering errors
   - Reports to Sentry
   - Shows a fallback UI
   - Provides a "Try again" button

### Non-functional requirements

1. Sentry must NOT block application startup — initialization is async, fail silently if DSN is not configured.
2. Source maps must be uploaded for production builds only — not for development.
3. Performance overhead must be minimal (<5% increase in p95 latency).

### Performance requirements

1. `tracesSampleRate`: 0.2 in production, 1.0 in development.
2. `profilesSampleRate`: 0.1 in production, 0 in development (profiling disabled by default).

### Security requirements

1. DSN must never be exposed to the client — use server-side Sentry for sensitive operations.
2. PII must be stripped from error reports using `beforeSend` callback.
3. Breadcrumbs must not include request bodies or auth tokens.
4. Source maps are uploaded (not shipped to client) — only Sentry can deobfuscate.

### Accessibility requirements

N/A — Sentry integration has no UI, except the ErrorBoundary fallback.

## Notes

- The `beforeSend` callback should redact: `event.request?.headers?.cookie`, `event.request?.headers?.authorization`, `event.user?.email` (keep anonymous ID), `event.extra?.password`.
- For Next.js apps, use `withSentryConfig` in `next.config.ts` to wrap the config. Each app's `next.config.ts` should import the Sentry config wrapper.
- Sentry's `tunnel` option should be configured to avoid ad-blockers blocking Sentry requests from the client.
- Add `@sentry/nextjs` as a dependency to each app's `package.json`. Add `@sentry/core` to `@repo/config` or create a standalone `@repo/sentry` package.
- The ErrorBoundary should accept a `fallback` prop (ReactNode) and a `onError` callback for custom error handling.
