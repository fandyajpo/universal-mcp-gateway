## Goal

Create a comprehensive verification suite that validates the integrity of all Phase 00 packages and their integration. This is not a replacement for unit tests — it is a higher-level suite that verifies package contracts, cross-package integration, dependency graph compliance (no circular dependencies), and architecture rule enforcement. The suite runs as the final quality gate for Phase 00 and serves as a regression detection tool for all future phases.

---

## Business Context

A monorepo with 15+ packages and complex dependency chains needs automated guardrails. A dedicated verification suite catches issues that individual package tests miss: a package exporting internal implementation details, a circular dependency introduced by a refactor, or a broken barrel export. This suite also serves as living documentation — it formally encodes the architecture rules from `ARCHITECTURE.md` and `AI_INSTRUCTIONS.md` as executable tests.

---

## Technical Context

The verification suite lives in a `scripts/verify/` directory or as a dedicated package. It uses Vitest for test execution. It imports from all packages to verify their public APIs work as expected. It also uses static analysis (the TypeScript compiler API and ESLint) to verify dependency graph rules and package boundaries.

## Requirements

### Functional requirements

1. **Package contract verification** (`verify/contracts.test.ts`):
   - Verify that every package's `src/index.ts` exports are documented and importable
   - Verify that no package exports internal modules (files outside `src/index.ts` barrel)
   - Verify that every public export has JSDoc
   - Verify that all `@repo/*` imports in packages are declared in their `package.json` dependencies

2. **Dependency graph verification** (`verify/dependencies.test.ts`):
   - Parse all `package.json` files and verify the dependency graph is a DAG (no cycles)
   - Verify that no package depends on any app (check for `@repo/*` imports in apps are OK, but not vice versa)
   - Verify that `@repo/types` has zero runtime dependencies
   - Verify that packages only depend on packages that are "below" them in the architecture (e.g., `@repo/ai` does not depend on `@repo/mcp`)
   - Verify the import rule: external → `@repo/*` → relative

3. **Architecture rule verification** (`verify/architecture.test.ts`):
   - Verify that no file outside `@repo/types` imports from `@repo/types` internal paths (only via barrel)
   - Verify that all Mongoose models use `baseSchema` from `@repo/database`
   - Verify that all repositories extend `BaseRepository` or `TenantAwareRepository`
   - Verify that all schemas in `@repo/validation` are exported from the barrel file
   - Verify that all logger usage goes through `@repo/logger` (no `console.log`)
   - Verify that no package uses `process.env` directly (must use `@repo/config`)

4. **Cross-package integration test** (`verify/integration.test.ts`):
   - Import and use the `@repo/types` branded IDs in a cross-package scenario
   - Create a logger via `@repo/logger`, log a message, read it from Sentry transport mock
   - Load config via `@repo/config`, verify values match env
   - Validate a user schema via `@repo/validation`
   - Use a utility function from `@repo/utils`
   - Hash and verify a password via `@repo/crypto`
   - Create a user document via `@repo/database` UserRepository (with mongodb-memory-server)
   - Set and get a cache value via `@repo/cache`
   - Render a `Button` component from `@repo/ui`

5. **Build integrity verification** (`verify/build.test.ts`):
   - Run `pnpm build` and verify all packages produce `dist/` with correct structure
   - Verify that each package's `dist/index.{js,d.ts}` has the expected exports
   - Verify no package has missing dependencies (detected by runtime import failures)

### Non-functional requirements

1. The full verification suite must complete in under 5 minutes.
2. The suite must not require external services (no database, no Redis, no Sentry DSN).
3. Failure messages must clearly indicate which rule was violated and which file(s) are responsible.

### Performance requirements

1. Static analysis tests must complete in under 10 seconds.
2. Integration tests that use mongodb-memory-server may take longer (under 2 minutes).

### Security requirements

1. Verification tests must NOT load any production secrets.
2. Integration tests must use mock/sandbox environments only.

### Accessibility requirements

N/A — verification suite has no UI.

## Architecture Constraints

2. The suite must NOT be imported by any other package — it is an independent script.

## Acceptance Criteria

1. `pnpm verify` runs all verification tests and exits with code 0.
2. Package contract tests detect when a new export is added to a barrel file without JSDoc (fails).
3. Dependency graph tests detect when a circular dependency is introduced (fails).
4. Architecture tests detect when `console.log` is used instead of `@repo/logger` (fails).
5. Integration tests pass, confirming all packages work together.
6. Build integrity tests confirm all packages produce correct build output.

## Notes

- For dependency graph verification, use a topological sort algorithm on the package dependency graph. If cycles exist, the sort fails.
- For import rule verification, parse each TS file's imports using the TypeScript compiler API or a regex-based approach (regex is simpler but less accurate — prefer the compiler API).
- For the integration test, create a factory function that provides mock implementations of external services (Sentry, Upstash Redis).
- The `build.test.ts` should run `pnpm build --filter @repo/verify` in a subprocess (via `execa` or `child_process`) and check exit codes and output files.
- Consider adding a `verify:ci` script that runs only the static analysis tests (faster) and a `verify:full` script that runs everything including integration and build tests.
