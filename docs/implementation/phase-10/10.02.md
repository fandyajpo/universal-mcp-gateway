## Goal

Implement the tool registry that manages the lifecycle of tool definitions within the MCP Gateway. The registry supports registering, listing, updating, deprecating, and removing tools. Each tool has a name, description, input schema (JSON Schema), output schema, handler function, and metadata. The registry is the authoritative source of truth for all available tools.

## Business Context

Tools are the core capability that MCP exposes to AI models. The registry enables dynamic tool discovery — AI models can list available tools and select the right one for their task. The registry also enables tool governance: approval workflows, versioning, and access control.

## Technical Context

Tools are registered in the registry with a unique name (scoped to workspace), human-readable description, JSON Schema input schema, optional output schema, and a handler function (async). The registry stores tool definitions in MongoDB for persistence and loads them into an in-memory cache for fast lookups. Tools can be registered programmatically (via SDK), via configuration, or via connector installation.

## Requirements

### Functional
- `registerTool(definition)` — register a new tool
  - `name`: unique within workspace (alphanumeric + hyphens)
  - `description`: human-readable description
  - `inputSchema`: JSON Schema draft 2020-12 for parameters
  - `outputSchema` (optional): JSON Schema for return value
  - `handler`: async function `(params, context) => Promise<result>`
  - `metadata`: `{ version, author, workspaceId, tags, categories, rateLimit, timeout }`
- `getTool(name, workspaceId)` — get tool definition by name (scoped to workspace)
- `listTools(workspaceId, filter?)` — list all tools in a workspace, with optional category/tag filter
- `updateTool(name, updates)` — update tool description, schema, or handler
- `deprecateTool(name, deprecationMessage, replacement?)` — mark tool as deprecated
- `removeTool(name, workspaceId)` — remove a tool (soft delete — mark as removed)
- `getToolVersions(name)` — list all versions of a tool
- Store tool definitions in MongoDB `tools` collection
- Cache active tools in-memory with 60-second TTL, invalidated on write
- Validate tool names: 1-64 chars, alphanumeric + hyphens, unique per workspace
- Validate inputSchema is valid JSON Schema
- Handle duplicate registration: return error with existing tool info
- Handle max tools per workspace: configurable limit (default 200)

### Non-functional
- Tool lookup (by name): < 5 ms (in-memory cache)
- Tool listing: < 20 ms for 200 tools
- Registration: < 50 ms (including MongoDB write)
- Cache TTL: 60 seconds for reads, immediate invalidation on writes

### Performance
- In-memory cache using Map (not Redis — tool definitions are small and rarely change)
- MongoDB compound index: `{ workspaceId: 1, name: 1 }` — unique compound key

### Security
- Tool names are workspace-scoped — tool `analyze-doc` in workspace A is distinct from the same name in workspace B
- Registration requires write permission on workspace
- Tool handler functions are isolated from the registry (no access to registry internals)

### Accessibility
- N/A (internal service)

## Architecture Constraints

- Tool names are immutable after creation (cannot be renamed)
- Tool versions follow semver: increment automatically on update unless explicitly specified
- The handler function is stored as part of the tool definition in memory (not in MongoDB)
- The registry is loaded on startup: all active tools from MongoDB are loaded into cache with their handlers
- Handler functions for first-party tools are imported statically; third-party tool handlers are loaded via the connector bridge
- Deprecated tools are still listed but marked with a deprecation warning

## Acceptance Criteria

- `registerTool` creates a tool definition and stores it in MongoDB
- `getTool` returns the correct tool by name (workspace-scoped)
- `listTools` returns all tools in a workspace
- `updateTool` updates description and schema; version increments
- `deprecateTool` marks tool as deprecated with message
- `removeTool` soft-deletes (marks removed)
- Same tool name in different workspaces are independent
- Invalid JSON Schema input returns clear validation error
- Duplicate registration in same workspace returns error
- Cache returns tool definitions within 5 ms

## Notes

- Tool MongoDB schema: `{ _id, workspaceId, name, description, inputSchema, outputSchema?, metadata: { version, author, tags, categories, createdAt, updatedAt, deprecated: boolean, deprecationMessage?, removed: boolean }, handler: string }` — handler is stored as a reference path, not the function itself
- For first-party tools, handler is a module path like `@repo/mcp/tools/analyze-doc`
- For third-party tools (connectors), the handler reference is the connector name + tool name
- In-memory cache: `Map<string, ToolDefinition>` keyed by `{workspaceId}:{toolName}`
- Cache invalidation: on any write operation, delete the cache entry for the affected tool
