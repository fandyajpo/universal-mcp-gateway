## Goal

Implement the HTTP API routes for the MCP Gateway that expose the MCP protocol over HTTP SSE transport and provide REST endpoints for tool management. These routes are the entry point for MCP clients connecting to the gateway.

## Business Context

The MCP Gateway needs to be accessible over the internet for remote MCP clients (e.g., Claude Desktop connecting to the enterprise gateway). The HTTP API routes provide SSE transport for real-time communication and REST endpoints for tool management operations.

## Technical Context

The API routes live in `apps/web` as Next.js App Router route handlers. The SSE transport route establishes a persistent connection for MCP protocol messages. Management routes provide CRUD operations for tool definitions via REST. All routes require authentication and workspace membership.

## Requirements

### Functional
- `GET /api/mcp/sse` — SSE transport endpoint
  - Establishes SSE connection for MCP protocol messages
  - Sends `endpoint` event with the POST endpoint URL
  - Keeps connection alive with periodic keepalive comments
  - Detects client disconnect and cleans up
- `POST /api/mcp/message` — Message endpoint (used by SSE transport)
  - Receives JSON-RPC messages from client
  - Delegates to protocol handler
  - Response may be JSON (for non-streaming) or the message is forwarded via SSE
- `POST /api/mcp/tools` — Register a new tool (REST)
  - Accepts tool definition JSON body
  - Validates schema
  - Returns created tool with ID
- `PUT /api/mcp/tools/:name` — Update a tool (REST)
  - Accepts partial tool definition updates
  - Returns updated tool
- `DELETE /api/mcp/tools/:name` — Remove/soft-delete a tool (REST)
  - Returns confirmation
- `GET /api/mcp/tools/:name/versions` — List tool versions (REST)
  - Returns version history
- All routes require:
  - Authentication (valid session)
  - Workspace membership
  - Rate limiting (adjustable per route)

### Non-functional
- SSE connection setup: < 100 ms
- Message endpoint latency: protocol handler overhead + tool execution + < 10 ms route overhead
- Tool management endpoints: < 200 ms

### Performance
- SSE connections: support up to 1000 concurrent connections per pod
- Message processing: up to 100 messages per second per connection

### Security
- Authentication on all routes
- Session validation on every SSE event (not just connection open)
- Rate limiting: 100 messages/second per connection
- Request body size limit: 10 MB (tool definitions can be large)
- CORS: restrict to configured origins

### Accessibility
- N/A (API endpoints)

## Architecture Constraints

- SSE transport: the `endpoint` event tells the client where to POST messages: `event: endpoint\ndata: /api/mcp/message?sessionId={id}\n\n`
- Each SSE connection gets a unique sessionId used to correlate messages
- The message endpoint uses the sessionId to find the corresponding SSE connection and forward responses
- Tool management endpoints follow REST conventions but are secondary to MCP protocol endpoints
- Rate limiting is per-session for SSE/message, per-user for REST endpoints

## Acceptance Criteria

- SSE connection establishes and sends `endpoint` event
- POST to message endpoint with valid JSON-RPC message returns response via SSE
- Tool registration via REST creates tool in registry
- Tool update via REST updates tool and increments version
- Tool deletion via REST soft-deletes tool
- Unauthenticated requests return 401
- Rate limited requests return 429
- SSE connection survives keepalive pings for 30+ minutes
- Client disconnect is detected within 30 seconds

## Notes

- SSE connection flow:
  1. Client opens `GET /api/mcp/sse`
  2. Server creates session, sends: `event: endpoint\ndata: /api/mcp/message?sessionId=abc123\n\n`
  3. Client POSTs JSON-RPC messages to `/api/mcp/message?sessionId=abc123`
  4. Server processes and sends responses via the SSE connection
- Session cleanup: use a `Map<sessionId, { transport, createdAt }>` with periodic cleanup of stale sessions (> 5 minutes idle)
- CORS: set `Access-Control-Allow-Origin` to configured frontend URL; for SSE, also set `Access-Control-Allow-Credentials: true`
