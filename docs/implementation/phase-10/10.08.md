## Goal

Implement the MCP tool discovery endpoints that enable clients to discover available tools and their capabilities. The discovery API returns tool definitions with input schemas, output schemas, metadata, and version information, filtered by workspace permissions and authentication.

## Business Context

AI models discover tools via the `tools/list` and `tools/call` MCP methods. The discovery API must return accurate, permission-filtered tool listings so that models can select appropriate tools. It also provides the human-facing developer portal with detailed tool documentation.

## Technical Context

The discovery endpoints are implemented as MCP protocol methods (`tools/list`, `tools/call`) and as REST API endpoints for developer tooling. The MCP methods are handled by the protocol handler (Step 10.01) and delegated to the registry (Step 10.02). The REST endpoints provide richer documentation with version history, examples, and usage statistics.

## Requirements

### Functional
- MCP `tools/list` method:
  - Returns all tools the current user has permission to use
  - Each tool includes: `name`, `description`, `inputSchema`, `outputSchema?`, `metadata: { categories, tags, version, supportsStreaming, deprecated?, deprecationMessage? }`
  - Support cursor-based pagination (cursor token in request, nextCursor in response)
  - Support filtering by category and tag
  - Respect workspace scoping: return only tools in the current workspace
  - Respect RBAC: return only tools the user's role can access
- MCP `tools/call` method:
  - Accept `name` and `arguments` (validated against inputSchema)
  - Execute tool in sandbox (Step 10.04)
  - Return result with `{ content: [{ type: "text" | "json" | "image", text? | json? | image? }], isError?: boolean }`
- REST `GET /api/mcp/tools` — list tools with full details
  - Returns all metadata: name, description, versions, input/output schemas, examples, usage stats
  - Supports same filters as MCP method plus status filter (active, deprecated, all)
- REST `GET /api/mcp/tools/:name` — get tool details
  - Returns full tool definition with all versions and metadata
- REST `GET /api/mcp/tools/:name/versions` — list versions
- Rate limiting: 30 `tools/list` requests per minute, 100 `tools/call` requests per minute per user
- Authentication: all endpoints require valid session
- Authorization: RBAC enforced per tool (some tools may be admin-only)

### Non-functional
- tools/list response: < 100 ms for 200 tools
- tools/call response: depends on tool execution; protocol overhead < 10 ms
- REST tool listing: < 200 ms including permission filtering

### Performance
- Pagination uses cursor-based (cursor is last tool ID, sorted by name)
- Tool definitions are cached in-memory (from registry cache)

### Security
- Authentication required on all endpoints
- Tool-level authorization: user must have `tool:execute:{toolName}` permission
- Workspace isolation: tools from other workspaces are never visible

### Accessibility
- API documentation should follow OpenAPI standards for REST endpoints

## Architecture Constraints

- MCP methods use JSON-RPC request/response format
- REST endpoints use standard HTTP JSON responses
- RBAC is enforced by checking user permissions against tool metadata.requiredPermissions
- Pagination: cursor is the last tool's `_id` from MongoDB; results sorted alphabetically by name
- The `tools/call` method is the only way to execute tools — no direct execution API

## Acceptance Criteria

- `tools/list` returns all tools the user can access, with correct schemas
- `tools/list` pagination returns nextCursor when more results exist
- `tools/list` with category filter returns only matching tools
- `tools/call` with valid params executes tool and returns result
- `tools/call` with invalid params returns validation error
- `tools/call` with unauthorized tool returns authorization error
- REST `GET /api/mcp/tools` returns full tool list with metadata
- REST `GET /api/mcp/tools/:name` returns tool details with versions
- RBAC enforcement: admin-only tools are hidden from non-admin users
- Rate limit exceeded returns error with Retry-After

## Notes

- MCP `tools/list` request: `{ method: "tools/list", params: { cursor?, category?, tag? } }`
- MCP `tools/list` response: `{ tools: ToolDefinition[], nextCursor?: string }`
- MCP `tools/call` request: `{ method: "tools/call", params: { name: string, arguments: object } }`
- MCP `tools/call` response: `{ content: [{ type: "text", text: "..." }] }` or `{ content: [...], isError: true }` on tool error
- Tool output content types: `text` for string results, `json` for structured data, `image` for base64-encoded images
