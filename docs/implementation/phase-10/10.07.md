## Goal

Implement comprehensive error handling for the MCP Gateway, covering tool execution errors, protocol errors, validation errors, sandbox errors, and unexpected failures. All errors are mapped to structured JSON-RPC error responses with appropriate error codes and actionable messages.

## Business Context

Clear, structured error handling is critical for MCP protocol compliance and developer experience. AI models receiving tool error responses need to understand what went wrong and how to fix it. Ambiguous or missing error information leads to repeated failed attempts and poor user experience.

## Technical Context

Errors flow through multiple layers: protocol parsing → method routing → parameter validation → tool execution → sandbox enforcement → result serialization. Each layer produces typed errors that are caught and mapped to JSON-RPC error responses. Error codes follow both JSON-RPC 2.0 standard codes and MCP-specific extension codes.

## Requirements

### Functional
- Define error classes for each error category:
  - `ProtocolError` — JSON-RPC parse/request errors (-32700, -32600, -32601)
  - `ValidationError` — parameter validation failures (-32602)
  - `InternalError` — unexpected server errors (-32603)
  - `ToolNotFoundError` — tool not found in registry (-32001, MCP extension)
  - `ToolExecutionError` — tool handler threw an error (-32002, MCP extension)
  - `ToolTimeoutError` — tool execution timed out (-32003, MCP extension)
  - `ToolMemoryError` — tool exceeded memory limit (-32004, MCP extension)
  - `ToolNetworkError` — tool tried to access blocked network endpoint (-32005, MCP extension)
  - `ToolOutputTooLargeError` — tool output exceeded size limit (-32006, MCP extension)
  - `ToolRateLimitedError` — tool rate limit exceeded (-32007, MCP extension)
  - `AuthenticationError` — not authenticated (-32008, MCP extension)
  - `AuthorizationError` — not authorized for this tool (-32009, MCP extension)
- `handleError(error): ErrorResponse` — map any error to JSON-RPC error response
- All error responses include: `{ code, message, data?: { details, errorId, toolName? } }`
- Error ID: UUID v4 generated for each error, logged for traceability
- Sensitive error details (stack traces, internal paths) are never included in error responses
- Stack traces are logged at DEBUG level (not sent to client)
- Unhandled errors (unexpected exceptions) are caught by a global error boundary and mapped to InternalError
- Tool handler errors: catch thrown errors, wrap in ToolExecutionError with handler's error message
- Sandbox enforcement errors (timeout, memory, network) produce typed errors with clear messages
- Rate limiting errors include `Retry-After` metadata in error data
- All errors are logged with errorId, workspaceId, toolName, and error code

### Non-functional
- Error mapping: < 1 ms per error
- Error logging: async (fire-and-forget)
- Global error boundary catches all unhandled rejections and exceptions

### Performance
- Error handling never involves API calls or database writes (logging is async)
- Error response is generated synchronously

### Security
- No stack traces in client-facing error messages
- No internal paths, IP addresses, or server details in error messages
- Error messages are reviewed for information leakage

## Architecture Constraints

- The global error boundary wraps the entire protocol handler: any unhandled exception in request processing is caught and mapped to InternalError
- Tool handler errors are caught by the sandbox and wrapped in ToolExecutionError
- All errors must have a unique errorId (UUID v4) for traceability
- Error logging includes: `errorId, code, message, stack (debug only), workspaceId, toolName, userId`

## Acceptance Criteria

- JSON-RPC parse errors return ParseError (-32700) with clear message
- Unknown method returns MethodNotFound (-32601) with method name
- Invalid tool parameters return InvalidParams (-32602) with specific field errors
- Unknown tool returns ToolNotFound (-32001) with available similar tools (if any)
- Tool handler error returns ToolExecutionError (-32002) with handler's error message
- Tool timeout returns ToolTimeoutError (-32003) with timeout duration
- Tool memory exceeded returns ToolMemoryError (-32004) with limit
- Network blocked returns ToolNetworkError (-32005) with blocked endpoint
- Unhandled exception returns InternalError (-32603) with errorId
- All error responses include errorId for traceability
- Stack traces logged but not returned to client

## Notes

- JSON-RPC error code ranges:
  - Standard: -32700 to -32000 (JSON-RPC 2.0)
  - Server errors: -32000 to -32099 (MCP extensions)
  - Application errors: positive codes or custom ranges
- ErrorId generation: `import { v4 as uuidv4 } from "uuid";`
- The global error boundary:
  ```typescript
  process.on("unhandledRejection", (reason) => {
    const error = reason instanceof Error ? reason : new Error(String(reason));
    const errorId = uuidv4();
    logger.error({ errorId, error }, "Unhandled rejection");
  });
  ```
