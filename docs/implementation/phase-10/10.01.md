## Goal

Implement the MCP (Model Context Protocol) protocol handler that manages JSON-RPC 2.0 message parsing, transport abstraction (SSE and stdio), method routing, and response formatting. The handler is the entry point for all MCP communication and must fully conform to the MCP specification.

## Business Context

MCP is an open protocol that allows AI models to discover and invoke tools. The gateway implements the server side of MCP, exposing tools and resources to any MCP-compatible client (Anthropic's Claude Desktop, custom agents, etc.). Protocol compliance is critical ŌĆö non-compliant implementations would fail to connect with MCP clients.

## Technical Context

MCP uses JSON-RPC 2.0 as its message protocol. The handler receives JSON-RPC requests, validates the JSON-RPC structure (jsonrpc, method, params, id), routes to the appropriate method handler, and returns JSON-RPC responses. Transport abstraction allows the same handler to work over SSE (HTTP), stdio (subprocess), or WebSocket. The handler manages request/response correlation via the `id` field and handles batch requests and notifications.

## Requirements

### Functional
- Parse incoming JSON-RPC 2.0 messages from raw text
- Validate message structure: `jsonrpc: "2.0"`, valid `method` string, valid `id` (string|number|null)
- Route requests to registered method handlers based on `method` field
- Supported MCP methods (initial):
  - `initialize` ŌĆö protocol initialization and capability negotiation
  - `ping` ŌĆö health check
  - `tools/list` ŌĆö list available tools
  - `tools/call` ŌĆö invoke a tool
  - `notifications/initialized` ŌĆö client notification that initialization is complete
  - `notifications/cancelled` ŌĆö client notification that a request is cancelled
- Return properly formatted JSON-RPC responses: `{ jsonrpc: "2.0", id, result: {...} }` or `{ jsonrpc: "2.0", id, error: { code, message, data? } }`
- Handle JSON-RPC error codes: -32700 (Parse error), -32600 (Invalid Request), -32601 (Method not found), -32602 (Invalid params), -32603 (Internal error)
- Support notifications (requests without `id`) ŌĆö no response sent
- Support request cancellation via `notifications/cancelled` ŌĆö propagate cancellation to running tool execution
- Transport abstraction: define `Transport` interface that the handler uses to read/write messages
- SSE transport: HTTP server with SSE for outgoing messages, POST endpoint for incoming messages
- Stdio transport: read from stdin, write to stdout (for subprocess MCP clients)
- Log all JSON-RPC messages at DEBUG level (request method, response success/error)
- Handle malformed JSON gracefully ŌĆö return Parse Error (-32700)

### Non-functional
- Message parsing overhead: < 1 ms per message
- Response formatting: < 1 ms
- Concurrent message handling: support up to 50 concurrent requests

### Performance
- Message buffer: max 1000 pending responses per connection

### Security
- Input validation: reject messages with excessively long strings (> 100KB per field)
- Error messages must not leak internal implementation details
- Rate limit: max 100 messages per second per connection

### Accessibility
- N/A (protocol implementation)

## Architecture Constraints

- The handler is transport-agnostic ŌĆö it communicates via the `Transport` interface only
- JSON-RPC `id` is preserved in the response for request-response correlation
- Notifications (no `id`) never produce responses
- The `initialize` method must be the first message received; all other methods return error until initialized
- Error codes follow JSON-RPC 2.0 specification exactly
- The handler must handle batch requests (array of requests) ŌĆö returns array of responses

## Acceptance Criteria

- Valid JSON-RPC request returns correctly formatted response
- Invalid JSON returns Parse Error (-32700)
- Unknown method returns Method Not Found (-32601)
- Invalid params returns Invalid Params (-32602)
- Notifications (no `id`) produce no response
- `initialize` method returns capabilities
- `ping` method returns `{}` result
- SSE transport receives messages via POST and sends responses via SSE
- Stdio transport reads from stdin and writes to stdout
- Request cancellation stops in-progress execution
- Message logging includes method, ID, and success/error status

## Notes

- JSON-RPC request: `{ "jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {} }`
- JSON-RPC response: `{ "jsonrpc": "2.0", "id": 1, "result": { "tools": [...] } }`
- JSON-RPC error: `{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32601, "message": "Method not found" } }`
- For SSE transport, the client sends POST to `/mcp` with JSON-RPC messages, and receives responses on an SSE stream established at `/mcp/sse`
- Stdio transport is used by desktop MCP clients that spawn the gateway as a subprocess
