## Goal

Implement the execution sandbox that runs tool handler functions in an isolated, secure environment. The sandbox enforces timeout, memory limits, output size caps, and restricts network access. It provides a consistent execution context with injected tenant ID, user ID, and trace ID for audit and logging.

## Business Context

Tool handlers execute arbitrary code (defined by the tool author). Without sandboxing, a buggy or malicious tool could crash the gateway, consume excessive resources, or access unauthorized data. The sandbox ensures that tool execution is safe, bounded, and observable.

## Technical Context

The sandbox uses Node.js `worker_threads` for CPU isolation, `AbortController` for timeout enforcement, and a proxy-based context injection for resource limits. Each tool execution runs in a separate worker thread with:
- Timeout: `AbortController` that aborts after timeoutMs (default 30s, configurable per tool)
- Memory limit: watchdog that monitors worker heap and terminates if exceeded
- Output limit: truncation of tool output exceeding maxResponseBytes (default 1 MB)
- Network access: fetch wrapper that enforces allowlist of allowed endpoints
- Context injection: tenant ID, user ID, trace ID, workspace ID available in handler

## Requirements

### Functional
- `executeTool(tool, params, context): Promise<ExecuteResult>` — execute tool in sandbox
- Sandbox isolation:
  - Each execution runs in a dedicated worker thread
  - Worker thread has its own V8 isolate and event loop
  - Worker cannot access parent process resources (filesystem, environment variables)
- Timeout enforcement:
  - Default timeout: 30 seconds (configurable per tool via metadata)
  - Abort the worker thread if timeout exceeded
  - Return `{ error: { code: "TIMEOUT", message: "Tool execution timed out" } }`
- Memory limit:
  - Default limit: 128 MB heap (configurable per tool)
  - Monitor heap usage every 100 ms during execution
  - Terminate worker if heap exceeds limit, return `{ error: { code: "MEMORY_EXCEEDED", message: "..." } }`
- Output limit:
  - Default limit: 1 MB serialized JSON (configurable per tool)
  - Truncate output if exceeds limit, return with `{ truncated: true, data: ... }`
  - Log warning on truncation
- Network restriction:
  - By default: no network access
  - Tool can declare `allowedEndpoints` in metadata (array of URL patterns)
  - Fetch wrapper checks endpoint against allowlist before making request
  - Non-allowed requests are rejected with `{ error: { code: "NETWORK_BLOCKED" } }`
- Context injection:
  - `context.workspaceId`, `context.userId`, `context.traceId`, `context.requestId`
  - `context.logger` — scoped logger with trace context
  - `context.env` — limited environment variables (named allowlist)
- Result handling:
  - Tool handler can return any JSON-serializable value
  - Non-JSON-serializable results (Functions, Symbols, circular references) are caught and error is returned
  - Errors thrown by handler are caught and returned as structured MCP errors
- Audit logging: every tool execution is logged with workspaceId, userId, toolName, duration, success/failure

### Non-functional
- Worker thread startup: < 50 ms (reuse worker pool to avoid cold start)
- Execution overhead (sandboxing): < 5 ms beyond handler execution time
- Worker pool: maintain pool of 10 idle workers, scale up to 50 under load

### Performance
- Worker pool: pre-allocate workers, avoid creating/destroying per execution
- Worker reuse: after execution, worker is returned to pool (not terminated)
- Heap monitoring: check every 100 ms, overhead < 1%

### Security
- Workers run in separate V8 isolates — no shared memory access
- Worker cannot access `process.env` directly — only injected context.env
- Filesystem access: none (read-only to a temp directory if explicitly needed)
- Network allowlist enforced at the fetch wrapper level

### Accessibility
- N/A (internal sandbox)

## Architecture Constraints

- Worker threads communicate with the main thread via `postMessage` (structured clone)
- Tool handler function must be serializable for transfer to worker: use `vm.Script` to evaluate handler code in worker
- The worker pool maintains idle workers; on execution request, pick an idle worker, send execution message, await result, return worker to pool
- If all workers are busy, queue execution (FIFO) with max queue depth 100
- Worker termination (on timeout or memory limit) is forceful: `worker.terminate()` — worker is removed from pool, new worker is created

## Acceptance Criteria

- Tool executes successfully within sandbox and returns correct result
- Timeout (30s) terminates worker and returns timeout error
- Memory limit (128 MB) exceeded terminates worker and returns memory error
- Output larger than 1 MB is truncated with `truncated: true` flag
- Network access to non-allowed endpoint returns network blocked error
- Allowed endpoints are reachable
- Context injection: handler receives workspaceId, userId, traceId
- Worker pool reuses workers: 10 executions reuse same worker without restart
- Audit log entry written for each execution with duration and status

## Notes

- Worker thread communication:
  ```typescript
  // main.ts
  const worker = new Worker("./worker.js");
  worker.postMessage({ tool: toolDefinition, params, context });
  worker.on("message", (result) => { ... });
  worker.on("error", (err) => { ... });
  ```
- For timeout: create `AbortController` with timeout; on timeout, call `worker.terminate()`
- Memory monitoring: in the worker, use `setInterval(() => { const usage = process.memoryUsage(); if (usage.heapUsed > limit) process.exit(1); }, 100)`
- Network allowlist: the fetch wrapper in the worker checks `allowedEndpoints.some(pattern => url.match(pattern))`
- Worker pool: simple array of available workers; on demand, shift first available; on return, push back
