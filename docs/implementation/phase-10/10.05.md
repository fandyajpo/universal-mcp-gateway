## Goal

Implement streaming responses for long-running tool executions. Tools that produce progressive results (e.g., search tools with incremental results, processing tools with status updates) can emit progress notifications via the MCP protocol, enabling the client to display intermediate results before completion.

## Business Context

Some tools take significant time to complete (e.g., "search through 10K documents" or "analyze a large codebase"). Without streaming, the client must wait for the entire result, creating a poor UX. Streaming tool responses allow the client to display progress, show partial results, and provide a responsive experience for slow tools.

## Technical Context

The MCP protocol supports `tool/call:progress` notifications that the server can emit during tool execution. These notifications are JSON-RPC notifications (no `id`) with a `progress` field containing the current value and total value. Tool handlers emit progress by calling a `progressCallback` injected into the execution context.

## Requirements

### Functional
- Tool handler receives a `reportProgress(current, total, message?)` function in execution context
- Handler calls `reportProgress(5, 10, "Processing page 5 of 10")` to emit progress
- Progress notifications are emitted as JSON-RPC notifications:
  ```json
  { "jsonrpc": "2.0", "method": "tool/call:progress", "params": { "toolCallId": "...", "progress": 5, "total": 10, "message": "Processing page 5 of 10" } }
  ```
- Progress notifications are sent to the client via the transport layer
- Tool declares `supportsStreaming: true` in its metadata to indicate streaming support
- Non-streaming tools: `reportProgress` is a no-op (never called by the handler)
- Streaming tools: the handler can emit 0 to N progress notifications before the final result
- Progress notifications are asynchronous — they don't block handler execution
- The handler can also emit partial result notifications with `type: "partial"`:
  ```json
  { "jsonrpc": "2.0", "method": "tool/call:progress", "params": { "toolCallId": "...", "type": "partial", "data": { ... } } }
  ```
- Transport layer (SSE) forwards notifications immediately without buffering
- Client can cancel a streaming tool execution mid-stream (cancellation request stops the handler)
- Progress notifications respect rate limits: max 10 notifications per second per tool call

### Non-functional
- Notification emission overhead: < 1 ms per notification
- No notification buffering: sent immediately via transport
- Max 1000 progress notifications per tool execution (safety limit)

### Performance
- Notifications are sent via the transport's `send()` method — no separate connection
- Progressive results are not persisted to MongoDB (only final result is stored)

### Security
- Progress messages must not contain sensitive data (they may be logged)
- Progress rate limits prevent notification floods

- N/A (protocol implementation)

## Architecture Constraints

- The `reportProgress` function is injected via the execution context (not a global or import)
- Progress notifications use JSON-RPC notification format (no `id`, no response expected)
- The handler emits progress asynchronously — the main thread forwards via transport
- If the transport connection is closed mid-stream, remaining notifications are silently dropped
- The progress rate limiter uses a simple token bucket (10 tokens, refill 10/second)

## Acceptance Criteria

- Tool handler calling `reportProgress(5, 10)` emits a progress notification to the client
- Progress notification has correct JSON-RPC format (no `id`, method `tool/call:progress`)
- Multiple progress notifications are sent in order for sequential progress updates
- Partial result notifications deliver intermediate data to the client
- Non-streaming tool's `reportProgress` is a no-op
- Streaming tool correctly declares `supportsStreaming: true`
- Cancelling a streaming tool stops progress notifications
- Progress rate limit (10/sec) prevents notification floods
- Final result is delivered after last progress notification

## Notes

- Progress notification format:
  ```json
  {
    "jsonrpc": "2.0",
    "method": "notifications/tool/call:progress",
    "params": {
      "toolCallId": "call_abc123",
      "progress": 5,
      "total": 10,
      "message": "Processing..."
    }
  }
  ```
- In the execution context: `context.reportProgress = (current, total, message) => transport.send(progressNotification)`
- For partial results: `context.reportPartial = (data) => transport.send({ jsonrpc: "2.0", method: "notifications/tool/call:partial", params: { toolCallId, data } })`
