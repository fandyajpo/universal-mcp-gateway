## Goal

Implement semantic versioning for tool definitions, enabling multiple versions of the same tool to coexist. Versioning supports backward-compatible updates (patch/minor) and breaking changes (major) with clear migration paths. Clients can request specific tool versions or use version ranges.

## Business Context

Tools evolve over time — schemas change, handlers improve, dependencies update. Without versioning, updating a tool would break existing clients that depend on the current interface. Versioning allows safe rolling updates and lets clients opt into new versions at their own pace.

## Technical Context

Tool versions follow Semantic Versioning (semver): `MAJOR.MINOR.PATCH`. Breaking changes (schema incompatible) require MAJOR bump. New optional parameters or expanded capabilities use MINOR. Bug fixes and performance improvements use PATCH. The registry stores all versions of a tool, and the protocol handler resolves version requests using semver ranges.

## Requirements

### Functional
- Every tool registration creates version 1.0.0
- `updateTool(name, updates)` auto-increments version:
  - PATCH: if only handler changes (no schema change)
  - MINOR: if optional fields added to input schema or description changes
  - MAJOR: if required fields removed, field types changed, or breaking schema changes
- Tools can specify version explicitly: `registerTool({ ..., version: "2.0.0" })`
- All versions are stored in MongoDB with full history
- `getTool(name, versionRange?)` resolves version range (default: latest):
  - `"1.0.0"` — exact version
  - `"^1.0.0"` — compatible with 1.x (>=1.0.0, <2.0.0)
  - `"~1.0.0"` — approximately equivalent (>=1.0.0, <1.1.0)
  - `"latest"` — highest version
  - `"*"` — any version (highest)
- `listToolVersions(name)` — returns all versions of a tool
- `deprecateVersion(name, version, message)` — deprecate specific version
- Breaking changes detection: compare input schemas to detect removed required fields or changed types
- Migration path: when a tool has a newer MAJOR version, deprecated versions show a migration message with link to the new version
- Clients that don't specify version get the latest non-deprecated version
- Store version metadata: `{ version, createdAt, deprecationMessage?, migrationUrl?, changelog? }`

### Non-functional
- Version resolution: < 2 ms (semver range comparison is fast)
- Full version history stored in MongoDB; cache only latest version in memory

### Performance
- Version history uses a separate MongoDB collection or embedded array
- Tool listing returns latest version by default; version history shown on explicit request

### Security
- Version manipulation requires the same workspace permissions as tool registration
- Deprecated versions still execute (clients may depend on them)

- N/A (internal service)

## Notes

- Auto-increment logic:
  ```typescript
  function determineBump(oldSchema, newSchema, handlerChanged, descriptionChanged): "major" | "minor" | "patch" {
    if (isBreakingChange(oldSchema, newSchema)) return "major";
    if (handlerChanged || descriptionChanged) return "minor"; // handler change is at least minor
    if (schemasDiffer(oldSchema, newSchema)) return "minor"; // schema changed but not breaking
    return "patch";
  }
  ```
- Breaking change definition: removed required field, changed field type, added constraint that invalidates existing data
- MongoDB schema: store versions as an array in the tool document: `{ versions: [{ version, schema, handlerRef, createdAt, deprecationMessage? }], currentVersion: "2.0.0" }`
- The current version is always the latest non-deprecated version
