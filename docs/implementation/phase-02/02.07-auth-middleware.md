# Step

Step ID: 02.07

Step Name: Auth Middleware

Estimated Time: 1 hour

Priority: Critical

Difficulty: Medium

Dependencies: 02.01, 02.06

---

## Goal

Create the Next.js Edge Middleware that validates sessions on every request, resolves the tenant (workspace) context, enforces public/protected route configuration, and redirects unauthenticated users to the login page. The middleware runs at the edge for minimal latency.

---

## Requirements

1. Create `apps/web/src/middleware.ts`:
   - Reads session cookie (`umgw_session`)
   - Validates session via Redis cache (fast) or MongoDB (fallback)
   - Resolves `workspaceId` from session
   - Attaches session and workspace context to request headers
   - Checks route permissions (public vs protected)
   - Redirects to `/login` if unauthenticated on protected route
   - Redirects to `/chat` if authenticated on public route (login page)
2. Route configuration: `publicRoutes` (login, register, password-reset, verify-email, api/auth/*), `authRoutes` (login, register), `apiRoutes` (api/*), `protectedRoutes` (everything else).
3. Edge-compatible: middleware must NOT use Node.js-specific APIs (fs, path, stream).
4. Rate limiting on auth routes in middleware (10 req/min per IP for login/register).
5. Security headers: CSP, X-Frame-Options, X-Content-Type-Options, HSTS, Permissions-Policy.

---

## Acceptance Criteria

1. Unauthenticated request to `/chat` redirects to `/login`.
2. Authenticated request to `/login` redirects to `/chat`.
3. API routes (`/api/*`) validate session in middleware.
4. Session cookie is read and validated.
5. Security headers are present on all responses.
6. Rate limiting blocks excessive auth requests.
7. Middleware does not block static assets (`/_next/*`, `/favicon.ico`).

---

## Notes

Next.js middleware runs on the Edge runtime — use `@repo/cache`'s Upstash Redis client (HTTP-based, edge-compatible). The middleware is intentionally thin — it only validates sessions and redirects. Heavy permission checks happen in the service layer. The route config is a shared constant that can be imported by both middleware and components for consistent route knowledge.
