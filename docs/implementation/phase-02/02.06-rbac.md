## Goal

Implement the role-based access control (RBAC) framework with 4 roles (owner, admin, member, viewer), a permission registry, and enforcement helpers for middleware, service layer, and UI. The RBAC system is workspace-scoped: a user can have different roles in different workspaces.

---

## Requirements

1. Define role hierarchy: `owner > admin > member > viewer`.
2. Define permission set in `@repo/types` (expandable): `workspace:read`, `workspace:write`, `workspace:delete`, `workspace:chat`, `workspace:admin`, `connector:read`, `connector:write`, `connector:install`, `agent:read`, `agent:write`, `agent:deploy`, `rag:read`, `rag:write`, `rag:index`, `admin:users`, `admin:billing`, `admin:audit`.
3. Create role-permission mapping: each role has a set of permissions. Owner has all permissions. Viewer has only `workspace:read`, `connector:read`, `agent:read`, `rag:read`.
4. Create `RBACService`: `checkPermission(userId, workspaceId, permission)` returns boolean. Uses Redis cache for fast permission checks.
5. Create `requirePermission(permission)` — higher-order function for Route Handlers and Server Actions that checks permission and returns 403 if unauthorized.
6. Create `usePermissions()` — React hook (via Zustand) for UI-level permission gating.
7. Permissions cached in Redis with TTL 5 minutes, invalidated on role change.

## Acceptance Criteria

1. `checkPermission(userId, workspaceId, "workspace:admin")` returns `true` for owner/admin roles.
2. `checkPermission(userId, workspaceId, "workspace:admin")` returns `false` for member/viewer roles.
3. `requirePermission("workspace:admin")` returns 403 for unauthorized requests.
4. Permission check is cached in Redis — subsequent checks are sub-10ms.
5. Permission cache invalidated when user role changes.
6. `usePermissions()` hook returns `{ can, isLoading, error }` object.

---

## Notes

Role assignments are stored in the `WorkspaceMember` MongoDB collection (field: `role`). The permission-to-role mapping is a static object — no database calls needed for the mapping itself. The RBAC service is tenant (workspace) aware — permissions are always checked within a workspace context.
