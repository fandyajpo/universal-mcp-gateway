## Goal

Implement session management with refresh token rotation, session caching in Redis via `@repo/cache`, configurable session TTL, multi-session support (user logged in on multiple devices), and session invalidation on password change or admin force-logout.

---

## Requirements

1. Session token stored in HTTP-only, Secure, SameSite=Strict cookie.
2. Session data cached in Redis via `@repo/cache` with TTL (default: 7 days).
3. Session refresh: when session is within 24h of expiry, automatically refresh (rotate).
4. Multi-session: user can have up to 10 concurrent sessions (configurable).
5. Session invalidation: change password or admin force-logout invalidates all sessions except current.
6. Session cleanup: TTL index on MongoDB sessions collection auto-removes expired.
7. Session verification: check session validity on every request (middleware), cache hit in Redis.
8. Session creation includes `workspaceId` (current active workspace), `ipAddress`, `userAgent`.

---

## Files To Create

- packages/auth/src/services/session-service.ts
- packages/auth/src/services/session-cache.ts

---

## Acceptance Criteria

1. Login creates a session and sets the session cookie.
2. Session is verified from Redis on subsequent requests (fast path).
3. Redis miss falls back to MongoDB lookup (slow path).
4. Session refreshes when within 24h of expiry (token rotation).
5. Invalidate sessions on password change.
6. Enforce max 10 concurrent sessions — oldest session evicted.
7. Expired sessions are automatically cleaned up.

---

## Notes

Session cookie name: `umgw_session`. Token: 64 random bytes, base64url encoded. Redis key: `tenant:{workspaceId}:session:{token}`. The session cache in Redis stores serialized session data with the same TTL as the session. MongoDB stores sessions with a TTL index for automatic cleanup.
