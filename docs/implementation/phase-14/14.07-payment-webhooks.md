## Goal

Build a robust Stripe webhook handler that processes all payment-related events reliably. Implement idempotent event processing, error handling with retry and dead-letter queue, event replay capability, webhook health monitoring, and comprehensive audit logging.

---

## Business Context

Stripe webhooks are the mechanism by which Stripe notifies the platform about payment events: successful payments, failed payments, subscription changes, invoice readiness, and more. Reliable webhook processing is critical for billing accuracy. A missed webhook could mean a customer is not billed, a subscription is not activated, or an invoice is not generated. The webhook system must be resilient, observable, and auditable.

---

## Technical Context

The webhook endpoint receives POST requests from Stripe at `/api/billing/stripe/webhook`. Each request includes a Stripe-Signature header that must be verified using the webhook signing secret. Events are processed by type, with dedicated handlers for each event type. Processing is idempotent — the same event ID can be safely processed multiple times. Failed events are stored in a `webhook_events` collection for debugging and replay.

## Requirements

### Functional requirements

- Verify Stripe webhook signature on every request
- Route events to type-specific processors: `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.paid`, `invoice.payment_failed`, `invoice.created`, `customer.created`, `customer.updated`, `checkout.session.completed`, `charge.refunded`, `payment_intent.succeeded`, `payment_intent.payment_failed`
- Enforce idempotent processing: skip events that have already been processed successfully
- Store all received events in a `webhook_events` collection for audit
- Handle processing failures gracefully: retry up to 3 times with exponential backoff, then move to dead-letter queue
- Provide admin UI for viewing webhook events (success, failed, pending) and manual replay
- Expose webhook health metrics: events received, processed, failed, pending retry
- Support event replay: re-process a specific event or batch of events
- Log every webhook event with full details for debugging
- Send alert on critical webhook failures (payment failures, subscription deletions)

### Non-functional requirements

- Webhook processing must complete within 10 seconds per event
- Event store must retain events for 90 days
- Idempotency check must be atomic (use MongoDB findAndModify or unique index)
- Webhook endpoint must respond with 200 quickly — processing can be asynchronous

### Performance requirements

- Signature verification: < 10ms
- Idempotency check: < 20ms
- Event processing: < 5s per event (async for complex operations)
- Event store query: < 200ms
- Event replay processing: < same as initial processing

### Security requirements

- Webhook signature must be verified using `stripe.webhooks.constructEvent()`
- Signature verification must fail closed — invalid signatures return 400
- Webhook secret must be different between environments (dev/staging/production)
- Event payload must never be logged in its entirety (truncate sensitive fields)
- Replay endpoint must require `admin:billing` permission

### Accessibility requirements

- Webhook event table in admin UI must be keyboard navigable
- Event status indicators must have text labels

## Architecture Constraints

- Webhook endpoint must return a 200 response immediately after signature verification and idempotency check — actual event processing runs asynchronously
- Idempotency uses a MongoDB unique index on `eventId` — attempting to process the same event twice will fail the insert
- Dead-letter queue is a separate collection `webhook_dead_letter` with the original event, error details, and retry count
- Event replay creates a new processing attempt with the same event ID — idempotency ensures no duplicate side effects
- Webhook processing order matters: subscription events must be processed before invoice events for the same subscription

## Notes

The most critical webhook events are `checkout.session.completed` (activates subscription), `invoice.payment_failed` (triggers dunning), and `customer.subscription.deleted` (deactivates features). Monitor these specifically and alert on failures. Use Stripe's event ID as the idempotency key and store it in MongoDB with a unique index. The TTL on the `webhook_events` collection should match the data retention policy (90 days). For development, use the Stripe CLI to forward events to your local endpoint.
