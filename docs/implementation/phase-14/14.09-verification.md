## Goal

Run the complete verification suite for Phase 14: Billing. Execute all tests, verify type safety, lint compliance, build integrity, and perform full manual end-to-end testing of the billing system using Stripe test mode. Create integration tests that verify the complete billing lifecycle.

---

## Business Context

Billing is the most financially sensitive part of the platform. Errors in billing calculations, subscription state management, or webhook processing can result in revenue loss, customer dissatisfaction, or compliance issues. Comprehensive verification is essential — this is not just a quality gate but a financial controls check. Every billing flow must be tested with Stripe test mode to verify real payment processing behavior.

---

## Technical Context

Verification follows the project's standard pattern but with additional focus on Stripe integration testing. Tests use Stripe test mode with test API keys. The verification includes: unit tests for all billing services, integration tests for Stripe API interactions, webhook event simulation, and end-to-end flows that exercise the complete billing lifecycle from checkout to invoice payment.

## Requirements

### Functional requirements

- Run all unit tests from steps 14.01-14.08 and verify 100% pass rate
- Run Stripe integration tests in test mode (no real charges)
- Simulate complete billing lifecycle: checkout → subscription active → usage tracking → invoice generation → payment → invoice paid
- Test subscription lifecycle: create, upgrade (immediate), downgrade (scheduled), cancel, reactivate, trial expiration
- Test webhook event processing for all event types
- Test idempotent webhook processing (replay same event)
- Test error scenarios: payment failure, expired card, insufficient funds, webhook signature mismatch
- Verify billing portal UI renders correctly for all states
- Verify admin billing overview shows correct data
- Test plan limit enforcement: exceeding limit blocks the action
- Test data isolation: workspace A cannot see workspace B's billing data
- Verify TypeScript strict mode with no errors across billing packages

### Non-functional requirements

- Test suite must complete in under 15 minutes
- Stripe test mode must be used — never use production API keys
- No test data should persist after test suite completion (cleanup)
- Test coverage must exceed 85% on billing service code

### Performance requirements

- Unit tests: < 2 minutes
- Stripe integration tests: < 8 minutes (API call latency)
- Type check: < 3 minutes
- Build: < 5 minutes

### Security requirements

- Verify that billing API routes require admin or workspace-owner permissions
- Verify that webhook signature verification rejects invalid signatures
- Verify that subscription data is workspace-scoped
- Verify that no Stripe secret keys are exposed in client-side code

### Accessibility requirements

- Quick accessibility check of billing portal pages

## Architecture Constraints

- Stripe test mode keys must be configured in `.env.test` — never use production keys
- Test Stripe data must be cleaned up after test completion (delete test customers, subscriptions)
- Webhook tests simulate Stripe events with test signatures — do not forward real events to test
- Integration tests must use a separate test database
