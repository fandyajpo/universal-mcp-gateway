## Goal

Implement metered billing for AI token consumption. Compute overage charges when a workspace exceeds its plan's included token allowance, generate metered usage line items in Stripe, and track costs by model tier.

---

## Business Context

AI token consumption is the primary variable cost for the platform. Each workspace has a monthly included token allowance based on their plan tier. Usage beyond the allowance incurs additional charges at a per-token rate that varies by model tier (cheap models cost less, premium models cost more). This metered billing model aligns the platform's costs with revenue and ensures heavy users pay proportionally.

---

## Technical Context

Metered billing builds on the usage tracking from step 14.04. At the end of each billing period, the metered billing service: (1) queries total token usage for the period by model tier, (2) subtracts the plan's included allowance, (3) computes overage charges at the configured per-tier rates, (4) creates metered usage records in Stripe via the `usage_record` API, and (5) stores the computed charges in a `metered_charges` collection for invoicing.

Model tier classification: `economy` (Claude Haiku, GPT-4o-mini), `standard` (GPT-4o, Claude Sonnet), `premium` (GPT-4, Claude Opus), `embedding` (text-embedding-3-large). Each tier has a separate usage counter and overage rate.

## Requirements

### Functional requirements

- Categorize token usage by model tier (economy, standard, premium, embedding)
- Track included vs overage tokens separately for each tier
- Compute overage charges at end of billing period using tier-specific rates
- Support multiple overage pricing models: per-token (most common), per-1K-tokens, per-1M-tokens
- Create Stripe usage records for metered billing (fire-and-forget during the period, or batch at period end)
- Store computed metered charges in `metered_charges` collection with full audit trail
- Support mid-period plan changes: prorate included allowance based on days in each plan
- Provide real-time overage estimate: "You have used X% of your included tokens and accumulated $Y in overage charges so far this period"
- Handle zero-usage periods correctly (no metered charges)
- Support model tier exclusion (some plans may not have access to premium models)

### Non-functional requirements

- Metered charge computation must complete within 5 minutes for the largest workspace (10M+ tokens/month)
- Real-time overage estimates must be computed in under 1 second (cached)
- Stripe usage records are eventually consistent — reconcile on next invoice
- All overage calculations must be auditable with full breakdown

### Performance requirements

- Real-time overage estimate: < 500ms (cached, refreshed every 5 minutes)
- End-of-period metered charge computation: < 5 minutes for all workspaces
- Historical metered charges query: < 1s

### Security requirements

- Overage calculations must use integer arithmetic to avoid floating-point rounding errors (use cents/microcents)
- Metered charge records must be immutable after the billing period closes
- Only the billing service can create metered charges

### Accessibility requirements

- Usage breakdown by tier must be displayed in tabular format
- Cost displays must use proper currency formatting

## Architecture Constraints

- Model tier classification must use the model's identifier from OpenRouter — maintain a mapping file in `@repo/ai`
- Overage rates are defined in the pricing plan, not hardcoded
- Mid-period plan changes prorate included allowance: `(days_remaining / total_days) * allowance`
- Stripe usage records are submitted with a timestamp matching the usage period — do not batch-summit at period end
- Metered charge records are created at period end and marked as `pending`, then `invoiced`, then `paid`

## Notes

Floating-point arithmetic is not suitable for financial calculations. Always use integer math: represent token counts as integers, rates as microcents per token (1 microcent = $0.00000001), and convert to dollars only for display. The model tier mapping must be kept in sync with the models available through OpenRouter — add a periodic sync job or manual update process.
