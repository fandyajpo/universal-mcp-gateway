## Goal

Build the customer-facing billing portal UI in the web app that allows workspace admins to view their current plan, usage statistics, invoice history, and manage payment methods. The portal provides self-service billing management without requiring admin intervention.

---

## Business Context

The billing portal is the self-service interface for workspace billing. Users should be able to see what plan they are on, how much they are using versus their plan limits, view and download invoices, update payment methods, upgrade/downgrade plans, and cancel subscriptions. A well-designed billing portal reduces support tickets and gives users control over their billing.

---

## Technical Context

The billing portal is a set of pages in `apps/web` under `/workspace/[id]/billing`. It uses data from the subscription service (14.03), usage tracking (14.04), invoicing (14.06), and Stripe customer portal (14.02). The Stripe customer portal handles payment method management — the app redirects users there for sensitive operations. Plan changes (upgrade/downgrade) are initiated through the app UI and handled by the subscription service.

## Requirements

### Functional requirements

- Display current plan card: plan name, price, billing period, status (active/trialing/canceled/past_due), next billing date
- Show usage summary for each tracked metric: AI tokens, storage, API calls, with current usage vs limit and percentage bar
- Display detailed usage breakdown by model tier for AI tokens
- Show overage estimate for current period (if applicable)
- Display plan comparison table when considering upgrade/downgrade
- Initiate plan upgrade: show available plans, confirm, redirect to Stripe Checkout (if upgrade) or schedule (if downgrade)
- Initiate plan cancellation with confirmation flow and feedback collection
- Display invoice list with: invoice number, period, amount, status, download PDF link
- Provide link to Stripe Customer Portal for payment method management
- Show billing contact information and update form
- Display trial status with days remaining
- Show past_due banner with link to update payment method

### Non-functional requirements

- Page must load in under 2 seconds with all data
- Usage bars must animate smoothly on load
- Plan change confirmation must clearly communicate proration details
- All Stripe portal links must open in same tab (no popup blockers)
- Usage data must auto-refresh every 60 seconds

### Performance requirements

- Initial page load: < 2s
- Usage data refresh: < 500ms
- Plan comparison table: < 1s
- Invoice list load: < 1s

### Security requirements

- Only workspace admins and owners can access billing page
- Plan changes require current subscription verification
- Cancellation requires a confirmation step
- Payment method changes redirect to Stripe — card data never touches the server

### Accessibility requirements

- Usage percentage bars must have accessible labels
- Plan comparison table must be keyboard navigable
- Status indicators must use icons and text
- Color coding must not be the only differentiator

## Package Boundaries

The billing portal depends on `@repo/billing` (subscription, usage, invoice services), `@repo/ui` (Card, Progress, Table, Button, Dialog, Select), `@repo/auth` (permission check), and `@repo/validation` (plan change schemas). No direct Stripe dependencies in the UI.

---

## Architecture Constraints

- Plan change flow: for upgrades, create Stripe Checkout session and redirect; for downgrades, schedule the change via the subscription service
- The Stripe Customer Portal is used for payment method changes — never implement card collection in the app
- Invoice PDFs are served via signed R2 URLs with 1-hour expiration
- Usage data is fetched from the usage service, not directly from the database
- All API routes must check workspace admin permission before returning data
