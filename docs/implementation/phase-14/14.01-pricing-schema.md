## Goal

Design and implement the pricing schema, plan tier definitions, feature mapping, and validation logic. This establishes the data model that the entire billing system is built upon.

---

## Business Context

The pricing schema defines what the platform charges for and how. It must support multiple plan tiers (Free, Pro, Business, Enterprise) with different feature sets, usage limits, and prices. The schema must be flexible enough to accommodate future pricing changes and new plans. It is the single source of truth for what each plan includes, what limits apply, and how much each feature costs.

---

## Technical Context

Pricing data is stored in a `pricing_plans` collection in MongoDB and synchronized with Stripe products and prices. The schema defines plan metadata, feature flags, usage limits, and price definitions. Plan tiers are referenced by subscription records and used throughout the application for feature gating and limit enforcement.

## Requirements

### Functional requirements

- Define plan tiers: Free, Pro ($29/mo), Business ($99/mo), Enterprise (custom pricing)
- Each plan must define: name, slug, description, monthly price, annual price (discount), feature list, usage limits (AI queries/month, storage GB, API calls/day, workspace count, member count, connector count), feature flags (chat, rag, mcp, connectors, api_keys, sso, audit_logs, custom_branding)
- Support add-on pricing: additional storage ($10/GB/mo), additional AI queries ($5/1k queries), additional members ($3/member/mo)
- Support both monthly and annual billing periods with configurable discount percentage
- Support per-unit and flat-rate pricing models
- Support enterprise plan with custom pricing (no fixed price, quote-based)
- Plans must be configurable via seed data — not hardcoded
- Support plan archival (cannot be selected for new subscriptions but existing ones continue)
- Include a `features` map that gates access to platform capabilities

### Non-functional requirements

- Schema must be forward-compatible — adding a new plan or feature must not require code changes
- Default plan assignments must be configurable (e.g., new users get Free plan)
- Plan limits must be queried in under 10ms (cached in Redis)

### Performance requirements

- Plan lookup: < 10ms (Redis cache with 1-hour TTL)
- Feature check: < 5ms (in-memory)
- Limit calculation: < 20ms

### Security requirements

- Plan data is read-only for non-admin users
- Only super-admins can create, modify, or archive plans
- Price modifications must be audited

### Accessibility requirements

- Price displays must use proper semantic markup
- Currency formatting must respect locale

---

## Architecture Constraints

- Plan definitions must never be imported as magic strings or hardcoded values — always use the repository or service
- The feature map must use a consistent naming convention: `feature:<domain>:<capability>` (e.g., `feature:chat:streaming`, `feature:rag:indexing`)
- Limit keys must follow the pattern: `limit:<resource>:<metric>` (e.g., `limit:ai:queries_per_month`)
- Annual pricing discount must be configurable per plan — not a global constant

## Notes

The plan seed data should be stored as a TypeScript file in `@repo/billing/src/plans/seed.ts`. Changes to pricing should go through the seed file and be applied via the migration system. Do not use Stripe as the source of truth for feature definitions — Stripe products/prices represent financial data only. Feature gating is driven by the local plan definitions. Enterprise plan pricing is null/quote-based — the sales team handles custom quotes outside the platform.
