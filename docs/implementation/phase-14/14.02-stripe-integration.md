## Goal

Integrate Stripe as the payment processor. Set up the Stripe SDK client, configure products and prices in Stripe that mirror the local pricing schema, implement the Stripe customer portal for self-service billing management, and build the webhook endpoint for processing Stripe events with full idempotency.

---

## Business Context

Stripe handles all financial transactions: payment processing, subscription management, invoicing, and customer portal. This integration is the backbone of the billing system. A robust Stripe integration with proper error handling, idempotency, and reconciliation is critical for financial accuracy and customer trust.

---

## Technical Context

The Stripe integration uses the official `stripe` npm package with TypeScript types. The client is initialized in `@repo/billing` with the secret key from `@repo/config`. All Stripe API calls go through a `StripeClient` abstraction that handles retries, logging, and error mapping. Webhook events are processed by a dedicated handler that verifies signatures and processes events idempotently.

## Requirements

### Functional requirements

- Initialize Stripe client with secret key from environment
- Create/update Stripe products for each local pricing plan
- Create/update Stripe prices for monthly and annual billing periods
- Sync Stripe product/price IDs back to local plan records
- Create Stripe customers for each workspace (with metadata)
- Generate Stripe Checkout session URLs for subscription creation
- Generate Stripe Customer Portal session URLs for self-service management
- Validate Stripe webhook signatures on every event
- Process webhook events idempotently using Stripe's Idempotency-Key
- Handle webhook event types: `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.paid`, `invoice.payment_failed`, `customer.created`, `customer.updated`
- Log all Stripe API calls and webhook events for audit
- Implement webhook event replay mechanism for failed events

### Non-functional requirements

- Stripe API calls must timeout after 30 seconds
- Retry failed Stripe API calls with exponential backoff (max 3 retries)
- Webhook processing must be idempotent — processing the same event twice has no side effects
- Stripe client must never log sensitive data (card numbers, CVC, etc.)

### Performance requirements

- Checkout session creation: < 2s
- Customer portal session: < 1s
- Webhook processing: < 5s per event

### Security requirements

- Stripe secret key must never be logged or exposed to the client
- Webhook signing secret must be verified on every event
- Stripe API calls must use idempotency keys for all mutation requests
- Customer portal sessions must expire after 24 hours
- Checkout success/cancel URLs must validate the session ID

### Accessibility requirements

- Checkout is handled by Stripe's hosted pages — no UI concerns
- Error messages returned to users must be user-friendly (not raw Stripe errors)

## Package Boundaries

The Stripe integration lives entirely within `@repo/billing`. The webhook route handler in `apps/web` is a thin wrapper that delegates to the billing package. The billing package now depends on the `stripe` npm package.

---

## Architecture Constraints

- StripeClient must be the only code that imports from the `stripe` package — no other code calls Stripe directly
- Webhook events must be processed in order per customer (sequential processing within a customer scope, parallel across customers)
- Idempotency is tracked in a MongoDB collection `webhook_idempotency` with a TTL of 24 hours
- Failed webhook events are stored in `webhook_failures` for manual replay
- Checkout session creation requires a `success_url` and `cancel_url` that include the session ID for verification

## Notes

Stripe test mode must be used for all development and CI testing. The `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` environment variables must be configured in `.env` for the target environment. The Stripe CLI (`stripe listen --forward-to localhost:3000/api/billing/stripe/webhook`) is essential for local webhook testing. Store the Stripe webhook event ID in the idempotency collection to prevent duplicate processing. This is especially important for the `invoice.paid` event which could cause duplicate billing.
