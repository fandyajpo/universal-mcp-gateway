## Goal

Implement the usage tracking service that records and aggregates usage metrics across AI tokens, storage, API calls, and other billable resources. Usage data feeds into metered billing (step 14.05) and displays in the billing portal (step 14.08).

---

## Business Context

Usage tracking is the data foundation for metered billing. Every AI query, storage byte, and API call must be counted against the workspace's plan limits and billing tier. Accurate usage tracking ensures customers are billed correctly for what they consume. Usage data also powers analytics dashboards, anomaly detection, and capacity planning.

---

## Technical Context

Usage records are stored in a MongoDB `usage_events` collection as immutable time-series data. Each event has: `workspaceId`, `metric` (ai_tokens, storage_bytes, api_calls, connector_syncs), `value` (numeric amount), `unit` (tokens, bytes, requests), `timestamp`, `metadata` (model name, document ID, etc.). Aggregated usage is computed via MongoDB aggregation pipelines and cached in Redis.

The AI cost tracker from step 09.06 already captures per-request token usage. This step integrates with that data source and adds tracking for storage and API usage.

## Requirements

### Functional requirements

- Record usage events for: AI tokens (input + output), storage (bytes stored + bandwidth), API calls (by endpoint), connector sync operations (by connector type), document processing (pages processed)
- Support real-time event recording (write path) with < 100ms latency
- Support batch event recording for high-volume metrics (AI tokens)
- Aggregate usage by billing period (monthly), workspace, and metric type
- Compute current period usage against plan limits
- Support usage rollup: daily snapshots stored for historical queries
- Expose usage query API: current period usage, historical trend (12 months), usage by metric breakdown, usage by day chart data
- Handle usage spikes gracefully — do not block the recording thread
- Support usage data backfill for historical data migration

### Non-functional requirements

- Event recording must not block the request path — fire-and-forget or batch queue
- Usage aggregation queries must complete in under 2 seconds for monthly data
- Daily snapshots must be computed via Inngest cron job
- Usage data retention: 24 months for aggregated data, 90 days for raw events

### Performance requirements

- Single event recording: < 50ms
- Batch event recording (100 events): < 200ms
- Current period usage query: < 500ms
- Historical trend query (12 months): < 2s

### Security requirements

- Usage data is workspace-scoped — a workspace can only see its own usage
- Usage recording API must authenticate the request
- Usage data must not expose individual user activity — only workspace aggregates

### Accessibility requirements

- Usage charts must provide data in tabular format for screen readers
- Color-coded usage bars must include text labels

## Notes

The usage tracking system must handle high-throughput scenarios gracefully. AI token events can arrive at hundreds per second during peak usage. Use batch inserts with a buffer that flushes every 5 seconds or 1000 events, whichever comes first. The daily snapshot cron job should process workspaces in batches to avoid long-running aggregation queries. Consider using MongoDB's `$merge` aggregation stage for efficient upsert of daily snapshots.
