## Goal

Implement the subscription management service that handles the complete subscription lifecycle: creation, upgrade, downgrade, cancellation, reactivation, trial management, and plan change scheduling.

---

## Business Context

Subscriptions are the core revenue mechanism. Users must be able to sign up for a paid plan, upgrade or downgrade their plan as their needs change, cancel when they leave, and reactivate if they return. The subscription management service must handle all these flows correctly, coordinating between local state and Stripe, handling edge cases like proration, and ensuring data consistency.

---

## Technical Context

Subscriptions are stored in a `subscriptions` MongoDB collection and synchronized with Stripe subscriptions. Each subscription record tracks: `workspaceId`, `planId`, `stripeSubscriptionId`, `stripeCustomerId`, `status` (active, past_due, canceled, trialing, incomplete), `currentPeriodStart`, `currentPeriodEnd`, `canceledAt`, `trialEnd`, `metadata`. The subscription service uses the Stripe integration from step 14.02 and the pricing service from step 14.01.

## Requirements

### Functional requirements

- Create subscription: after successful Stripe checkout, create local subscription record, activate plan features, send confirmation email
- Upgrade plan: change to higher-tier plan mid-cycle with prorated credit, activate new features immediately, send upgrade confirmation
- Downgrade plan: schedule downgrade at end of current billing period, deactivate features at period end, send downgrade notification
- Cancel subscription: cancel at period end (no refund), keep access until period end, send cancellation confirmation
- Immediate cancellation with refund: for enterprise-managed cancellations, process refund through Stripe
- Reactivate subscription: reactivate before cancellation takes effect (within same period), or create new subscription
- Trial management: track trial period, send expiration warnings at 7 days, 3 days, and 1 day before trial ends, auto-convert to paid plan or cancel at trial end
- Plan change scheduling: store pending plan changes that take effect at next billing period
- Handle past_due subscriptions: send payment failure notifications, apply grace period, suspend after grace period
- Sync subscription status from Stripe webhook events

### Non-functional requirements

- Subscription status must be eventually consistent with Stripe (within 30 seconds via webhook)
- Read path (check subscription status) must be fast (< 50ms) — use Redis cache
- Write path (create/change subscription) goes through Stripe first, then local database
- All subscription changes must be recorded in the audit log

### Performance requirements

- Subscription status lookup: < 50ms (cached)
- Plan change processing: < 3s
- Trial expiration check: < 5s for all active trials (batch job)

### Security requirements

- Only workspace admins/owners can change subscription
- Plan downgrades that would exceed usage limits must warn the user
- Refund operations require `admin:billing` permission
- Subscription data must be isolated per workspace

### Accessibility requirements

- Subscription status indicators must use text labels in addition to color
- Plan comparison tables must be keyboard navigable

## Package Boundaries

The API routes in `apps/web` delegate to the billing service.

---

## Architecture Constraints

- Subscription creation must be initiated through Stripe Checkout — no direct subscription creation
- Local subscription record is created in the Stripe webhook handler (`checkout.session.completed`), not in the checkout redirect
- Plan changes that take effect immediately (upgrades) must update both Stripe and local state atomically
- Plan changes that take effect at period end (downgrades) only update local `pendingPlanId` — Stripe handles the actual change
- Past_due subscriptions must be checked on every request — if past grace period (3 days), reject all paid features

## Notes

The subscription service must handle the edge case where Stripe's webhook arrives before the user is redirected back to the application (or vice versa). The `checkout.session.completed` handler must be idempotent. Trial expiration notifications should use Inngest scheduled functions. Consider implementing a "downgrade warning" flow that checks current usage against the target plan's limits before allowing a downgrade.
