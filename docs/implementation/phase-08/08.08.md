## Goal

Implement REST API routes that expose the RAG Engine's capabilities to clients. The routes provide query endpoints for retrieving context, endpoints for document ingestion tracking (checking indexing status), and admin endpoints for index management. All routes require authentication and workspace membership.

## Business Context

The RAG Engine must be accessible from multiple consumers: the Chat UI (Phase 12), the AI Gateway (when it needs grounded responses), and potentially third-party integrations. REST API routes provide a standardized interface that any authenticated client can use.

## Technical Context

The API routes follow Next.js App Router conventions and are implemented as Route Handlers in `apps/web`. They delegate to the RAG Engine in `@repo/rag` for business logic. Routes use Zod validation (from `@repo/validation`) for request/response validation and auth middleware (from `@repo/auth`) for authentication.

## Requirements

### Functional
- `POST /api/rag/query` — RAG query endpoint
  - Request body: `{ query: string, workspaceId: string, options?: { strategy?, rerank?, topK?, topN?, filters?, conversationHistory?, systemInstructions? } }`
  - Response: `{ context: string, chunks: RetrievalResult[], tokenCount: number, pipelineMetadata: object }`
  - Auth: required, workspace membership required
- `GET /api/rag/status/:documentId` — document indexing status
  - Response: `{ documentId, status: "indexed" | "processing" | "failed", chunksCount, lastIndexedAt }`
  - Auth: required, workspace membership required
- `POST /api/rag/reindex/:documentId` — trigger re-indexing of a document
  - Response: `{ documentId, status: "reindexing" }`
  - Auth: required, workspace write permission required
- `DELETE /api/rag/index/:documentId` — delete document index (chunks)
  - Response: `{ documentId, chunksDeleted: number }`
  - Auth: required, workspace write permission required
- All routes return standard error format: `{ error: { code: string, message: string, details?: object } }`
- Support streaming response for queries (SSE or JSON, depending on content negotiation)
- Rate limiting: 30 RAG queries per minute per user

### Non-functional
- Query endpoint latency: < 3 seconds p95 (matching engine performance)
- Status endpoint latency: < 100 ms p95
- All endpoints: < 50 ms overhead for auth + validation + routing

### Performance
- Rate limiting: token bucket, 30 req/min per user, burst 5
- Pagination for status list endpoints (not needed for MVP, but consider future)

### Security
- All endpoints require valid session token
- Workspace membership validated for every request
- Query and document IDs validated for injection attacks
- Rate limit per user, not per IP (user can be on multiple devices)

### Accessibility
- N/A (API endpoints)

## Architecture Constraints

- The query route is thin — it validates input, calls engine, formats response
- Streaming: if client sends `Accept: text/event-stream`, the route returns SSE (for future use with chat)
- Rate limiting uses the `@repo/cache` Redis-based rate limiter
- The route must handle errors from the engine: validation errors → 400, auth errors → 401, engine errors → 500
- Document status/reindex/delete routes interact with the vector store service (08.02)

## Acceptance Criteria

- `POST /api/rag/query` with valid query returns 200 with context and metadata
- `POST /api/rag/query` without auth returns 401
- `POST /api/rag/query` with invalid body returns 400 with validation error details
- `GET /api/rag/status/:documentId` returns correct indexing status
- `POST /api/rag/reindex/:documentId` triggers re-indexing
- `DELETE /api/rag/index/:documentId` deletes chunks and returns count
- Rate limit exceeded returns 429 with Retry-After header
- All endpoints return consistent error format

## Notes

- Route handler pattern:
  ```typescript
  export async function POST(request: NextRequest) {
    const session = await auth();
    if (!session) return NextResponse.json({ error: { code: "UNAUTHORIZED", message: "..." } }, { status: 401 });
    const body = await request.json();
    const parsed = querySchema.safeParse(body);
    if (!parsed.success) return NextResponse.json({ error: { code: "VALIDATION_ERROR", message: "...", details: parsed.error } }, { status: 400 });
    const result = await ragEngine.query(parsed.data.query, { workspaceId: session.workspaceId, ...parsed.data.options });
    return NextResponse.json(result);
  }
  ```
- Workspace membership validation: check that the user's session includes the requested workspaceId
