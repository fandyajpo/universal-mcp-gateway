## Goal

Implement a cross-encoder re-ranker service that takes the top-K retrieval results (typically 20-50) and re-scores them against the original query using a cross-encoder model. The re-ranker significantly improves precision by evaluating query-chunk pairs jointly, rather than relying on the embedding similarity (which is computed independently for query and chunk). The re-ranker returns the top-N results (typically 3-10) after re-ranking.

## Business Context

Vector similarity search and keyword search are efficient but imprecise — they use bi-encoder representations that lose query-chunk interaction information. A cross-encoder re-ranker evaluates each query-chunk pair together, producing much more accurate relevance scores. This is a standard technique in production RAG systems, typically improving top-5 relevance by 15-25% over raw retrieval.

## Technical Context

The re-ranker uses a cross-encoder model accessed through OpenRouter (or a local model if available). It sends the query and each chunk text as a pair, receives a relevance score (0-1), and re-ranks results by this score. Since cross-encoder inference is expensive (O(n) API calls for n chunks), the re-ranker operates on a limited candidate set (top K from retrieval). The service includes batching (multiple pairs per API call if supported), caching of re-ranking results, and a fallback to raw retrieval scores if the re-ranker API is unavailable.

## Requirements

### Functional
- `rerank(query, results, options)` — re-rank an array of retrieval results
- Accept results from any retriever strategy (vector, keyword, hybrid)
- For each result, compute a cross-encoder relevance score against the query
- Return results sorted by cross-encoder score (highest first)
- Support configurable candidate set size: `topK` (default 20, max 50) — number of results sent to re-ranker
- Support configurable final result count: `topN` (default 5, max topK) — number of results to return
- Support batching: send up to 10 query-chunk pairs per API call (if model supports batch)
- Cache re-ranking results: key = `sha256(query + chunkContent)`, TTL = 1 hour
- Fallback: if re-ranker API is unavailable, return results sorted by original retrieval score with a warning
- Include re-ranker metadata in results: `originalScore`, `rerankScore`, `scoreImprovement`
- Log re-ranker performance: candidates sent, API latency, score distribution

### Non-functional
- Re-ranking latency: < 2 seconds for topK=20 (including API calls)
- Re-ranker cache hit: < 10 ms per cached pair
- Fallback activation within 100 ms of API failure detection

### Performance
- Batch re-ranking: process candidates in parallel batches of 10
- Maximum 2 concurrent API calls to avoid rate limiting

### Security
- Query and chunk content sent to re-ranker API — ensure no PII in content
- API calls logged with content length metadata only (not content itself)

### Accessibility
- N/A (internal service)

## Architecture Constraints

- Primary re-ranker model: Cohere `rerank-english-v3.0` (accessed via OpenRouter or direct Cohere API)
- Fallback re-ranker model: `cross-encoder/ms-marco-MiniLM-L-6-v2` (if available locally)
- Each API call sends query + up to 10 candidate texts; receives relevance scores per candidate
- If batching is not supported by the model, send individual requests with concurrency limit of 2
- Re-ranker cache is separate from embedding cache — different key structure, different TTL
- The fallback path (no re-ranker) should be transparent to the caller — results still come back with original scores

## Acceptance Criteria

- 20 candidate chunks re-ranked to top 5 with cross-encoder scores
- Re-ranked top 5 are more relevant than raw retrieval top 5 (validated on test queries)
- Batch re-ranking sends up to 10 pairs per API call
- Cache hit returns re-ranking score in < 10 ms
- Re-ranker API failure triggers graceful fallback to original scores
- Re-ranker metadata included in each result
- Performance: 20 candidates re-ranked in < 2 seconds
- Same query+chunk pair gets cached score on second request
