## Goal

Implement the context window builder that takes re-ranked retrieval results and constructs an optimized prompt context for the LLM. The builder manages token budgets (allocating between context, instructions, and conversation history), formats chunks with source citations, handles overflow (truncation with sliding window), and produces a structured context object ready for injection into the AI Gateway prompt.

## Business Context

LLMs have finite context windows (typically 8K-200K tokens depending on model). The context window builder must maximize the useful information packed into that window while ensuring the model receives clear instructions and relevant conversation context. Poor context management leads to truncated responses, lost information, or confused models.

## Technical Context

The builder takes the top-N re-ranked chunks, the original query, conversation history, and system instructions. It computes token counts for each component, allocates the token budget (configurable percentages), and assembles the context in a structured format. It handles cases where the total exceeds the budget by truncating lowest-ranked chunks first, then conversation history, while always preserving system instructions.

## Requirements

### Functional
- `buildContext(query, results, options)` — build context from retrieval results
- Accept configuration:
  - `modelMaxTokens` — the target model's maximum context window (e.g., 128000 for claude-sonnet-4)
  - `budgetAllocation` — percentage breakdown: `{ context: 70, instructions: 15, history: 15 }` (configurable)
  - `maxChunks` — maximum number of chunks to include (default 10)
  - `includeCitations` — whether to add citation markers (default true)
  - `includeScores` — whether to add relevance scores in context (default false for production)
- Format each chunk with:
  - Source document title and ID
  - Section path (e.g., "2.3 > Architecture > Components")
  - Page number(s)
  - Text content (with citation marker like `[1]`, `[2]`)
  - Relevance score (optional)
- Build conversation history section (abbreviated — include only last N exchanges to fit budget)
- Build instructions section (system prompt, task description, formatting instructions)
- Assemble final context string with clear section separators
- Handle overflow: truncate lowest-score chunks first, then oldest conversation history
- Never truncate system instructions (guaranteed allocation)
- Return context object: `{ context: string, tokenCount: number, chunksUsed: number, truncated: boolean, truncationDetails: { reason, removedChunks, removedHistory } }`
- Support sliding window for very long conversations: summarize oldest exchanges into a single condensed entry

### Non-functional
- Context assembly time: < 100 ms for up to 20 chunks
- Token counting accuracy: within 5% of actual model tokenization
- Context token count never exceeds `modelMaxTokens * budgetAllocation.context / 100`

### Performance
- Token counting uses gpt-tokenizer (fast, local, no API calls)
- Chunk sorting by score for truncation priority

### Security
- Context includes only chunks from authorized workspace
- Source document titles are sanitized (no injection vectors)

### Accessibility
- N/A (internal service, output consumed by LLM)

## Architecture Constraints

- Token counting uses `gpt-tokenizer` (GPT family tokenizer) — it's fast and accurate for most models
- Budget allocation is hard-coded with sensible defaults but overridable via configuration
- The context output must be compatible with the AI Gateway prompt input format
- Truncation prefers sacrificing conversation history over retrieved context (context is more valuable)
- The sliding window approach: when conversation history exceeds budget, summarize the oldest half into a 1-2 sentence summary, keep the most recent exchanges intact
- Citation markers are sequential `[1]`, `[2]`, etc., corresponding to the order chunks appear in context

## Acceptance Criteria

- Context built from 10 chunks fits within 70% of 128K token budget
- Truncation removes lowest-score chunks first when budget exceeded
- Conversation history truncated before retrieved context when budget exceeded
- System instructions always included in full
- Truncation metadata accurately reports what was removed and why
- Context formatting includes source citations for each chunk
- Token count reported is within 5% of actual model tokenization
- Context assembly completes in < 100 ms for 20 chunks
- Sliding window correctly summarizes oldest conversation exchanges
