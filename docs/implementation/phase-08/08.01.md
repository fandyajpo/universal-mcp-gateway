## Goal

Create and configure MongoDB Atlas Vector Search indexes for storing and querying embedding vectors. This step defines the index schema (fields, dimensions, similarity metric), creates the index via MongoDB management commands or the Atlas API, verifies the index is queryable, and provides helper functions for vector search queries used by the vector store service (Step 08.02).

## Business Context

MongoDB Atlas Vector Search is the core infrastructure for semantic retrieval. Without the correct index configuration, vector similarity searches cannot execute. The index must be carefully configured to match the embedding model's output — wrong dimensions or similarity metric means the index is useless. This step ensures the infrastructure is ready for the vector store service.

## Technical Context

MongoDB Atlas Vector Search indexes are created as dedicated search indexes (separate from regular database indexes). They define which field contains the vector, the number of dimensions, and the similarity metric (cosine, euclidean, or dotProduct). For text-embedding-3-large (3072 dimensions) with cosine similarity, the index is created on the `chunks` collection. The index creation is done via MongoDB shell commands or the Atlas API — not through Mongoose schema.

## Requirements

### Functional
- Define vector search index for the `chunks` collection with:
  - Name: `chunk_vector_index`
  - Type: `vectorSearch`
  - Field: `embedding`
  - Dimensions: 3072 (for text-embedding-3-large primary model)
  - Similarity: `cosine`
  - Number of probes: configurable (default 10)
  - Dynamic: false (static index)
- Define a secondary vector search index for lower-dimension models:
  - Name: `chunk_vector_index_small`
  - Field: `embedding`
  - Dimensions: 1536 (for voyage-code-2 fallback)
  - Similarity: `cosine`
- Provide `createVectorSearchIndex()` function for programmatic index creation (via Atlas API or MongoDB shell)
- Provide `ensureVectorSearchIndexes()` that checks if indexes exist and creates them if missing
- Provide `dropVectorSearchIndex(name)` for index cleanup
- Provide vector search aggregation pipeline builder: `buildVectorSearchPipeline(vector, { limit, numCandidates, filter })`
- Include tenant-scoped pre-filter in all queries (workspaceId filter before vector search)
- Support dynamic `numCandidates` parameter (default: `limit * 10`, min 10, max 10000)

### Non-functional
- Index creation is a one-time operation — the `ensureVectorSearchIndexes()` function is called at application startup
- Index creation via Atlas API may take 5-15 minutes; the function should handle this asynchronously
- Vector search queries must complete within 500 ms p95

### Performance
- `numCandidates` default: `limit * 10` (e.g., limit=20, numCandidates=200)
- Index uses `nProbes: 10` for the base index (adjustable for performance tuning)

### Security
- Tenant isolation: every vector search query includes a mandatory `workspaceId` pre-filter
- Index configuration does not expose or embed credentials

### Accessibility
- N/A (infrastructure configuration)

## Architecture Constraints

- Vector Search indexes are NOT managed through Mongoose — they are native Atlas Search indexes
- The `embedding` field in the Chunk document must be an array of numbers (float64)
- The index is created on the Atlas cluster, not the local development database
- `ensureVectorSearchIndexes()` should be idempotent — safe to call multiple times
- The pipeline builder returns an aggregation pipeline array compatible with MongoDB `aggregate()`
- Pre-filter uses `$match` stage before `$vectorSearch` for tenant isolation

## Acceptance Criteria

- `ensureVectorSearchIndexes()` successfully creates both indexes on Atlas cluster
- Index creation is idempotent (calling twice doesn't error)
- `buildVectorSearchPipeline()` returns a valid MongoDB aggregation pipeline
- Vector search query with cosine similarity returns correct nearest neighbors
- Pre-filter with workspaceId correctly scopes results to a single workspace
- Search with `numCandidates=100` and `limit=10` returns exactly 10 results
- The pipeline can be used with MongoDB `collection.aggregate()` successfully

## Notes

- Vector Search index JSON (create via Atlas UI or API):
  ```json
  {
    "name": "chunk_vector_index",
    "type": "vectorSearch",
    "fields": [{
      "type": "vector",
      "path": "embedding",
      "numDimensions": 3072,
      "similarity": "cosine"
    }]
  }
  ```
- For the pre-filter, add before the `$vectorSearch` stage: `{ $match: { workspaceId: targetWorkspaceId } }`
- The aggregation pipeline for vector search: `[{ $vectorSearch: { queryVector, path, numCandidates, limit, index } }, { $match: ... }, { $project: ... }, { $limit: ... }]`
- `numCandidates` should be `limit * 10` for good recall; increase for better recall at cost of latency
