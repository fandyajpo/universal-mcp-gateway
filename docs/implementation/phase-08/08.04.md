## Goal

Implement hybrid search that combines vector similarity search with full-text keyword search using a weighted scoring algorithm. Hybrid search provides the best of both worlds: semantic understanding from vector search and precise term matching from keyword search. The implementation uses MongoDB Atlas Search's keyword index for the text portion and merges results algorithmically.

## Business Context

Vector search excels at semantic similarity but can miss results that require exact term matching (e.g., specific product codes, technical terms, or proper names). Keyword search catches exact matches that vector search might miss. Hybrid search combines both approaches to maximize recall and precision, which is critical for enterprise RAG where missing a relevant document can have serious consequences.

## Technical Context

MongoDB Atlas Search provides built-in full-text search capabilities with its Lucene-based index. The keyword search uses a regular Atlas Search index on the `chunks` collection with a standard analyzer. The hybrid approach: (1) run vector search and keyword search independently, (2) normalize scores from both, (3) combine using weighted sum (default 0.7 vector + 0.3 keyword), (4) deduplicate and re-rank. Configuration must support adjusting weights per query.

## Requirements

### Functional
- Create Atlas Search index on `chunks` collection for full-text search:
  - Name: `chunk_text_index`
  - Analyzer: `lucene.standard`
  - Field mappings: `content` (with search highlighting), `sectionPath`, `documentId`
- `hybridSearch(query, options)` — perform combined vector + keyword search
- Execute vector search and keyword search in parallel
- Normalize scores from both search paths to 0-1 range
- Combine scores with configurable weights: default `vectorWeight = 0.7`, `keywordWeight = 0.3`
- Accept per-query weight overrides via options
- Deduplicate results (same chunk from both searches)
- Re-rank combined results by weighted score
- Support all filters from the retriever (workspaceId, documentIds, metadataFilter) in both searches
- Support query expansion: optionally generate 2-3 alternative query phrasings via LLM (requires AI Gateway) for improved recall
- Return combined results in the same `RetrievalResult[]` format as vector-only retrieval
- Include `searchType` in result metadata indicating which search path(s) returned each chunk

### Non-functional
- Hybrid search latency: < 2 seconds p95 (parallel execution limits the slower of the two searches)
- Keyword search latency: < 500 ms p95
- Score normalization: min-max scaling across each result set

- Keyword search uses MongoDB aggregation with `$search` stage

### Security
- Both searches include `workspaceId` pre-filter
- Combined results scoped to workspace

## Architecture Constraints

- Keyword search uses `$search` aggregation stage (MongoDB Atlas Search), not `$text` (which is limited)
- The Atlas Search index is separate from the Vector Search index — both must exist on the collection
- Score normalization: min-max scaling for each set: `normalized = (score - minScore) / (maxScore - minScore)`
- Weighted score: `finalScore = vectorScore * vectorWeight + keywordScore * keywordWeight`
- Results with `finalScore` below `minScore` are filtered out after combination
- The parallel execution uses `Promise.all` — both searches must complete; if one fails, the other's results are still returned with a warning log

## Acceptance Criteria

- keywordSearch("specific term") returns chunks containing that exact term
- hybridSearch("semantic query") returns results from both search paths
- Vector-only results (no keyword match) still appear with reduced score
- Keyword-only results (no vector match) still appear with reduced score
- Duplicate results (same chunk from both searches) appear only once with combined score
- Weighted scoring: adjusting weights produces different rankings
- Per-query weight overrides work correctly
- All workspace and metadata filters apply to both searches
- Hybrid search latency < 2 seconds p95

## Notes

- Atlas Search index definition:
  ```json
  {
    "name": "chunk_text_index",
    "analyzer": "lucene.standard",
    "mappings": {
      "fields": {
        "content": { "type": "string", "analyzer": "lucene.standard" },
        "sectionPath": { "type": "string" },
        "documentId": { "type": "string" },
        "workspaceId": { "type": "string" }
      }
    }
  }
  ```
- Keyword search aggregation: `[{ $search: { index: "chunk_text_index", text: { query, path: "content" } } }, { $match: { workspaceId } }, { $limit: numCandidates }, { $project: { score: { $meta: "searchScore" }, content: 1, ... } }]`
- Score normalization is critical because vector scores (cosine) and keyword scores (TF-IDF based) are on different scales
- Consider adding Reciprocal Rank Fusion (RRF) as an alternative to weighted sum for better ranking quality
