## Goal

Implement the retriever service that mediates between query input and the vector store. The retriever embeds the query text, performs vector search, applies metadata filters, scores results, and returns ranked chunks. It provides a clean interface for the RAG Engine (Step 08.07) and supports pluggable retrieval strategies (vector-only, keyword, hybrid from Step 08.04).

## Business Context

The retriever is the entry point for all knowledge retrieval in the platform. Chat (Phase 12), agents, and API consumers call the retriever to find relevant context for AI model queries. The retriever must be fast, accurate, and tenant-aware — returning the most relevant results from the correct workspace's document corpus.

## Technical Context

The retriever orchestrates the retrieval pipeline: (1) embed the query text using the embedding service (07.01), (2) build the vector search query with workspace filter and optional metadata filters, (3) execute the search against the vector store (08.02), (4) score and rank results, (5) return top-K chunks with metadata. The retriever is model-agnostic — it doesn't know whether the underlying search is pure vector or hybrid (that's configured in the strategy, Step 08.04).

## Requirements

### Functional
- `retrieve(query, options)` — main retrieval method
- Embed query text using the embedding service (same model as indexed chunks)
- Support multiple retrieval strategies: `vector` (pure vector search), `keyword` (fallback), `hybrid` (Step 08.04)
- Accept retrieval options:
  - `workspaceId` (mandatory) — tenant filter
  - `documentIds` (optional) — scope to specific documents
  - `metadataFilter` (optional) — additional field-level filters
  - `limit` (default 20) — max results to return
  - `minScore` (default 0.0) — minimum similarity threshold
  - `strategy` (default `vector`) — retrieval strategy selector
- Return `RetrievalResult[]` with: `chunk`, `score`, `metadata`
- Include retriever metadata: `strategyUsed`, `queryTokens`, `totalResults`, `elapsedMs`
- Support caching: if the same query (by normalized text hash) was made recently within TTL, return cached results
- Log retrieval performance: query text length, tokens, strategy, count, elapsed time
- Handle empty queries (return empty results)
- Handle queries too short for meaningful retrieval (log warning, return empty or perform keyword-only)

### Non-functional
- Retrieval latency: < 1 second p95 for vector strategy (including embedding time)
- Cache TTL: 5 minutes for identical queries
- Cached query returns in < 50 ms

### Performance
- Query embedding time: < 500 ms for reasonable query lengths (< 500 tokens)
- Vector search time: < 500 ms p95
- Total pipeline: < 1.5 seconds p95 for standard queries

### Security
- Query text is logged only at DEBUG level (not INFO)
- Retrieval results are tenant-scoped by workspaceId

### Accessibility
- N/A (internal service)

## Architecture Constraints

- The retriever always embeds the query — even for keyword-only mode, embedding is optional (controlled by strategy)
- Result scoring: for vector strategy, score = `vectorSearchScore` from MongoDB; for hybrid, score = weighted combination
- Caching uses Redis with key `retrieval:{workspaceId}:{sha256(queryText + strategy + JSON.stringify(filter))}`
- Cache invalidation: clearing the embedding cache also invalidates retrieval cache
- The retriever is stateless — all state (cached results, embeddings) lives in Redis or MongoDB

## Acceptance Criteria

- `retrieve("What is machine learning?")` returns relevant chunks from the workspace
- `retrieve` with workspaceId filter returns only that workspace's chunks
- `retrieve` with documentIds filter returns only chunks from specified documents
- `retrieve` with strategy `vector` uses pure vector search
- `retrieve` with `minScore=0.8` filters out low-similarity results
- `retrieve` returns metadata including strategy used and elapsed time
- Same query within 5 minutes returns cached results (< 50 ms)
- Empty query returns empty results immediately (no API call)
- Retrieval completes within 1.5 seconds p95

## Notes

- Query embedding uses the SAME model as the chunk embeddings (text-embedding-3-large) for dimensional compatibility
- Cache key includes strategy and filters because different configurations may return different results for the same query
- Score normalization: MongoDB returns `vectorSearchScore` which is already normalized (0 to 1 for cosine)
- For logging: use `logger.debug({ queryLength, tokens, strategy, count, elapsedMs }, "retrieval completed")`
