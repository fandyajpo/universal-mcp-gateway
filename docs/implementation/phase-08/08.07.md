## Goal

Implement the RAG Engine composition that orchestrates the complete retrieval pipeline: query embedding → retrieval (vector/hybrid) → re-ranking → context building. The engine provides a single entry point (`query(query, options)`) that the Chat service (Phase 12) and API routes call. It manages the pipeline lifecycle, error handling, tracing, and performance monitoring.

## Business Context

The RAG Engine is the central intelligence layer of the platform. Every AI interaction that requires grounded knowledge flows through this engine. It must be reliable, observable, and performant. The composition layer orchestrates the pipeline steps, handles failures gracefully (degrading retrieval quality rather than erroring), and provides consistent tracing across all steps.

## Technical Context

The RAG Engine composes the steps from 08.03-08.06 into a single pipeline. It uses a plugin-style architecture where each step is a composable function with a standard interface: `(input, context) => Promise<output>`. This allows steps to be added, removed, or reordered without changing the engine. The engine also handles cross-cutting concerns: tracing (via OpenTelemetry or structured logs), error recovery (fallback to earlier step's output on failure), and performance monitoring.

## Requirements

### Functional
- `ragQuery(query, options)` — single entry point for RAG queries
- Pipeline steps:
  1. **Query embedding**: embed the query text via embedding service
  2. **Retrieval**: retrieve top-K (default 20) chunks via selected strategy (vector/hybrid)
  3. **Re-ranking**: re-rank top-K results (optional, default enabled)
  4. **Context building**: build context window from top-N re-ranked results
- Return `RAGResult`: `{ context, query, chunks: RetrievalResult[], tokenCount, pipelineMetadata }`
- Pipeline metadata: `{ steps: [{ name, durationMs, success, error }], totalDurationMs, strategyUsed, modelUsed }`
- Support pipeline options:
  - `strategy`: `vector` | `hybrid` (default `hybrid`)
  - `rerank`: boolean (default true)
  - `topK`: number (default 20)
  - `topN`: number (default 5)
  - `filters`: metadata filters
  - `modelMaxTokens`: for context budget
  - `conversationHistory`: for context assembly
  - `systemInstructions`: for context assembly
- Handle step failures gracefully:
  - If embedding fails: return error (can't retrieve without query vector)
  - If retrieval fails: return error (no context to build)
  - If re-ranking fails: fall back to un-ranked results (log warning)
  - If context building fails: return raw retrieval results (fallback)
- Emit tracing events for each step (log duration, success/failure)
- Support pipeline middleware: pre-processing (query normalization) and post-processing (result filtering)

### Non-functional
- Pipeline overhead (orchestration): < 10 ms (negligible)
- End-to-end latency (all steps): < 3 seconds p95 for typical queries
- Graceful degradation: a failed non-critical step (re-ranking) doesn't fail the entire pipeline

### Performance
- Steps run sequentially (some could be parallelized in future if retriever and re-ranker become independent)
- Memory: pipeline holds intermediate results (chunks) in memory — max 50 chunks at any point

### Security
- Pipeline execution is tenant-scoped via workspaceId in options
- No pipeline state shared across tenants

### Accessibility
- N/A (internal service)

## Architecture Constraints

- Each pipeline step implements `PipelineStep<I, O>`: `{ name: string, execute: (input: I, context: PipelineContext) => Promise<O> }`
- PipelineContext carries `workspaceId`, `traceId`, and shared state across steps
- Steps are executed in order by the pipeline orchestrator
- If a step throws, the orchestrator checks if the step is critical (`required: true`) or non-critical
- Critical step failure stops the pipeline; non-critical step failure logs and continues with previous step's output
- The engine is stateless — instantiated per request, not shared

## Acceptance Criteria

- `ragQuery("What is revenue?")` returns context with relevant chunks and metadata
- Pipeline with `rerank: false` skips the re-ranking step
- Pipeline with `strategy: "vector"` uses vector-only retrieval
- If re-ranker API fails, pipeline continues with un-ranked results
- Pipeline tracing logs duration and success/failure for each step
- RAGResult includes correct token counts and source citations
- End-to-end query completes in < 3 seconds p95
- Invalid workspaceId returns empty result (not error)
- Middleware correctly normalizes query text before embedding

## Notes

- Pipeline step interface:
  ```typescript
  interface PipelineStep<I, O> {
    name: string;
    required: boolean;
    execute(input: I, context: PipelineContext): Promise<O>;
  }
  ```
- Pipeline orchestrator loops through steps, calling `execute` with the previous step's output
- For tracing, use `logger.info({ step: step.name, durationMs, success: true }, "pipeline step completed")`
- The `PipelineContext` is passed through all steps and can carry shared data (e.g., query embedding vector for both retrieval and re-ranking metadata)
- Middleware example: pre-processing strips HTML/URLs from query text before embedding
