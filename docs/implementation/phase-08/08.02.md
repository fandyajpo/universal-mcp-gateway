## Goal

Implement the vector store service that provides CRUD operations for embedding vectors in MongoDB Atlas. The service wraps the chunks collection with methods for inserting, upserting, deleting, and querying embedded chunks. It uses the vector search indexes (Step 08.01) for similarity search and provides a clean repository-style interface for the retriever service (Step 08.03).

## Business Context

The vector store is the permanent persistence layer for embeddings. Every embedded chunk (from Phase 07) must be stored here for retrieval. The service must handle bulk inserts (for batch processing), updates (for document re-processing), deletions (for document removal), and of course vector search queries. It enforces tenant isolation at the database query level.

## Technical Context

The vector store operates on the `chunks` MongoDB collection, where each document represents a single chunk with its embedding vector and metadata. The service uses Mongoose for standard CRUD operations and the native MongoDB aggregation pipeline (built in Step 08.01) for vector search. This keeps vector search fast (native Atlas execution) while leveraging Mongoose for data validation and type safety.

## Requirements

### Functional
- `insertChunk(embeddedChunk)` — insert a single embedded chunk document
- `insertChunks(embeddedChunks)` — bulk insert with ordered:false for performance
- `upsertChunk(chunkId, embeddedChunk)` — update existing chunk or insert if new
- `deleteChunk(chunkId, workspaceId)` — delete a single chunk with tenant check
- `deleteChunksByDocument(documentId, workspaceId)` — delete all chunks for a document
- `deleteChunksByWorkspace(workspaceId)` — delete all chunks for a workspace (workspace deletion)
- `getChunk(chunkId, workspaceId)` — get single chunk by ID with tenant check
- `getChunksByDocument(documentId, workspaceId)` — get all chunks for a document
- `vectorSearch(queryVector, options)` — perform vector similarity search with:
  - `limit`: max results (default 20)
  - `numCandidates`: candidates for vector search (default `limit * 10`)
  - `workspaceId`: mandatory tenant filter
  - `documentIds`: optional filter to specific documents
  - `metadataFilter`: optional additional metadata filters (e.g., tags, date range)
  - `minScore`: minimum similarity score threshold (default 0.0)
- `countChunks(workspaceId)` — count total chunks in a workspace
- `getDocumentChunkCount(documentId)` — count chunks for a specific document

### Non-functional
- Bulk insert throughput: > 1000 chunks/second
- Vector search latency: < 500 ms p95 for limit=20, numCandidates=200
- Single chunk operations: < 50 ms p95
- All operations tenant-scoped (workspaceId filter in every query)

### Performance
- Bulk inserts use `insertMany` with `{ ordered: false }` for speed
- Vector search uses the pre-built aggregation pipeline from Step 08.01
- `getChunksByDocument` uses a compound index on `{ documentId: 1, chunkIndex: 1 }`

### Security
- Every query includes `workspaceId` filter — cannot read other tenant's data
- Delete operations verify `workspaceId` before deletion

### Accessibility
- N/A (internal data service)

## Architecture Constraints

- The vector store uses the existing Mongoose model for chunks for CRUD operations
- Vector search uses the native MongoDB aggregation pipeline (not Mongoose) for performance
- The `workspaceId` filter is mandatory for all queries — enforced at the service level
- Chunk documents must have an index (`chunkIndex`) within the document for ordering
- The `embedding` field is stored as an array of numbers in MongoDB

## Acceptance Criteria

- `insertChunks` bulk-inserts 1000 chunks in < 1 second
- `vectorSearch` returns nearest neighbors by cosine similarity
- `vectorSearch` with `workspaceId` filter returns only that workspace's chunks
- `vectorSearch` with `documentIds` filter returns chunks only from specified documents
- `deleteChunksByDocument` removes all chunks for a document
- `getChunksByDocument` returns chunks in correct order (by chunkIndex)
- Tenant isolation: querying with wrong workspaceId returns empty results
- All operations handle errors gracefully (missing chunk, invalid query, etc.)

## Notes

- Chunk document schema: `{ _id, workspaceId, documentId, chunkIndex, content, tokenCount, embedding: number[], pageNumbers: number[], sectionPath: string, metadata: object, createdAt, updatedAt }`
- Compound index for document queries: `db.chunks.createIndex({ workspaceId: 1, documentId: 1, chunkIndex: 1 })`
- Vector search pipeline: `[{ $vectorSearch: { index, path, queryVector, numCandidates, limit, filter } }, { $addFields: { score: { $meta: "vectorSearchScore" } } }, { $match: { workspaceId, ...metadataFilters } }, { $sort: { score: -1 } }, { $limit }]`
- The `filter` in `$vectorSearch` is used for pre-filtering; `$match` after is for post-filtering
