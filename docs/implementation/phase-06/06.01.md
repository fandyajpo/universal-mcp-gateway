## Goal

Implement the PDF upload endpoint that stores the PDF in Cloudflare R2 and emits an Inngest event to trigger the async processing pipeline. The upload must validate file type (PDF only), enforce the 50 MB size limit, generate a unique document ID, store metadata in MongoDB, and return immediately with a processing status response.

## Business Context

PDF ingestion is the primary document ingestion path for enterprise users. Users upload contracts, reports, whitepapers, and documentation. The upload must feel instant — the user should not wait for processing to complete. The async pipeline approach (upload → queue → process) provides immediate feedback and scales to handle large files and high concurrency.

## Technical Context

The PDF upload builds on the Phase 05 storage infrastructure (R2 client, signed URLs, file validation). The Inngest client should already be configured (Step 00.12). This step adds the PDF-specific upload handler that creates a document record in MongoDB and fires the `pdf/uploaded` Inngest event. The upload endpoint lives in `apps/web` and imports from `@repo/database` for document persistence and `@repo/rag` for processing orchestration.

## Requirements

### Functional
- Accept multipart/form-data POST with a PDF file
- Validate file is a PDF (magic bytes check `%PDF` at offset 0)
- Enforce 50 MB maximum file size
- Generate UUID v4 document ID
- Store PDF in R2 at path `/{workspaceId}/pdfs/{documentId}.pdf`
- Create Document record in MongoDB with status `uploading`
- Emit `pdf/uploaded` Inngest event with document ID, workspace ID, user ID, R2 key
- Return HTTP 201 with `{ documentId, status: "uploading" }`
- Handle duplicate uploads (same file hash within workspace) — return existing document ID
- Support password-protected PDFs via optional `password` form field

### Non-functional
- Upload response time < 500 ms (excludes upload to R2 which may take longer)
- Concurrent upload limit: 10 per workspace
- All uploads logged with document ID, workspace ID, user ID, file size, duration

### Performance
- R2 upload timeout: 60 seconds
- Maximum concurrent uploads per pod: 20

### Security
- Authentication required (session token in Authorization header)
- Workspace membership required (validated before upload)
- File content never stored outside of R2
- Temporary files on disk deleted immediately after upload

### Accessibility
- N/A (API endpoint)

## Architecture Constraints

- The upload handler must NOT await processing — fire-and-forget via Inngest
- Document status transitions: `uploading` → (processing states) → `ready` or `failed`
- File deduplication uses SHA-256 hash of file content, scoped to workspace
- R2 key follows tenant-isolated path pattern `{workspaceId}/pdfs/{documentId}.pdf`

## Acceptance Criteria

- Uploading a valid PDF returns `{ documentId, status: "uploading" }` within 500 ms
- Uploading a non-PDF file returns HTTP 400 with descriptive error
- Uploading a file > 50 MB returns HTTP 413 with file size limit message
- Uploading the same file twice in the same workspace returns the same document ID
- An `pdf/uploaded` event appears in the Inngest event log after upload
- The Document record exists in MongoDB with status `uploading`
- Password-protected PDF upload succeeds when password is provided, fails without it
