## Goal

Implement OCR (Optical Character Recognition) integration for scanned PDFs and image-only documents. The OCR module receives PDF pages flagged as low-text-density from Step 06.02, renders each page as an image, runs Tesseract.js OCR, and produces structured text output with positional metadata compatible with the extraction output format. OCR is a fallback — it only activates when native text extraction yields insufficient content.

## Business Context

Enterprise PDFs often include scanned signatures, handwritten notes, and photographed documents. OCR enables the platform to extract meaningful text from these documents that native extraction cannot handle. Without OCR, a significant portion of enterprise documents would yield no searchable content, making the RAG pipeline blind to information in scanned documents.

## Technical Context

Tesseract.js provides JavaScript bindings for the Tesseract OCR engine, running locally in Node.js without external dependencies. The OCR pipeline: render PDF page to PNG image (using sharp or pdf.js canvas), pass image to Tesseract.js, parse the output (words with bounding boxes), and format into the same `{ pages: [{ blocks: [{ lines: [...] }] }] }` structure used by the native extractor. This ensures downstream chunking (Step 06.04) can consume both extraction paths interchangeably.

## Requirements

### Functional
- Receive PDF pages flagged for OCR (text density < 10 chars/page average)
- Render each PDF page as a PNG image at 300 DPI for optimal OCR accuracy
- Run Tesseract.js OCR on each rendered image
- Extract text with per-word bounding boxes and confidence scores
- Format OCR output identically to native extraction format
- Support English as default language; detect and use document language if available
- Handle OCR of rotated pages (auto-detect and correct orientation)
- Output UTF-8 encoded text with correct Unicode character recognition
- Merge OCR results across pages into a single structured document

### Non-functional
- OCR throughput: minimum 1 page per 5 seconds at 300 DPI
- Confidence threshold: words with < 60% confidence are flagged as low-confidence
- Memory limit: 512 MB for OCR processing (images are large at 300 DPI)
- Accuracy target: > 90% character accuracy for clean scanned documents at 300 DPI

### Performance
- Maximum concurrent OCR jobs: 2 per worker (CPU-intensive)
- Image resolution: 300 DPI default, configurable per workspace (150-600 DPI)
- Page rendering timeout: 30 seconds per page

### Security
- Image files deleted after OCR processing completes
- No image data written to logs
- OCR worker runs in isolated process to prevent memory leaks

### Accessibility
- N/A (internal processing service)

## Deliverables

- `runOCR()` function that takes PDF page images and returns structured text
- Tesseract.js integration with language configuration
- OCR confidence scoring and low-confidence flagging
- Rendering pipeline (PDF page → PNG at 300 DPI)
- Output format conversion to match native extraction structure

## Architecture Constraints

- OCR runs only when native extraction yields < 10 chars/page average (entire document or per-page fallback)
- Tesseract.js runs in a worker thread — the main thread must not block
- Language data files are bundled with the package or downloaded at first use
- 300 DPI is the minimum for acceptable accuracy on standard printed documents
- Low-confidence words (< 60%) are preserved but flagged in the output for potential review

## Acceptance Criteria

- Scanned document OCR returns text with > 90% accuracy at 300 DPI
- OCR output format matches native extraction format exactly
- Low-confidence words are flagged in the output metadata
- Rotated pages are auto-oriented before OCR
- Non-English text (e.g., accented characters) is correctly recognized
- OCR fallback activates only when native extraction is insufficient
- OCR processing times meet the 1 page per 5 seconds benchmark

## Notes

- Tesseract.js language data for English (`eng`) is ~2 MB; include it in the package
- For production, consider pre-warming the Tesseract.js worker to avoid cold-start latency
- The renderer uses `sharp` for image processing; it handles PDF rendering natively in recent versions
- If a page has both native text and scanned content (hybrid PDFs), process the native text portion and OCR only the image portion
