## Goal

Verify the complete PDF processing pipeline end-to-end. Run comprehensive tests covering all components: upload, extraction, OCR, chunking, metadata extraction, table extraction, progress tracking, and the UI. Validate that the pipeline produces correct output, handles errors gracefully, meets performance targets, and integrates correctly with downstream consumers.

## Business Context

The PDF processing pipeline is the primary data ingestion path for the knowledge base. If it fails or produces poor-quality output, the entire RAG pipeline is compromised. Verification ensures that the pipeline is production-ready before proceeding to Phase 07 and Phase 08.

## Technical Context

Verification spans unit tests, integration tests, and end-to-end tests across all PDF processing components. Test fixtures include native PDFs, scanned PDFs, mixed-content PDFs, password-protected PDFs, corrupted PDFs, and PDFs with complex tables. Performance benchmarks measure extraction throughput, chunking speed, and OCR accuracy. Error scenarios test retry logic, timeout handling, and dead-letter queue behavior.

## Requirements

### Functional
- Verify each pipeline step produces correct output given valid input
- Verify each pipeline step produces appropriate errors given invalid input
- Verify the complete pipeline end-to-end for native PDFs (upload → extract → chunk → metadata)
- Verify the complete pipeline for scanned PDFs (upload → extract → detect scanned → OCR → chunk)
- Verify error handling: corrupted PDF, oversized file, invalid file type
- Verify retry logic: temporary Inngest step failures recover correctly
- Verify progress tracking: events emitted in correct order, MongoDB updated correctly
- Verify performance: throughput meets targets defined in each step
- Verify UI components render correctly for all states
- Verify type exports are correct and consumable by downstream packages
- Verify package boundaries — no circular dependencies or incorrect imports

### Non-functional
- Test suite must complete within 5 minutes
- All integration tests use a real MongoDB and Inngest (or accurate mocking)
- Test coverage: > 85% on core pipeline logic, > 70% on UI components

### Performance
- Extraction throughput: native PDF > 5 pages/sec
- OCR throughput: > 1 page per 5 seconds
- Chunking throughput: > 1000 pages/sec
- Upload response: < 500 ms

### Security
- No credentials in test fixtures or test output
- Test documents contain no real sensitive data

### Accessibility
- N/A (verification tests)

## Architecture Constraints

- Test fixtures are small generated PDFs (< 100 KB each) created with pdf-lib
- Integration tests run against a test MongoDB instance (mongod or mongodb-memory-server)
- Inngest steps are unit-tested by calling the step function directly (not via Inngest HTTP)
- Performance benchmarks run on CI with a 2x timeout buffer
- Scanned PDF test fixtures are generated by rendering text to an image and embedding in a PDF

## Acceptance Criteria

- All unit tests pass (> 85% line coverage on pipeline logic)
- End-to-end native PDF test: upload → process → verify extracted text matches source
- End-to-end scanned PDF test: upload → detect scanned → OCR → verify extracted text matches source
- Error handling tests: corrupted PDF returns `failed` status with error details
- Retry test: step failure triggers retry and eventually succeeds
- Progress test: all events emitted in correct order with correct payloads
- Performance tests: all throughput targets met
- UI tests: all components render correctly for loading, empty, error, and data states
- TypeScript compilation passes for all updated packages
- No test leaks (test documents cleaned up after test suite)
