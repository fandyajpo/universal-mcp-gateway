## Goal

Implement a PDF-aware semantic chunking strategy that splits extracted text into coherent chunks suitable for embedding (Phase 07) and retrieval (Phase 08). The chunker must respect document structure (headings, paragraphs, page boundaries), produce chunks of consistent token size with configurable overlap, and preserve all metadata (page number, document ID, section hierarchy) required for citation and retrieval.

## Business Context

Chunking quality is the single most important factor in RAG retrieval quality. Chunks that break mid-sentence, split headings from their content, or mix unrelated topics produce garbage-in/garbage-out for the retrieval pipeline. PDF-aware chunking uses the positional metadata from extraction (font sizes, headings, page breaks) to create semantically coherent chunks that map back to the original document structure.

## Technical Context

The chunker receives structured extraction output (pages with blocks, lines, and font metadata) and produces an array of `Chunk` objects. Each chunk has: content text, token count (estimated), page numbers, section path (e.g., "2.3 > Configuration"), document ID, workspace ID, and metadata. Multiple strategies are supported for different document types, configured via a strategy selector.

## Requirements

### Functional
- Split extracted text into chunks at semantic boundaries (headings, paragraphs, page breaks)
- Detect headings by font size, weight, and style (bold, larger font) ‚Äî treat as section boundaries
- Maintain a section hierarchy path for each chunk (e.g., "Introduction > Architecture > Components")
- Support configurable chunk size: default 1024 tokens, configurable per workspace (256-4096)
- Support configurable overlap: default 128 tokens (12.5%), configurable 0-25%
- Preserve page number(s) for each chunk (single page or page range)
- Handle orphaned headings (heading at end of page without content) ‚Äî merge with following content
- Produce chunks from three strategies:
  - `recursive`: hierarchical splitting by double newline ‚Üí single newline ‚Üí sentence ‚Üí word (general text)
  - `semantic`: split at topic boundaries detected via heading analysis (long-form docs)
  - `pdf`: section-aware splitting using PDF structure, font analysis, and page layout (PDF-specific default)
- Return chunks with a unique chunk ID (UUID v4), sorted in document order
- Include chunk metadata: strategy used, token count, character count, section path, confidence

### Non-functional
- Chunking throughput: minimum 1000 pages per second (fast, CPU-light)
- Token count estimation uses tiktoken-compatible counting (approximate, not exact)
- Memory efficient: process one page at a time rather than loading entire document
- Deterministic: same input always produces same chunks (for caching)

### Performance
- Chunking is single-threaded and CPU-light (string operations only)
- Large documents: process in batch of 100 pages, merge chunk sequences

### Security
- All chunk metadata includes `workspaceId` and `documentId` for tenant isolation
- No sensitive content in chunk metadata fields

### Accessibility
- N/A (internal processing service)

## Architecture Constraints

- Chunks are produced synchronously within the Inngest step function
- Token estimation uses a fast heuristic (character_count / 4 for English, configurable ratio)
- The `pdf` strategy is the default for PDF documents; `recursive` is fallback
- Section hierarchy is built by tracking heading levels (H1 detected by font size > 150% of body)
- Overlap only applies between chunks that are semantically adjacent (not across section boundaries)

## Acceptance Criteria

- `pdf` strategy correctly splits on heading boundaries detected from font metadata
- `recursive` strategy splits on paragraph boundaries as fallback
- Chunk size stays within ¬±20% of the configured target (configurable)
- Chunks include correct page numbers and section hierarchy path
- Headings are included as the first line of the following chunk (not orphaned)
- Orphaned headings are merged with the content of the next page
- Token count estimation is within 10% of actual tiktoken count
- Same input produces identical output chunks (deterministic)
