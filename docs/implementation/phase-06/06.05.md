## Goal

Implement metadata extraction from PDF documents, capturing bibliographic information, document properties, and structural metadata. Extract title, author, subject, keywords, creation date, modification date, PDF version, page count, page dimensions, and custom metadata properties. Store extracted metadata alongside the document record for search filtering, display, and retrieval scoping.

## Business Context

Document metadata powers the search and filtering capabilities of the knowledge base. Users need to find documents by author, date range, or title. Enterprise compliance requires tracking document provenance, version, and access dates. Rich metadata also improves RAG retrieval by enabling metadata-pre-filtering (e.g., "only search documents from 2024").

## Technical Context

PDF files contain two levels of metadata: (1) the document information dictionary (title, author, subject, keywords, creator, producer, creation date, modification date) accessible via PyMuPDF, and (2) XMP metadata (extensible metadata platform) for more detailed metadata. The extraction service reads these during the extraction phase and stores them in the MongoDB Document record.

## Requirements

### Functional
- Extract standard PDF metadata: title, author, subject, keywords, creator, producer
- Extract creation and modification dates (formatted as ISO 8601)
- Extract PDF version (e.g., "1.4", "2.0")
- Count total pages and dimensions (width, height) of each page
- Extract custom metadata from the document information dictionary
- Extract XMP metadata if present (preferred over document info dictionary when both exist)
- Detect document language from metadata `Language` field or content analysis
- Extract document structure: table of contents (outline) entries if available
- Map metadata to Document model fields and store in MongoDB
- Handle missing metadata gracefully — use sensible defaults or null

### Non-functional
- Metadata extraction completes within 1 second for documents of any size (reads from PDF header, not full content)
- No false positives for language detection

### Performance
- Metadata extraction is O(1) — constant time regardless of page count
- Metadata storage overhead: < 1 KB per document

### Security
- Metadata does not contain file content — safe to log at debug level
- Custom metadata properties are sanitized for XSS before storage

### Accessibility
- N/A (internal processing service)

## Architecture Constraints

- XMP metadata (XML-based) is parsed with fast-xml-parser; if invalid XML, fall back to document info dictionary
- Date fields normalized to ISO 8601 format; PDF date format (D:YYYYMMDDHHmmSS) is converted
- Document title from metadata is used as the document name; if missing, derive from filename
- Outline (table of contents) entries are stored as nested `{ title, page, children }` structure

## Acceptance Criteria

- Title, author, subject, keywords extracted correctly from PDF metadata
- Creation/modification dates correctly parsed and stored as ISO 8601
- PDF version string extracted correctly
- XMP metadata preferred over document info dictionary when both present
- Missing metadata fields stored as null (not empty string or "Unknown")
- Table of contents entries extracted and stored as nested structure
- All metadata extracted within 1 second regardless of document size
- Metadata stored alongside document record in MongoDB

## Notes

- PyMuPDF's `document.metadata` returns a dict with keys: title, author, subject, keywords, creator, producer, creationDate, modDate
- XMP metadata is read via PyMuPDF's `document.get_xml_metadata()` which returns raw XML
- PDF date format: `D:20240610143000+0530` → convert to `2024-06-10T14:30:00+05:30`
- Some PDF producers (e.g., macOS Preview) write metadata in non-standard ways — handle these edge cases
