## Goal

Implement comprehensive progress tracking for the PDF processing pipeline using Inngest event chains. Each processing step emits lifecycle events (started, completed, failed) with timing, status, and metadata. The progress state is aggregated in MongoDB so that the UI (Step 06.08) can display real-time processing status, and downstream systems can react to processing completion.

## Business Context

Users need visibility into document processing progress, especially for large documents that take minutes to process. Progress tracking enables the UI to show which step is currently running, estimated time remaining, and detailed error messages if something fails. Operations teams can monitor pipeline health, identify bottlenecks, and debug failures without reading through logs.

## Technical Context

Inngest provides event-driven orchestration with built-in retries, failure handling, and observability. Each PDF processing step (upload, extract, OCR, chunk, embed, index) emits events at start and completion. The progress tracker listens for these events and updates the MongoDB Document record with the current step, status, and timing. A separate event listener fires a notification when processing completes.

## Requirements

### Functional
- Define typed Inngest events for each pipeline step: `pdf/{step}/started`, `pdf/{step}/completed`, `pdf/{step}/failed`
- Steps: `uploaded`, `extract`, `ocr` (if needed), `chunk`, `embed`, `index`
- Each event payload includes: `documentId`, `workspaceId`, `userId`, `step`, `status`, `timestamp`, `durationMs`, `metadata`
- Progress state aggregated in MongoDB Document record:
  - `status`: one of `uploading`, `processing`, `ready`, `failed`
  - `currentStep`: current processing step name
  - `stepsCompleted`: array of completed step names
  - `processingStartedAt`: timestamp when processing began
  - `processingCompletedAt`: timestamp when processing finished
  - `error`: error message and details if failed
  - `progress`: percentage 0-100 estimated from step count
- Listen for `pdf/*` events in a centralized progress tracker
- Update Document record on each event
- Emit `pdf/processing-completed` when all steps finish successfully
- Emit `pdf/processing-failed` when a step fails beyond retry limit
- Handle retry events — update progress to show retry count
- Store processing metrics per step (duration, retries, memory used)

### Non-functional
- Event processing latency < 100 ms from event receipt to MongoDB update
- No dropped events — all events must result in a database update
- Idempotent event handling — duplicate events do not corrupt state

### Performance
- Maximum 10 progress updates per second per document (rate-limit to avoid spamming MongoDB)

### Security
- Progress events contain document IDs and workspace IDs — must not leak to other tenants
- Event payloads scoped to the emitting workspace

### Accessibility
- N/A (internal event processing)

## Architecture Constraints

- Progress events are fire-and-forget from processing steps — they don't block the pipeline
- The progress tracker is a separate Inngest function triggered by event matching
- MongoDB updates use `$set` with upsert: false to avoid creating duplicate records
- Progress percentage is estimated: steps have weights (upload: 5%, extract: 25%, ocr: 20%, chunk: 15%, embed: 25%, index: 10%)
- Failed steps include a `retriesRemaining` count in the progress state
- If a step fails with no retries remaining, the document status becomes `failed` and `pdf/processing-failed` is emitted

## Acceptance Criteria

- Each processing step emits started/completed/failed events
- Document record in MongoDB updates with current step and progress percentage
- Progress percentage accurately reflects completed vs total steps
- Failed events update document with error details and retry information
- `pdf/processing-completed` event fires when all steps succeed
- `pdf/processing-failed` event fires when a step exhausts retries
- Duplicate events (e.g., from Inngest retries) don't corrupt progress state
- Progress updates complete within 100 ms of event receipt
