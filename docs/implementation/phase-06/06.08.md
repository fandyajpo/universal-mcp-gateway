## Goal

Build the user interface for PDF upload and processing status display. The UI includes a file upload dropzone with drag-and-drop, a document list showing processing status and progress, and individual document detail views showing extraction results, metadata, and error details. The UI polls the document status or receives updates via the progress tracking backend.

## Business Context

Users need a visual interface to upload documents and monitor processing progress. The upload experience must feel modern and responsive (drag-and-drop, progress bar, instant validation feedback). The document list provides an at-a-glance view of all uploaded documents, their processing status, and quick access to viewing extracted content or retrying failed documents.

## Technical Context

The UI is built using the existing shadcn/ui component library (button, card, dialog, toast, progress bar) and TailwindCSS. The upload API from 06.01, document list API from the database repository, and progress tracking from 06.07 power the UI. The document list uses TanStack Table for sorting and filtering. File upload uses the `react-dropzone` library for drag-and-drop support.

## Requirements

### Functional
- Drag-and-drop file upload zone accepting only PDF files
- Upload progress bar showing R2 upload percentage (not Inngest progress)
- File type validation on drop (reject non-PDF with visual error)
- File size validation (reject > 50 MB with visual error)
- Document list table with columns: name, status, upload date, page count, actions
- Status badges: `uploading` (yellow), `processing` (blue with spinner), `ready` (green), `failed` (red)
- Processing progress indicator showing current step name and percentage
- Document detail view showing: metadata (title, author, date, pages), extraction status per step, error details if failed
- Retry button for failed documents
- Delete document button with confirmation dialog
- Document search/filter by name and status
- Toast notifications on upload success and failure
- Empty state when no documents exist
- Loading skeleton for document list during fetch

### Non-functional
- Page load time < 1 second for document list (cached, paginated)
- Upload progress updates at least every 200 ms
- Re-fetch document status every 2 seconds for documents in `processing` state (polling)
- Responsive design: works on desktop (1024px+) and tablet (768px+)

### Performance
- Document lists paginated (20 items per page)
- Virtual scrolling for documents > 100 items (TanStack Virtual)

### Security
- Upload button disabled when not authenticated
- Document list filtered by workspace membership
- Delete confirmation requires re-authentication for documents (modal with session check)

### Accessibility
- All form inputs have labels
- Status badges have aria-labels for screen readers
- Error messages are announced to screen readers via aria-live
- Keyboard navigation for document list (arrow keys, enter to open)

## Architecture Constraints

- Document list uses React Query for data fetching with automatic refetching for processing documents
- Upload is handled via direct fetch to `/api/documents/pdf/upload` (not through React Query mutations)
- The upload zone component is reusable — it can be placed on both the documents page and within the chat UI (Phase 12)
- Document detail page uses dynamic route parameter `[id]` with server-side data loading
- Status polling stops when document reaches `ready` or `failed` state

## Notes

- Use `react-dropzone` for drag-and-drop (or shadcn's upload component pattern)
- For the progress polling, use React Query's `refetchInterval` option set to 2000 ms, but only when status is `processing`
- The upload progress can use `XMLHttpRequest` with `upload.onprogress` event (fetch doesn't support upload progress natively)
- Status badge colors: uploading: `bg-yellow-100 text-yellow-800`, processing: `bg-blue-100 text-blue-800`, ready: `bg-green-100 text-green-800`, failed: `bg-red-100 text-red-800`
