## Goal

Implement the sync engine that periodically synchronizes data from third-party services into the platform's knowledge base. The sync engine manages sync schedules, incremental sync with cursor-based pagination, full re-syncs, conflict resolution, and sync status tracking. It runs as background jobs via Inngest.

## Business Context

Connectors need to import data from third-party services into the knowledge base for RAG retrieval. For example, syncing Notion pages, Confluence documents, Google Drive files, or GitHub issues. The sync engine provides a standardized approach to data ingestion with support for full and incremental syncs.

## Technical Context

The sync engine uses a cursor-based approach (store the last synced cursor/timestamp, only fetch changes since then). It runs as Inngest scheduled functions with configurable intervals. Each connector defines its sync behavior: what entities to sync, the sync strategy (full vs incremental), and how to transform external data into platform documents.

## Requirements

### Functional
- `SyncConfig: { connectorId, workspaceId, enabled, interval: "5min" | "15min" | "30min" | "1hour" | "6hours" | "1day", entities: string[], lastSyncCursor?, lastSyncAt?, status: "active" | "paused" | "error" }`
- `startSync(syncConfig)` — start a sync job:
  - Create/update sync configuration in MongoDB
  - Schedule recurring sync via Inngest
- `stopSync(connectorId, workspaceId)` — stop sync
  - Remove Inngest schedule
  - Set sync status to paused
- `executeSync(connectorId, workspaceId)` — perform a sync (called by Inngest):
  - Load connector instance
  - Ensure connector is authenticated (refresh if needed)
  - Determine sync strategy: incremental (cursor exists) or full (no cursor)
  - Call connector's `sync(cursor)` method (connector-specific)
  - Process returned entities: transform to platform documents, store in knowledge base
  - Update cursor and lastSyncAt
  - Handle errors: retry 3 times, then mark sync as error
- Support incremental sync: connector returns only entities changed since cursor
- Support full sync: connector returns all entities (for initial sync or re-sync)
- Sync result: `{ entitiesSynced: number, entitiesFailed: number, cursor, durationMs, errors: string[] }`
- Track per-entity sync status: `{ externalId, status: "synced" | "failed", lastSyncedAt, error? }`
- Rate limiting: respect provider API rate limits during sync (use sliding window per provider)
- Concurrency: max 1 sync per connector instance at a time (prevent overlapping syncs)
- Locking: use Redis lock to prevent multiple workers from syncing the same connector simultaneously

### Non-functional
- Sync interval accuracy: ±30 seconds of scheduled time (Inngest handles scheduling)
- Entity processing: > 100 entities per minute per connector
- Sync state storage: MongoDB for persistence, Redis for locks

### Performance
- Batch processing: process entities in batches of 50
- Rate limiting: token bucket per provider (from sync config)

### Security
- Sync config stored encrypted (contains no secrets)
- Synced data stored tenant-scoped (workspaceId on every entity)
- Lock timeout: 10 minutes (released after sync completes or crashes)

### Accessibility
- N/A (internal engine)

## Architecture Constraints

- Sync runs as Inngest scheduled functions — one schedule per connector instance
- The sync engine calls the connector's `sync(cursor)` method which returns `{ entities, newCursor }`
- Entities are connector-specific types; the processor transforms them to platform documents
- The cursor is an opaque string managed by the connector (could be a timestamp, page token, or offset)
- Lock key in Redis: `sync:lock:{connectorId}:{workspaceId}` with 10-minute TTL
- If sync takes longer than the schedule interval, the next scheduled run is skipped (no overlap)

## Acceptance Criteria

- Sync schedule created and executed by Inngest
- Incremental sync calls connector with cursor, processes only new entities
- Full sync calls connector without cursor, processes all entities
- Sync result records entities synced, failed, and duration
- Rate limiting respects provider limits during sync
- Redis lock prevents concurrent syncs for same connector
- Sync errors trigger retry then mark as error
- Sync status tracked per connector in MongoDB
- Sync can be paused and resumed

## Notes

- Connector sync method contract: `sync(cursor?: string): Promise<{ entities: ExternalEntity[], newCursor?: string, hasMore: boolean }>`
- The processor transforms ExternalEntity to Document: `{ title, content, metadata: { source, externalId, url, author, createdAt, updatedAt } }`
- For incremental sync, the cursor is typically a timestamp or event ID from the provider
- Rate limit per provider: store in Redis sorted set `sync:ratelimit:{providerId}` — sliding window of last 60 seconds
- Lock: use `redis.set(key, workerId, { nx: true, ex: 600 })` to acquire; `redis.del(key)` to release
