## Goal

Implement the Notion connector that integrates Notion workspaces with the platform. The connector provides tools for reading pages, databases, and comments, searching Notion content, and receiving updates via Notion's webhook API. It exposes Notion content as MCP tools and syncs pages into the knowledge base for RAG retrieval.

## Business Context

Notion is widely used for documentation, project management, and knowledge management. Integrating Notion allows AI models to search across company wikis, project docs, and databases. The connector transforms Notion's block-based content into searchable text for the RAG pipeline.

## Technical Context

The Notion connector uses the Notion SDK (`@notionhq/client`) for API access. Authentication is via OAuth2 (Internal Integration) with Notion's bot token. Notion's API is block-based — pages are composed of blocks (paragraphs, headings, lists, tables). The connector converts blocks to plain text and markdown for knowledge base storage.

## Requirements

### Functional
- **Authentication**: OAuth2 Internal Integration
  - Required capabilities: `Read content`, `Read comments`, `Search`
  - Bot token expires never (Notion internal integrations)
- **Tools**:
  - `notion_list_pages` — search pages with filtering (supports `query`, `filter: { property, value }`, `sort`, `page_size`)
  - `notion_get_page` — get page content as markdown (retrieves all blocks, converts to markdown)
  - `notion_list_databases` — list accessible databases
  - `notion_query_database` — query a database with filtering (supports Notion filter and sort objects)
  - `notion_get_database` — get database schema (properties, types)
  - `notion_search` — search all Notion content (pages, databases) with query
  - `notion_get_comments` — get comments for a page or discussion
  - `notion_get_page_property` — get a specific property of a page
- **Block conversion**: Convert Notion blocks to markdown:
  - Heading 1/2/3 → `#/##/### text`
  - Paragraph → `text\n\n`
  - Bulleted list → `- text`
  - Numbered list → `1. text`
  - To-do → `- [x] text` or `- [ ] text`
  - Code block → ```code``` with language
  - Quote → `> text`
  - Callout → `**Callout:** text`
  - Image → `![caption](url)`
  - Table → markdown table format
  - Divider → `---`
- **Webhook events**: Polling-based (Notion doesn't have real-time webhooks for content changes)
- **Sync**: Incremental sync using `last_edited_time` cursor
  - Sync pages modified since last sync
  - Convert blocks to markdown
  - Store as documents in knowledge base
- **Rate limiting**: 3 requests per second (Notion's limit for internal integrations)
- **Error handling**: Map Notion API errors; handle rate limits with 1-second backoff

### Non-functional
- Block-to-markdown conversion: < 10 ms for a typical page (< 100 blocks)
- Page retrieval: < 1 second for pages with up to 100 blocks

### Performance
- Batch block retrieval: use `blocks.children.list` with `page_size=100`
- Rate limit: queue requests with 350ms minimum interval between calls

### Security
- Bot token stored encrypted
- Only content the bot has access to is retrievable
- No write operations performed (read-only integration)

### Accessibility
- N/A (connector service)

## Architecture Constraints

- Notion API has a 3 req/sec rate limit — the connector must enforce this strictly
- Block retrieval requires recursive fetching for nested blocks (blocks within blocks)
- Notion page content is stored as markdown in the platform's document format
- The connector is read-only — no write tools are implemented initially
- Sync uses `last_edited_time` filter on the search endpoint for incremental updates

## Notes

- Notion SDK: `@notionhq/client` provides `Client` with methods for all API endpoints
- Client initialization: `new Client({ auth: accessToken })`
- Retrieve page blocks: iterate with `client.blocks.children.list({ block_id: pageId })` — recurse for nested blocks
- Rate limiting: use a simple queue — push requests to an array, process one every 350ms
- Notion's search endpoint: `client.search({ query, filter: { property: "object", value: "page" } })`
- Database query: `client.databases.query({ database_id, filter, sorts, page_size })`
