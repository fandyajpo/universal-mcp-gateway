## Goal

Implement the webhook handler that receives incoming webhooks from third-party services and dispatches them to the appropriate connector instance. The handler validates webhook signatures, parses payloads, routes to the correct connector, and manages webhook lifecycle (registration, renewal, revocation).

## Business Context

Webhooks enable real-time event-driven integration with third-party services. For example, when a new message arrives in Slack or a new issue is created in GitHub, the webhook delivers that event to the platform. This enables the chat UI to show real-time updates and triggers sync operations.

## Technical Context

Each connector defines the webhooks it supports (events it wants to receive). The webhook handler manages the webhook registration with the third-party service (via the connector's API), receives incoming HTTP requests, validates signatures, deserializes events, and dispatches typed event objects to the connector's event handler.

## Requirements

### Functional
- `registerWebhook(connector, events, targetUrl)` — register webhook with provider
  - Use connector's API to register webhook with specific events
  - Store webhook ID and secret returned by provider
  - Return WebhookRegistration: `{ id, secret, targetUrl, events }`
- `handleWebhook(request, connectorId)` — handle incoming webhook HTTP request
  - Parse incoming HTTP request body (JSON or URL-encoded)
  - Validate webhook signature using provider-specific algorithm
  - Deserialize event into typed ConnectorEvent
  - Dispatch to connector's `handleEvent(event)` method
  - Return appropriate HTTP response (200 OK for verified, 401 for invalid)
- `renewWebhook(webhookId, connector)` — refresh expiring webhook
- `revokeWebhook(webhookId, connector)` — remove webhook registration
- Support common webhook signature methods:
  - HMAC-SHA256 (Slack, GitHub): verify `X-Slack-Signature`, `X-Hub-Signature-256`
  - HMAC-SHA1 (GitHub legacy): verify `X-Hub-Signature`
  - Token-based (Notion): compare token in request
  - No signature (public webhooks, with caution)
- Handle webhook retries: providers may send the same event multiple times; deduplicate by event ID
- Support webhook secret rotation (periodic renewal for security)
- Store webhook state: `{ connectorId, webhookId, secret (encrypted), events, createdAt, expiresAt, lastEventAt }`

### Non-functional
- Webhook signature verification: < 5 ms
- Event dispatching: < 2 ms
- Webhook registration: depends on provider API (100-2000 ms)

### Performance
- Webhook endpoint handles up to 100 requests/second
- Deduplication: store last 1000 event IDs in Redis with 5-minute TTL

### Security
- Webhook secret stored encrypted (same as OAuth tokens)
- Signature verification is mandatory for all providers that support it
- Request body is read once and buffered for signature verification (don't consume stream twice)
- Rate limit webhook requests per provider IP: 100/min per connector

### Accessibility
- N/A (internal handler)

## Architecture Constraints

- The incoming webhook endpoint is a single HTTP route: `POST /api/webhooks/:connectorId`
- Signature verification is provider-specific — the verifier factory selects the correct algorithm based on connector type
- Webhook secret is decrypted in memory for verification, then zeroed (overwritten with zeros)
- Event deduplication uses event ID + provider as key in Redis with 5-minute TTL
- If signature verification fails, return 401 and log the event details (without exposing the secret)

## Acceptance Criteria

- Webhook registration creates webhook on provider and stores registration
- Incoming webhook with valid signature dispatches event to connector handler
- Incoming webhook with invalid signature returns 401
- Slack-style HMAC-SHA256 signature verification works
- GitHub-style HMAC-SHA1 and SHA256 signature verification works
- Duplicate events (same event ID within 5 minutes) are silently acknowledged but not dispatched
- Webhook renewal refreshes expiring webhooks
- Webhook revocation removes registration from provider
- Stored secret is encrypted

## Notes

- Slack signature verification: `v0={HMAC-SHA256(version + ":" + timestamp + ":" + body)}` compared to `X-Slack-Signature`
- GitHub signature verification: `X-Hub-Signature-256={HMAC-SHA256(body, secret)}` or `X-Hub-Signature={HMAC-SHA1(body, secret)}`
- Raw body must be available for signature verification — use `req.text()` before any JSON parsing
- For deduplication: Redis key = `webhook:dedup:{provider}:{eventId}`, TTL = 300 seconds
- Rate limit tracking: Redis key = `webhook:ratelimit:{connectorId}:{ip}`, sliding window
