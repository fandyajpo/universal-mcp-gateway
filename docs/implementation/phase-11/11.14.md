## Goal

Implement REST API routes for connector installation and management. The API provides endpoints for listing available connectors, installing/uninstalling connectors, managing configuration and authentication, checking health, and triggering syncs. All endpoints require authentication and workspace membership.

## Business Context

Connector management needs to be accessible from the UI (future), CLI, and automated workflows. The REST API provides a standardized interface for all connector lifecycle operations. The OAuth2 callback endpoint is particularly important — it handles the redirect from third-party OAuth providers after authorization.

## Technical Context

The API routes live in `apps/web` as Next.js App Router route handlers. They delegate to the connector registry (Step 11.13) for business logic, apply auth middleware from `@repo/auth`, and use Zod validation for request/response schemas. The OAuth2 callback route handles provider redirects, exchanges authorization codes for tokens, and redirects back to the UI.

## Requirements

### Functional
- `GET /api/connectors` — list available connector types
  - Returns: `[{ id, name, version, description, author, capabilities, authMethods, configSchema }]`
- `POST /api/connectors/install` — install a connector
  - Request: `{ workspaceId, connectorId, config }`
  - Returns: `{ installationId, status, authUrl? }` — includes `authUrl` if OAuth2 is needed
- `POST /api/connectors/:installationId/configure` — update configuration
  - Request: `{ config }`
  - Returns: `{ installationId, status }`
- `POST /api/connectors/:installationId/authenticate` — initiate authentication
  - For OAuth2: returns `{ authUrl }` for the user to visit
  - For API key: accepts `{ apiKey }` in request body
- `GET /api/connectors/oauth/callback` — OAuth2 callback endpoint
  - Receives `?code=&state=` from provider
  - Exchanges code for token
  - Stores encrypted auth data
  - Redirects to success URL or returns JSON
- `POST /api/connectors/:installationId/sync` — trigger sync
  - Returns: `{ syncId, status: "started" }`
- `GET /api/connectors/:installationId` — get installation details
  - Returns: `{ installationId, connectorId, status, health, lastSyncAt, toolsRegistered }`
- `DELETE /api/connectors/:installationId` — uninstall connector
  - Returns: `{ success: true }`
- `GET /api/connectors/:installationId/health` — get health status
  - Returns: `{ status: "healthy" | "unhealthy", lastCheckAt, lastError }`
- All routes require authentication and workspace membership

### Non-functional
- Installation listing: < 100 ms
- Installation/uninstallation: < 500 ms (includes tool registration/deregistration)
- OAuth2 callback: < 2 seconds (includes code exchange + token storage)
- OAuth2 callback URL is configurable per environment

### Performance
- Sync trigger is async — returns immediately, sync runs via Inngest

### Security

- Workspace membership validated
- OAuth2 state parameter validated for CSRF protection
- API keys validated for format before storage

## Architecture Constraints

- The OAuth2 callback route must handle dynamic state parameter that encodes the installation ID
- OAuth2 state: JSON encoded as `{ installationId, csrfToken, redirectUrl }`, encrypted or HMAC-signed
- After OAuth2 callback, redirect user to `redirectUrl` or to the workspace's connectors page
- Sync trigger is fire-and-forget — the response confirms the request, not the sync completion
- Error responses: standard format `{ error: { code, message } }`

## Notes

- OAuth2 state structure: `{ installationId: string, nonce: string, redirectUrl: string }` — sign with HMAC to prevent tampering
- State validation: verify HMAC signature, extract installationId, verify it matches the connector installation
- OAuth2 callback flow:
  1. User visits provider's auth URL
  2. Provider redirects to `/api/connectors/oauth/callback?code=xxx&state=yyy`
  3. Server validates state, exchanges code, stores token
  4. Server redirects to workspace connectors page with success message
- For API key auth: the `authenticate` endpoint accepts the key directly and stores it encrypted
