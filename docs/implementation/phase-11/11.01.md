## Goal

Define the `Connector` interface and supporting types that all connector implementations must adhere to. The interface specifies how connectors declare metadata, authentication requirements, tools, resources, lifecycle hooks, and event handlers. This interface is the contract between the connector SDK and all connector implementations.

## Business Context

The connector ecosystem enables third-party service integration. The Connector interface is the contract that every connector (Slack, Notion, GitHub, etc.) must implement. A well-designed interface ensures consistency across connectors, simplifies the SDK, and enables third-party developers to build custom connectors.

## Technical Context

The interface is defined in TypeScript with strict typing. Each connector is a class that implements `Connector` and provides tools that are registered with the MCP Gateway's Tool Registry (Step 10.02). The interface covers configuration, authentication, tool definitions, lifecycle management, and webhook handling.

## Requirements

### Functional
- Define `Connector` interface:
  ```typescript
  interface Connector<C = Record<string, unknown>, A = Record<string, unknown>> {
    readonly id: string;
    readonly name: string;
    readonly version: string;
    readonly description: string;
    readonly author: string;
    readonly configSchema: JSONSchema;  // Configuration schema
    readonly authMethods: AuthMethod[];  // Supported auth methods
    readonly tools: ToolDefinition[];    // Tools this connector provides
    readonly events: ConnectorEvent[];   // Events this connector emits
    readonly capabilities: ConnectorCapability[];

    initialize(config: C): Promise<void>;
    authenticate(auth: A): Promise<AuthResult>;
    getTools(): Promise<ToolDefinition[]>;
    executeTool(name: string, params: unknown, context: ToolContext): Promise<ToolResult>;
    shutdown(): Promise<void>;
  }
  ```
- Define `AuthMethod` types:
  - `oauth2` ‚Äî OAuth2 authorization code flow
  - `api_key` ‚Äî static API key (header-based)
  - `basic` ‚Äî HTTP Basic Auth
  - `none` ‚Äî no authentication (public data)
- Define `OAuth2Config`: `{ authorizationUrl, tokenUrl, scopes, clientId, clientSecret? }`
- Define `ConnectorCapability`: `tools`, `resources`, `webhooks`, `sync`, `search`
- Define `ConnectorEvent`: `{ name: string, description: string, schema: JSONSchema }`
- Define `ToolContext`: `{ workspaceId, userId, traceId, logger, reportProgress }`
- Define `ToolResult`: `{ content: ContentPart[], metadata?: Record<string, unknown>, isError?: boolean }`
- Define `AuthResult`: `{ success: boolean, credentials?: Record<string, unknown>, expiresAt?: Date, error?: string }`
- All types should be generic where appropriate to support connector-specific config/auth types
- Provide JSDoc documentation for all interface methods and properties

## Architecture Constraints

- The interface MUST NOT depend on any connector-specific packages (no Slack SDK, no Notion SDK, etc.)
- Tools returned by `getTools()` must be compatible with the MCP Gateway's ToolDefinition type
- The `executeTool` method receives the same context shape as the MCP sandbox execution context
- Configuration and auth data are stored encrypted (handled by the connector registry, not the interface)
- The interface is designed for async operations (all lifecycle methods return Promise)

## Acceptance Criteria

- Connector interface compiles with strict TypeScript
- AuthMethod types cover OAuth2, API key, Basic, and none
- ToolContext includes workspaceId, userId, traceId, logger, progress reporter
- ToolResult supports text, JSON, and image content types
- JSDoc documents all public API surface
- Interface is generic enough to support all planned connectors (Slack, Notion, GitHub, etc.)
