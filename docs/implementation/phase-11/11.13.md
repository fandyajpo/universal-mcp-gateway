## Goal

Implement the connector registry that manages connector lifecycle: installation, configuration, authentication, tool registration with MCP Gateway, health monitoring, and removal. The registry is the central management point for all connector instances across workspaces.

## Business Context

The connector registry manages which connectors are installed in which workspaces, their configuration, authentication state, and health. Without the registry, each connector would need to be manually configured and tracked. The registry provides a consistent management layer that the UI (future) and API can interact with.

## Technical Context

The registry stores connector instances in MongoDB with their configuration (encrypted), auth state (encrypted), and status. It dynamically loads connector implementations from `@repo/connectors`, instantiates them with workspace-specific configuration, and registers their tools with the MCP Gateway's Tool Registry (Step 10.02). The registry also manages the connector lifecycle: initialize → authenticate → register tools → health monitoring → shutdown.

## Requirements

### Functional
- `installConnector(workspaceId, connectorId, config)` — install a connector in a workspace
  - Validate config against connector's `configSchema`
  - Create connector instance record in MongoDB
  - Return installation result with ID
- `uninstallConnector(installationId)` — remove a connector installation
  - Call `connector.shutdown()`
  - Remove tools from MCP registry
  - Delete installation record
- `configureConnector(installationId, config)` — update connector configuration
  - Validate new config
  - Re-initialize connector with new config
  - Re-authenticate if needed
- `authenticateConnector(installationId, authData)` — authenticate connector
  - Call `connector.authenticate(authData)`
  - Store encrypted auth data
  - Register connector's tools with MCP Tool Registry
  - Return auth result
- `getConnector(installationId)` — get connector instance (for tool execution)
  - Load cached connector or instantiate from stored config/auth
  - If credentials expired, attempt refresh
  - Return initialized connector ready for `executeTool()`
- `listInstalledConnectors(workspaceId)` — list all installed connectors in workspace
- `getConnectorHealth(installationId)` — get connector health status
- `getAvailableConnectors()` — list all available connector types
- Store connector state in MongoDB:
  - `Installation: { _id, workspaceId, connectorId, config (encrypted), auth (encrypted), status: "uninitialized" | "active" | "error" | "removed", lastHealthCheckAt, lastError, toolsRegistered: string[], createdAt, updatedAt }`
- In-memory cache: cache active connector instances (Map keyed by installationId)
- Cache invalidation: on config/auth update or health change, invalidate cache

### Non-functional
- Connector instantiation: < 100 ms (includes config decryption + auth decryption + tool registration)
- Connector lookup from cache: < 5 ms
- Health check: periodic (every 30 seconds), async

### Performance
- Instance cache: LRU with max 100 entries, TTL 5 minutes
- Lazy loading: connectors are loaded on first use, not at workspace initialization

### Security
- Config and auth data encrypted at rest using `@repo/crypto`
- Decrypted config/auth only exists in memory for the active connector instance
- On cache eviction, zero out decrypted data

### Accessibility
- N/A (internal registry)

## Architecture Constraints

- Connector implementations are loaded dynamically from `@repo/connectors/src/{connectorId}/`
- The registry uses a static map of available connectors (not dynamic discovery)
- Tool registration: when a connector is authenticated, its tools are registered with the MCP Gateway Tool Registry
- Tool unregistration: when a connector is uninstalled, its tools are removed from the Tool Registry
- Configuration changes require re-authentication if the auth method changed
- Health check: the registry periodically checks connector health and updates status
