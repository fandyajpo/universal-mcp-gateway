## Goal

Implement the GitHub connector that integrates GitHub repositories with the platform. The connector provides tools for reading repositories, issues, pull requests, file content, and searching code. It exposes GitHub data as MCP tools and syncs issues and PRs into the knowledge base.

## Business Context

GitHub stores a wealth of technical knowledge: code, documentation, issues, pull requests, and discussions. Integrating GitHub allows AI models to reference code, search technical discussions, and access repository documentation. This is critical for developer-focused use cases.

## Technical Context

The GitHub connector uses the Octokit REST SDK (`@octokit/rest`) for GitHub API access. Authentication is via OAuth2 or GitHub App installation token. The connector provides read-only access to repositories, issues, pull requests, file contents, and search functionality.

## Requirements

### Functional
- **Authentication**: OAuth2 (user-to-server) or GitHub App installation
  - Required scopes: `repo` (private repos), `public_repo` (public repos only)
- **Tools**:
  - `github_list_repos` — list repositories (supports `type: all|owner|member`, `sort`, `per_page`)
  - `github_get_repo` — get repository details (description, language, topics, license)
  - `github_list_issues` — list issues with filters (`state`, `labels`, `assignee`, `milestone`, `since`)
  - `github_get_issue` — get issue with comments
  - `github_search_issues` — search issues and PRs across repos (`q` query, `sort`, `order`)
  - `github_list_pull_requests` — list PRs (`state`, `head`, `base`, `sort`, `direction`)
  - `github_get_pull_request` — get PR details with diff summary
  - `github_get_file_content` — get file content from a repository (`path`, `ref`)
  - `github_search_code` — search code across repositories (`q` query)
  - `github_list_commits` — list commits (`sha`, `path`, `since`, `until`)
- **Webhook events**: GitHub Webhooks for real-time updates
  - `issues` — opened, edited, closed, reopened
  - `pull_request` — opened, edited, closed, merged
  - `push` — code pushed to repository
- **Sync**: Incremental sync of issues and PRs using `updated` since cursor
- **Rate limiting**: Respect GitHub's rate limits (5000 req/hour for authenticated requests)
- **Error handling**: Map GitHub API errors; handle rate limits with Retry-After

### Non-functional
- Search results: GitHub search has a 1000-result limit per query; handle with pagination
- File content: files > 1 MB return `too_large` error from GitHub; handle gracefully

### Performance
- Pagination: use Octokit's built-in pagination for all list operations
- Caching: repository list cached for 5 minutes (repos don't change frequently)

### Security
- Token scoped to minimal required permissions
- Private repository content only accessible with appropriate token scope
- No write operations (read-only integration)

### Accessibility
- N/A (connector service)

## Architecture Constraints

- GitHub API authentication: OAuth2 token passed as `Authorization: Bearer {token}` header
- Octokit pagination: use `octokit.paginate.iterator()` for efficient pagination of large result sets
- File content fetching: use `octokit.repos.getContent()` which returns base64-encoded content — decode to string
- Search API: GitHub's search has different rate limits than REST API (30 req/min for code search unauthenticated, 3000/min authenticated)
- The connector is read-only — no create/update/delete operations

## Notes

- Octokit initialization: `new Octokit({ auth: token, userAgent: "universal-mcp-gateway" })`
- For `github_get_file_content`, use `octokit.repos.getContent({ owner, repo, path, ref })` — response is base64-encoded
- For `github_search_code`, use `octokit.search.code({ q, sort, order, per_page })`
- GitHub webhook signature: `X-Hub-Signature-256={HMAC-SHA256(body, secret)}` — use webhook verifier from Step 11.04
- For sync, store `since = lastSyncAt.toISOString()` for `updated` filter on issues/PRs
