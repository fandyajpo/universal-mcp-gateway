## Goal

Implement the OAuth2 helper utility that connectors use to manage the OAuth2 authorization code flow. The helper handles authorization URL generation, token exchange, token refresh, and token revocation. It ensures secure credential storage and automatic token refresh for long-running connector instances.

## Business Context

Most third-party integrations use OAuth2 for authentication. Slack, Notion, GitHub, Google Drive, Linear, Jira, and Confluence all use OAuth2. The OAuth helper is the most reused component across all connectors, making its correctness and security critical.

## Technical Context

The OAuth helper implements the OAuth2 Authorization Code flow (RFC 6749). It generates authorization URLs with state parameters for CSRF protection, exchanges authorization codes for tokens, and handles token refresh using refresh tokens. Tokens are encrypted before storage and decrypted in memory only during use.

## Requirements

### Functional
- `generateAuthUrl(config, state, redirectUri): string` ŌĆö generate OAuth2 authorization URL
  - Required params: `authorizationUrl`, `clientId`, `redirectUri`, `scope`, `state`
  - Optional: `responseType` (default "code"), `accessType` (default "offline"), `prompt`
  - State parameter must be a cryptographically random string for CSRF protection
  - Return fully constructed URL with all parameters
- `exchangeCode(config, code, redirectUri): Promise<TokenResponse>` ŌĆö exchange auth code for tokens
  - POST to `tokenUrl` with authorization code
  - Parse response: `access_token`, `refresh_token`, `expires_in`, `token_type`, `scope`
  - Validate response contains access_token
  - Return TokenResponse
- `refreshToken(config, refreshToken): Promise<TokenResponse>` ŌĆö refresh expired token
  - POST to `tokenUrl` with refresh token
  - Handle provider-specific refresh token behavior
  - Update stored expiration
  - Return new TokenResponse
- `revokeToken(config, token): Promise<void>` ŌĆö revoke token (if provider supports)
  - POST to provider's revocation endpoint (if available)
  - Gracefully handle if provider doesn't support revocation
- `isTokenExpired(tokenResponse): boolean` ŌĆö check if token is expired (with 5-minute buffer)
- Store encrypted tokens via encryption utility
- Support PKCE (Proof Key for Code Exchange) for providers that require it (RFC 7636)
- Handle common OAuth2 error responses: `invalid_grant`, `invalid_client`, `unauthorized_client`
- Log OAuth2 flow steps at DEBUG level (never log tokens)
- Rate limit token refresh: maximum 1 refresh per 30 seconds per connector instance

### Non-functional
- Auth URL generation: < 1 ms
- Token exchange: depends on provider API (typically 200-1000 ms)
- Token refresh: depends on provider API (typically 200-1000 ms)
- Token expiration check: < 1 ms

### Performance
- Token refresh is synchronous ŌĆö the requesting execution waits for refresh to complete
- Token refresh results are cached for 30 seconds (prevent multiple concurrent refreshes)

### Security
- State parameter: 32-byte random hex string via `crypto.randomBytes(32).toString('hex')`
- PKCE code verifier: 43-byte random string; code challenge: base64url(sha256(verifier))
- Tokens are encrypted before storage; only decrypted in memory during use
- Token never appears in logs (redacted as `token=[redacted]`)
- Access token is never sent to any URL except the provider's authorized endpoints

### Accessibility
- N/A (internal helper)

## Architecture Constraints

- The OAuth helper is stateless ŌĆö state must be managed by the caller (connector registry handles persistence)
- Token storage encryption uses `@repo/crypto` with a connector-specific key derivation
- PKCE is mandatory for connectors that support it (per provider documentation)
- Token refresh is attempted automatically when `executeTool` receives a 401 response from the provider API
- If token refresh fails (invalid_grant), the connector transitions to `unauthorized` state

## Notes

- Standard OAuth2 token endpoint request format:
  ```
  POST /oauth/token
  Content-Type: application/x-www-form-urlencoded
  grant_type=authorization_code&code={code}&redirect_uri={uri}&client_id={id}&client_secret={secret}
  ```
- Token refresh request:
  ```
  POST /oauth/token
  Content-Type: application/x-www-form-urlencoded
  grant_type=refresh_token&refresh_token={token}&client_id={id}&client_secret={secret}
  ```
- PKCE S256 code challenge: `base64url(sha256(codeVerifier))` where `codeVerifier` is a 43-128 char random string
- Store encrypted tokens in MongoDB via the connector registry; key = connector instance ID
- Token refresh concurrency: use a simple lock (`let refreshing = false`) with queue ŌĆö if refresh is already in progress, wait for it
