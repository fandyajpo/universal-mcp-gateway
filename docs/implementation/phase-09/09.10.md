## Goal

Verify the complete AI Gateway end-to-end. Run comprehensive tests covering all components: OpenRouter client, provider abstraction, model router, streaming, fallback logic, cost tracker, rate limiter, prompt templates, and API routes. Validate that the gateway correctly routes, streams, tracks costs, rate-limits, and recovers from failures.

## Business Context

The AI Gateway is the central nervous system of the platform — every AI interaction passes through it. If it fails, the entire platform is non-functional. Verification ensures production readiness, cost accuracy, rate limit correctness, and graceful failure handling.

## Technical Context

Verification spans unit tests, integration tests (against real OpenRouter API), and end-to-end tests. Performance benchmarks measure latency at each step. Error scenarios test every failure mode: provider timeout, rate limit, auth failure, streaming interruption, and fallback activation.

## Requirements

### Functional
- Verify OpenRouter client handles all API methods correctly
- Verify provider abstraction returns consistent types across providers
- Verify model router selects optimal model based on all criteria
- Verify streaming delivers tokens correctly and handles cancellation
- Verify fallback chain activates and succeeds on provider failure
- Verify cost tracker records correct token usage and pricing
- Verify rate limiter enforces all three tiers correctly
- Verify prompt templates render with correct variable substitution
- Verify API routes respond correctly for all endpoints
- Verify end-to-end: API route → auth → rate limit → router → provider → stream → cost tracking

### Non-functional
- Test suite completes within 10 minutes
- Integration tests with real OpenRouter API (test key, minimal token usage)
- Test coverage: > 85% on core AI Gateway logic

### Performance
- Chat completion (non-streaming): < 5 seconds p95
- Streaming first token: < 2 seconds p95
- Embedding (batch of 10): < 3 seconds p95
- Route overhead (auth + validation + rate limit): < 50 ms p95
- Rate limit check: < 5 ms p95
- Fallback decision: < 5 ms per attempt

### Security
- Auth: all routes require valid session
- Rate limit: enforced before processing
- No API keys in test output
- Request size limits enforced

## Architecture Constraints

- Integration tests against real OpenRouter API use a test key with a spending limit
- Rate limiter tests use a separate Redis database or Redis mocking
- Fallback tests mock provider failures at the HTTP client level
- Streaming tests verify both SSE format and content correctness
- Cost tracker tests verify pricing against a known model configuration
- Performance benchmarks run 3 iterations and take the median

## Acceptance Criteria

- All unit tests pass (> 85% line coverage on core logic)
- End-to-end test: chat completion through full pipeline (auth → rate limit → router → provider → cost → response)
- Streaming test: SSE output delivers correct tokens in order
- Fallback test: primary failure → fallback succeeds → correct model in response metadata
- Cost test: record written to MongoDB with correct workspaceId, model, tokens, cost
- Rate limiter test: > 30 req/min blocked, < 30 req/min allowed
- Prompt test: template rendering with variables, conditionals, iteration all correct
- API route tests: auth (401), validation (400), rate limit (429), success (200)
- Performance benchmarks: all latency targets met
