## Goal

Implement the fallback chain that activates when the primary model/provider fails. The fallback chain tries alternative models/providers in priority order, with circuit breaker protection to avoid repeatedly failing models. It ensures high availability of AI inference even when specific models or providers are degraded or down.

## Business Context

AI providers experience outages, rate limits, and degraded performance. A production AI gateway must handle these failures transparently. The fallback chain ensures that a user's request succeeds on the first available model rather than failing with an obscure provider error. This is critical for enterprise SLAs and user trust.

## Technical Context

The fallback chain is configured per task type. For example, chat fallback: claude-sonnet-4 → gpt-4o → gemini-2.0-flash → claude-haiku-3.5. Each entry in the chain specifies the model and provider. The chain respects a circuit breaker per model: if a model fails 3 consecutive times within 5 minutes, it's marked as degraded and skipped in the chain for 60 seconds. The fallback service also tracks per-request fallback depth for observability.

## Requirements

### Functional
- `createFallbackChain(primaryModel, options)` — build a fallback chain for a request
- Fallback chain order for chat: `claude-sonnet-4 → gpt-4o → gemini-2.0-flash → claude-haiku-3.5 → gpt-4o-mini`
- Fallback chain order for embedding: `text-embedding-3-large → voyage-code-2`
- Fallback chain is configurable per task type via environment or configuration
- Circuit breaker per (model, workspace) pair:
  - Track consecutive failures
  - After 3 consecutive failures within 5 minutes: open circuit (skip model) for 60 seconds
  - After 60 seconds: half-open (try one request) → if success: close circuit; if failure: reopen for 120 seconds
  - Exponential backoff on circuit reopen: 60s → 120s → 240s → max 600s
- `executeWithFallback(primaryModel, request)` — execute request with fallback chain
  - Try primary model → on failure → try next in chain → repeat until success or chain exhausted
  - Return `FallbackResult`: `{ response, modelUsed, fallbackDepth: number, totalAttempts: number, errors: { model, error }[] }`
- Only fallback on retryable errors: timeout, rate limit (429), server error (5xx)
- Do NOT fallback on: auth errors (401), bad request (400), content filter (422)
- Log each fallback attempt with model, error, latency, and fallback depth
- Update model health in registry on failure/success

### Non-functional
- Fallback decision overhead: < 5 ms per attempt
- Circuit breaker state stored in Redis (distributed across workers)
- Fallback chain traversal: typically 0 additional latency (primary succeeds)

### Performance
- Fallback attempts are sequential (not parallel) to minimize cost
- Each attempt has its own timeout (same as primary)

### Security
- Fallback models must respect same user tier restrictions as primary
- Circuit breaker state is per workspace to prevent noisy neighbor issues

## Architecture Constraints

- Circuit breaker state key: `circuit:{workspaceId}:{modelId}` — scoped to workspace
- State stored in Redis as JSON: `{ open: boolean, failureCount: number, lastFailureAt: timestamp, openedAt: timestamp }`
- The fallback chain only activates if the provider abstraction throws a retryable error
- Non-retryable errors are thrown immediately (no fallback attempted)
- If all models in chain fail, the last error is thrown with accumulated errors in metadata

## Acceptance Criteria

- Primary model succeeds → no fallback latency added
- Primary model fails with timeout → fallback to next model, success
- All models in chain fail → final error thrown with accumulated errors
- Circuit breaker opens after 3 consecutive failures on same model
- Open circuit skips the model for 60 seconds
- Half-open state tries one request; success closes circuit
- Circuit breaker state persists across worker restarts (Redis-backed)
- Non-retryable errors (400, 401) skip fallback and throw immediately
- FallbackResult includes correct fallback depth and error details

## Notes

- Circuit breaker state machine: CLOSED (normal) → OPEN (failures > threshold) → HALF_OPEN (after timeout) → CLOSED (success) or OPEN (failure)
- Exponential backoff for circuit reopen: `Math.min(60000 * Math.pow(2, attempts), 600000)` — capped at 10 minutes
- Retryable error check: `error.retryable === true` — set by provider abstraction
- For the fallback chain config, allow environment variable override: `AI_FALLBACK_CHAIN_CHAT` as comma-separated model IDs
