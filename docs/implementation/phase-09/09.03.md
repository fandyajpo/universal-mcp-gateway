## Goal

Implement the model router that selects the optimal model for each request based on task type, cost, capability requirements, user tier, and current provider health. The router maintains a model capability registry, applies routing rules, and provides the selected model and provider to the execution layer.

## Business Context

Different AI tasks require different models. Complex reasoning needs powerful models (Claude, GPT-4o), while simple classification can use cheaper models (Claude Haiku, GPT-4o-mini). The model router ensures cost-efficiency by selecting the cheapest model capable of handling each request. It also handles fallback scenarios (preferred model unavailable) and tier-based restrictions (free users get cheaper models).

## Technical Context

The router maintains a registry of models with their capabilities, pricing, and current health status. When a request arrives, the router evaluates the request's requirements against the registry to find the optimal model. The routing logic follows a priority chain: user tier restrictions → task capability requirements → cost optimization → provider health check → final selection.

## Requirements

### Functional
- Maintain model registry: `{ modelId: string, providerId: string, capabilities: string[], contextWindow: number, pricing: { prompt: number, completion: number }, tier: string[], health: "healthy" | "degraded" | "down" }`
- Pre-populate registry with common models and their capabilities:
  - `openai/gpt-4o` — full capabilities, high cost
  - `openai/gpt-4o-mini` — most capabilities, low cost
  - `anthropic/claude-sonnet-4` — full capabilities, high cost (coding, reasoning)
  - `anthropic/claude-haiku-3.5` — chat, low cost
  - `google/gemini-2.0-flash` — full capabilities, medium cost
  - `openai/text-embedding-3-large` — embedding only
- `route(request: RouteRequest): RouteResult` — select best model and provider
- RouteRequest: `{ taskType: "chat" | "completion" | "embedding", capabilities: string[], maxCost?: number, preferredModel?: string, userTier: "free" | "pro" | "enterprise" }`
- RouteResult: `{ model: string, provider: string, estimatedCost: number, reasoning: string }`
- Apply user tier restrictions: free → only cheap models, pro → all models, enterprise → all models + custom models
- Apply task capability filter: only models with required capabilities
- Apply cost optimization: select cheapest model that meets all requirements
- Apply health check: skip models with `down` health; prefer `healthy` over `degraded`
- Support preferred model override (if available and meets requirements)
- Cache model registry in Redis with 5-minute TTL (refresh from API periodically)
- Log routing decisions: request characteristics, selected model, alternatives considered

### Non-functional
- Route decision time: < 10 ms (mostly registry lookup, no API calls)
- Registry cache updates: every 5 minutes via background refresh

### Performance
- Registry stored in memory (with Redis cache backing for persistence across restarts)

### Security
- Tier enforcement: free users cannot access expensive models
- Model allowlist: enterprise admin can restrict available models per workspace

## Architecture Constraints

- The router never calls external APIs during routing (no latency impact)
- Registry is loaded from a static default set and updated asynchronously from OpenRouter model list
- Tier restrictions are defined in a configuration map: `{ free: { maxCost: 0.0001, allowlist: ["gpt-4o-mini", "claude-haiku"] }, pro: { maxCost: 0.01, allowlist: [] }, enterprise: { maxCost: Infinity, allowlist: [] } }`
- Health status is tracked in Redis and updated by the circuit breaker in the fallback service (Step 09.05)
- Cost optimization uses prompt + estimated completion tokens to calculate total cost

## Acceptance Criteria

- `route({ taskType: "chat", userTier: "free" })` returns a cheap model (e.g., gpt-4o-mini)
- `route({ taskType: "chat", userTier: "pro" })` returns a capable model (e.g., claude-sonnet-4)
- `route({ capabilities: ["vision"] })` returns only models with vision capability
- `route({ preferredModel: "gpt-4o" })` returns gpt-4o if healthy and meets requirements
- `route` with `maxCost` restriction returns model under cost limit
- Unhealthy model (`health: "down"`) is skipped in routing
- Tier restriction: free user requesting expensive model gets restricted to cheap model
- Route decision includes reasoning string explaining the selection
