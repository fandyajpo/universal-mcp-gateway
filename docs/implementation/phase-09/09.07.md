## Goal

Implement rate limiting for the AI Gateway to prevent abuse and ensure fair resource allocation across users and workspaces. The rate limiter uses a token bucket algorithm with Redis-backed state, enforcing limits at three tiers: global, per-workspace, and per-user. Excessive requests are throttled with appropriate backoff responses.

## Business Context

AI model calls are expensive and limited by provider rate limits. Without rate limiting, a single user or workspace could consume the entire API budget, degrade performance for other users, or trigger provider bans. Rate limiting ensures fair allocation of AI resources across the platform.

## Technical Context

The rate limiter uses a token bucket algorithm implemented in Redis for atomic operations and distributed state. Three tiers are enforced simultaneously: global (platform-wide), per-workspace (tenant-level), and per-user (individual). Each tier has configurable capacity (burst) and refill rate (sustained). Redis Lua scripts ensure atomic token consumption.

## Requirements

### Functional
- Token bucket algorithm: bucket capacity = max burst, refill rate = sustained rate per second
- Three tiers:
  - Global: 1000 requests/minute, burst 50 (across all workspaces)
  - Per-workspace: 100 requests/minute, burst 20
  - Per-user: 20 requests/minute, burst 5
- `checkLimit(key, cost, options): { allowed: boolean, remaining: number, resetAt: timestamp }` — check if request is allowed
- `consume(key, cost): Promise<RateLimitResult>` — consume tokens from bucket
- `resetLimit(key)` — admin function to reset rate limit for a key
- Rate limit keys: `ratelimit:ai:global`, `ratelimit:ai:workspace:{workspaceId}`, `ratelimit:ai:user:{userId}`
- All three tiers must ALL allow for the request to proceed (AND logic)
- Cost per request: 1 for simple queries, adjustable per task type (embedding = 2, chat = 1, streaming = 1)
- Return `429 Too Many Requests` with headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`
- Log rate limit events: when limit exceeded, by which tier, remaining count
- Support bypass for internal/admin requests (configurable via API key type)
- Dynamic rate limit configuration: limits can be changed without restart (stored in Redis, with environment variable fallback)
- Sliding window: rate limits reset on a rolling 60-second window (not fixed minute boundaries)

### Non-functional
- Rate limit check latency: < 5 ms (single Redis Lua script call)
- No false positives: rate limiter must never block requests that should be allowed

### Performance
- Lua script: single Redis round trip for check + consume
- All three tier checks done in parallel via Redis pipeline

### Security
- Rate limit keys are deterministic — no data leakage
- Rate limit state is scoped — global limits don't reveal workspace traffic patterns

### Accessibility
- N/A (internal rate limiting)

## Architecture Constraints

- All three tiers are checked via a single Lua script for atomicity
- Limits are configured in Redis key `ratelimit:ai:config` as JSON, with env var fallback
- Token consumption: `tokens = max(0, bucket.tokens - (now - bucket.lastRefill) * refillRate)` then `bucket.tokens = min(capacity, tokens - cost)`
- The Lua script returns `{ allowed: 0|1, remaining: number, ttl: number }`
- If Redis is unavailable, the rate limiter should allow requests (fail-open) but log a critical warning

## Notes

- Lua script for token bucket:
  ```lua
  local key = KEYS[1]
  local capacity = tonumber(ARGV[1])
  local refillRate = tonumber(ARGV[2])
  local cost = tonumber(ARGV[3])
  local now = tonumber(ARGV[4])
  local bucket = redis.call("GET", key)
  if not bucket then
    bucket = { tokens = capacity - cost, lastRefill = now }
    redis.call("SET", key, cjson.encode(bucket), "EX", 60)
    return { 1, capacity - cost, 60 }
  end
  bucket = cjson.decode(bucket)
  local elapsed = math.max(0, now - bucket.lastRefill)
  bucket.tokens = math.min(capacity, bucket.tokens + elapsed * refillRate)
  if bucket.tokens >= cost then
    bucket.tokens = bucket.tokens - cost
    bucket.lastRefill = now
    redis.call("SET", key, cjson.encode(bucket), "EX", 60)
    return { 1, bucket.tokens, 60 }
  else
    local retryAfter = math.ceil((cost - bucket.tokens) / refillRate)
    return { 0, bucket.tokens, retryAfter }
  end
  ```
- Fail-open behavior: wrap Redis calls in try-catch; on Redis error, log critical and return `{ allowed: true }`
- Dynamic config stored in Redis at `ratelimit:ai:config` with 300-second TTL, refreshed on each check
