## Goal

Implement cost tracking for all AI model calls (chat, completion, embedding). The cost tracker records per-request token usage and cost to MongoDB, provides aggregation queries for workspace-level cost reports, and feeds cost data into the billing system (Phase 14). This is similar to the embedding cost tracker (07.05) but generalized for all AI Gateway operations.

## Business Context

AI model costs are a significant operational expense. The cost tracker provides visibility into which workspaces are consuming the AI budget, which models are most expensive, and how costs trend over time. Cost data is essential for metered billing (Phase 14), workspace budget management, and operational cost optimization.

## Technical Context

The cost tracker intercepts responses from the provider abstraction, extracts token usage (prompt_tokens, completion_tokens), computes cost using model pricing, and writes a cost record to MongoDB. It handles batching for efficiency and provides aggregation queries for reports. The tracker is non-blocking — cost recording happens asynchronously after the response is sent to the consumer.

## Requirements

### Functional
- Record cost entry for each AI call: `{ workspaceId, userId, requestId, model, provider, taskType: "chat" | "embedding", promptTokens, completionTokens, totalTokens, cost, timestamp, metadata?: { streamed, cached, fallbackDepth } }`
- Look up model pricing from the model registry (prompt and completion per-token pricing)
- Compute cost: `cost = (promptTokens * promptPrice + completionTokens * completionPrice) / 1_000_000`
- Handle streaming: record final cost after stream completes (don't update during streaming)
- Handle caching (if provider returns cached response with reduced cost): check `usage.prompt_tokens_details.cached_tokens` for OpenAI — apply 50% discount on cached tokens
- Non-blocking writes: cost recording must not add latency to the AI response
- Batching: buffer cost records and flush to MongoDB every 10 seconds or 100 records
- Aggregation queries:
  - `getWorkspaceCost(workspaceId, startDate, endDate)` — total cost for workspace
  - `getModelCostBreakdown(startDate, endDate)` — cost per model across all workspaces
  - `getUserCost(userId, startDate, endDate)` — cost per user
  - `getDailyCostTrend(workspaceId, days)` — daily cost for trend charts
- Store cost records in MongoDB `ai_costs` collection
- Handle missing pricing data: log warning and estimate based on similar model

### Non-functional
- Cost write overhead: < 5 ms per record (via batch buffer)
- Aggregation queries: < 500 ms for monthly data on 1M+ records
- MongoDB indexes: `{ workspaceId: 1, timestamp: -1 }`, `{ model: 1, timestamp: -1 }`, `{ userId: 1, timestamp: -1 }`

### Performance
- Batch buffer: flush every 10 seconds or 100 records, whichever comes first
- Cost records are write-only (append-only), no updates

### Security
- Cost records are workspace-scoped — cross-workspace aggregation requires admin role
- No cost data in client-facing responses without authorization

### Accessibility
- N/A (internal tracking)

## Architecture Constraints

- Cost writes are asynchronous (fire-and-forget) — never await cost record insertion
- The batch buffer is flushed on process exit (graceful shutdown) to avoid data loss
- Cost tracking must handle partial token usage (some provider responses may not include usage) — fall back to estimation
- Estimation fallback: `estimatedPromptTokens = Math.ceil(messagesString.length / 4)` for English text
- Cost data TTL: 365 days (MongoDB TTL index), adjustable per workspace for enterprise customers
