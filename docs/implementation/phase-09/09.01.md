## Goal

Implement the OpenRouter HTTP client that serves as the primary interface for all AI model interactions. The client wraps OpenRouter's API (chat completions, embeddings) with retry logic, timeout handling, error classification, and response parsing. It provides a clean, typed interface that the provider abstraction layer (Step 09.02) builds upon.

## Business Context

OpenRouter is the single gateway to 300+ AI models. The client abstracts away HTTP details, authentication, error handling, and response parsing. Every AI interaction in the platform — chat, completion, embedding — goes through this client. Reliability is critical: timeouts, retries, and error classification must handle all failure modes without propagating raw API errors to users.

## Technical Context

The client uses the standard `fetch` API (available in Node.js 20+) with AbortController for timeouts. It communicates with OpenRouter's REST API at `https://openrouter.ai/api/v1`. Key headers: `Authorization: Bearer {key}`, `HTTP-Referer` (for OpenRouter ranking), `X-Title` (for identification). The client supports both JSON and SSE streaming responses.

## Requirements

### Functional
- `POST /v1/chat/completions` — chat completions (JSON and streaming modes)
- `POST /v1/embeddings` — text embeddings
- `GET /v1/models` — list available models with capabilities and pricing
- Authentication via `OPENROUTER_API_KEY` environment variable
- Configurable request timeout: default 60 seconds for chat, 30 seconds for embeddings
- Retry logic: max 3 retries, exponential backoff (1s, 2s, 4s), with jitter
- Retry only on: 429 (rate limit), 503 (service unavailable), 5xx (server errors)
- Do NOT retry on: 400 (bad request), 401 (auth), 404 (not found), 422 (validation error)
- Parse `Retry-After` header on 429 responses
- Response parsing: typed response objects with proper error handling
- Error classification: categorize errors as `rate_limited`, `timeout`, `auth_error`, `server_error`, `invalid_request`, `unknown`
- Support custom headers: `HTTP-Referer`, `X-Title` for OpenRouter identification
- Log all API calls with method, model, token count, latency, status code

### Non-functional
- Request setup overhead: < 5 ms
- Streaming response parsing: no buffering — emit chunks as they arrive
- Error classification: < 1 ms

### Performance
- Connection pooling: use `keep-alive` header for connection reuse
- Timeout enforcement via AbortController with configurable timeout per request type

### Security
- API key never logged directly (log only `key=[redacted]`)
- API key loaded from environment, never from request input
- No credentials in error messages returned to callers

### Accessibility
- N/A (internal client)

## Architecture Constraints

- The client is stateless — it doesn't track rate limits or costs (those are separate services)
- Streaming responses use `response.body.getReader()` for manual chunk reading
- The client never buffers entire streaming responses — chunks are forwarded as received
- All methods return typed Promise/AsyncIterable — never raw Response objects
- The client should be mockable for tests (injectable fetch function)

## Acceptance Criteria

- `chatCompletion({ model, messages })` returns typed response with content and usage
- `chatCompletionStream({ model, messages })` yields typed chunks via AsyncIterable
- `embed({ model, input })` returns embedding vector and token usage
- `listModels()` returns array of available models with IDs and pricing
- 429 response triggers retry with exponential backoff
- 401 response immediately throws `AuthError` (no retry)
- Request timeout aborts and throws `TimeoutError` after configured duration
- Streaming correctly parses SSE `data:` lines and emits parsed chunks
- All API calls logged with model, token count (when available), latency, status

## Notes

- Chat completion request: `{ model, messages: [{role, content}], stream?: boolean, max_tokens?, temperature?, ... }`
- SSE stream parsing: each line is `data: {json}` or `data: [DONE]`
