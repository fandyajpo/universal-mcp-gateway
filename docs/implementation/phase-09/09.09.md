## Goal

Implement REST API routes that expose the AI Gateway's capabilities to clients. The routes provide chat completion, streaming, embedding, and model listing endpoints. All routes require authentication and workspace membership, apply rate limiting, and return responses in a consistent format.

## Business Context

The AI Gateway must be accessible from multiple consumers: the Chat UI (Phase 12), the RAG Engine (for generation), the Admin dashboard (for testing), and potentially third-party integrations. REST API routes provide a standardized interface for all consumers.

## Technical Context

The API routes follow Next.js App Router conventions in `apps/web`. They delegate to the AI Gateway composition (Step 09.10) for business logic, apply auth middleware from `@repo/auth`, validate requests with Zod (from `@repo/validation`), and apply rate limiting from Step 09.07.

## Requirements

### Functional
- `POST /api/ai/chat` — chat completion (non-streaming)
  - Request: `{ model?, messages, temperature?, max_tokens?, stream?: false }`
  - Response: `{ id, choices: [{ message, finish_reason }], usage, model }`
- `POST /api/ai/chat/stream` — streaming chat completion
  - Request: same as above with `stream: true`
  - Response: SSE stream of token deltas and finish event
- `POST /api/ai/embed` — text embedding
  - Request: `{ model?, input: string | string[] }`
  - Response: `{ data: [{ embedding, index }], model, usage }`
- `GET /api/ai/models` — list available models
  - Response: `{ data: [{ id, name, capabilities, pricing, contextWindow }] }`
- All routes apply:
  - Authentication (valid session required)
  - Workspace membership validation
  - Rate limiting (per user tier)
  - Request validation (Zod schemas)
  - Consistent error format: `{ error: { code, message, details? } }`
- Support model routing: if no model specified, the router selects the best model
- Support fallback: if primary model fails, the fallback chain activates transparently
- Include usage metadata in responses: model used, token counts, cost (for authenticated responses)

### Non-functional
- Chat endpoint (non-streaming): < 5 seconds p95 including model inference
- Streaming: first token within < 2 seconds p95
- Embedding: < 3 seconds p95 for 32 texts
- Model listing: < 100 ms (reads from cache)

### Performance
- Streaming route uses `Transfer-Encoding: chunked` or `text/event-stream` content type
- Rate limiting check at route entry, before any processing

### Security
- Authentication on all routes
- Rate limiting per user
- Request body size limit: 1 MB
- Max messages: 100 per request
- Max prompt tokens: 128K (configurable)

### Accessibility
- N/A (API endpoints)

## Architecture Constraints

- The streaming route is the most critical — it must handle SSE correctly with proper headers and keepalive
- Non-streaming route is a convenience wrapper around streaming: collect all chunks and return as JSON
- Rate limiting is applied at the route level (before model routing) to avoid unnecessary computation
- The embedding route delegates to the embedding service (07.01) for actual computation
- Model listing reads from the model registry cache (Step 09.03)

## Acceptance Criteria

- `POST /api/ai/chat` returns chat completion with correct format
- `POST /api/ai/chat/stream` returns SSE stream with token-by-token delivery
- `POST /api/ai/embed` returns embedding vectors
- `GET /api/ai/models` returns model list from registry
- Unauthenticated requests return 401
- Rate limited requests return 429 with correct headers
- Invalid request body returns 400 with validation details
- Model routing selects correct model when no model specified
- Fallback chain activates when primary model fails

## Notes

- Streaming SSE response headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`
- In Next.js App Router, create a streaming response:
  ```typescript
  const stream = new ReadableStream({ start(controller) { /* push data */ } });
  return new Response(stream, { headers: { "Content-Type": "text/event-stream" } });
  ```
- Keepalive: send a comment line (`: keepalive\n\n`) every 15 seconds if no data is being sent
- The streaming route should detect client disconnect via `request.signal.aborted`
