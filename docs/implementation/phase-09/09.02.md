## Goal

Implement a provider abstraction layer that defines a unified interface for AI model providers, with OpenRouter as the primary implementation. The layer allows the gateway to treat all providers uniformly — switching between OpenRouter, direct provider APIs, or future providers requires changing only the provider adapter, not the consuming code.

## Business Context

While the platform primarily uses OpenRouter, certain use cases may require direct provider access (e.g., Anthropic's Claude for high-security workloads, Google's Gemini for specific capabilities). The abstraction layer ensures the gateway doesn't become tied to OpenRouter — providers can be swapped, added, or removed without changing model routing, cost tracking, or the chat service.

## Technical Context

The provider abstraction defines a `Provider` interface with methods for chat completion, streaming, embedding, and model listing. Each provider implementation (starting with OpenRouter) adapts the provider-specific API to the unified interface. The provider factory (`createProvider()`) instantiates the correct provider based on configuration. The interface uses discriminated types for request options (different providers support different parameters).

## Requirements

### Functional
- Define `Provider` interface with methods:
  - `chatCompletion(request: ChatRequest): Promise<ChatResponse>`
  - `chatCompletionStream(request: ChatRequest): AsyncGenerator<StreamChunk>`
  - `embed(request: EmbedRequest): Promise<EmbedResponse>`
  - `listModels(): Promise<Model[]>` — models available through this provider
- Define request types that are common across providers (`ChatRequest`, `EmbedRequest`) with provider-specific extensions via discriminated union
- Define response types common across providers (`ChatResponse`, `EmbedResponse`, `StreamChunk`)
- Implement `OpenRouterProvider` adapter implementing `Provider` interface
- Implement `createProvider(type: ProviderType, config: ProviderConfig): Provider` factory
- Support provider configuration: API key, base URL, default model, timeout, retry config
- Provider metadata: `{ id: string, name: string, capabilities: string[], models: Model[] }`
- Capabilities: `chat`, `streaming`, `embedding`, `function_calling`, `vision`, `json_mode`
- Error mapping: provider-specific errors mapped to standard `ProviderError` types
- Graceful handling of unsupported features (e.g., vision for text-only provider returns capability error)

### Non-functional
- Factory instantiation: < 1 ms
- Provider method overhead (wrapping): < 2 ms per call

### Performance
- Provider methods are thin wrappers — no additional latency beyond the underlying client

### Security
- Provider config (API keys) never exposed outside the provider implementation
- Error messages sanitized — no provider credentials in error output

### Accessibility
- N/A (internal abstraction)

## Architecture Constraints

- The `Provider` interface is the ONLY way consuming code interacts with AI models — never use provider-specific clients directly
- Provider implementations are stateless singletons (one instance per provider configuration)
- The factory reads configuration from environment variables with provider-specific keys
- Provider capabilities are declared statically (not discovered at runtime)
- Unsupported features throw `CapabilityError` with a clear message

## Acceptance Criteria

- `createProvider("openrouter", config)` returns an `OpenRouterProvider`
- `OpenRouterProvider.chatCompletion()` works identically to direct client call
- `OpenRouterProvider.chatCompletionStream()` yields correct stream chunks
- `OpenRouterProvider.listModels()` returns models from OpenRouter
- Calling `.embed()` on a provider without embedding capability throws `CapabilityError`
- Error mapping converts OpenRouter errors to standard `ProviderError` types
- Provider metadata correctly declares capabilities
- Factory with invalid provider type throws clear error
