## Goal

Identify and eliminate all N+1 query patterns across the platform. Implement batch loading, data loader pattern, and optimized aggregation pipelines to replace iterative queries with efficient batch operations.

---

## Business Context

N+1 query patterns are one of the most common and impactful performance anti-patterns in database-backed applications. They occur when a query is executed in a loop, resulting in N additional queries for N items. In a listing page with 50 items that each trigger a related query, this becomes 51 queries instead of 2. N+1 patterns are often invisible during development (when datasets are small) and become severe performance problems at production scale.

---

## Technical Context

N+1 patterns in MongoDB are typically caused by: loading a parent document and then iterating to load child documents, resolving references without using `$lookup`, service-layer loops that query repositories individually, and GraphQL-like resolvers that fetch related data per item.

The elimination strategy: (1) identify N+1 patterns via query logging and code review, (2) replace loop queries with `$in` batch queries, (3) use MongoDB aggregation `$lookup` for joins, (4) implement a DataLoader pattern for batch loading, (5) add query batching utilities to repositories.

## Requirements

### Functional requirements

- Scan all repository methods for N+1 patterns (queries inside loops)
- Scan all service methods for N+1 patterns (service-level loops calling repositories)
- Scan all API route handlers for N+1 patterns
- Replace loop-based queries with batch queries using `$in` operator
- Replace manual reference resolution with MongoDB `$lookup` aggregation
- Implement DataLoader pattern for batch loading with request-scoped caching
- Add batch loading methods to repositories (e.g., `findByIds()`, `findByWorkspaceIds()`)
- Add request-scoped cache to prevent duplicate queries within the same request
- Add query counting middleware to detect N+1 patterns in development
- Create N+1 detection test that fails if any route exceeds a query count threshold

### Non-functional requirements

- N+1 detection must not add overhead in production (disabled via NODE_ENV)
- DataLoader cache must be request-scoped (cleared per request)
- Batch loading must handle empty result sets correctly
- All existing query patterns must be reviewed

### Performance requirements

- Eliminate all N+1 patterns in the codebase
- Reduce query count per page load by 80%+
- Reduce total query execution time by 60%+

### Security requirements

- Batch queries must respect tenant isolation (include tenantId in batch queries)
- DataLoader cache must not leak data between tenants

### Accessibility requirements

- No accessibility impact

- DataLoader must use Redis as the backing store for request-scoped cache (with request ID as key prefix)
- Batch query methods must accept an array of IDs and return a Map<ID, T> for efficient lookup
- Aggregation pipelines with `$lookup` must always include `$match` before `$lookup` where possible
- The N+1 detection middleware counts every `findOne`/`findById` call and warns if a single request exceeds 10 such calls
- Request cache is cleared at the beginning and end of each request

## Notes

N+1 patterns are notoriously easy to introduce and hard to catch. The N+1 detection middleware is the most important deliverable — it should log a warning for any request that makes more than N database queries (N = 10 for development, 5 for CI). The DataLoader pattern from GraphQL/Facebook is the standard solution — implement a simple version that collects keys and dispatches a single batch query. The request-scoped cache prevents the same document from being loaded multiple times within a single request (e.g., when a user object is referenced by multiple child documents).
