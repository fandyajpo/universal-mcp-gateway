## Goal

Implement a comprehensive CDN caching strategy using Cloudflare. Configure cache rules, cache tags, surrogate-key-based purging, edge caching for static assets, and stale-while-revalidate for dynamic content.

---

## Business Context

CDN caching is the most impactful performance optimization for a globally distributed user base. A well-configured CDN serves content from edge locations near the user, dramatically reducing latency and offloading origin servers. The platform serves a mix of static assets (JS, CSS, images), dynamic API responses, and user-generated content — each requires a different caching strategy.

---

## Technical Context

The platform uses Cloudflare as the CDN and reverse proxy. Cloudflare's caching capabilities include: static asset caching with immutable cache tags, cache key customization (bypass cache on auth), edge caching for API responses with Cache-API, cache tags for granular purging, and stale-while-revalidate for background refresh. The implementation configures Cloudflare rules via the Cloudflare API and sets appropriate Cache-Control headers from the application.

## Requirements

### Functional requirements

- Static asset caching: JS, CSS, fonts, images served from `/public` and `/_next/static` with `Cache-Control: public, max-age=31536000, immutable`
- Cache key customization: differentiate cache entries by language, device type, and authentication state
- Edge caching for API: cache public API responses (marketplace listings, health check, public docs) at the edge with configurable TTL
- Authenticated content bypass: never cache responses that contain user-specific data
- Cache tags: set `Cache-Tag` headers on all cacheable responses for granular purging
- Surrogate-key purging: use Cloudflare API to purge cache by tags when content changes (listing updated, document modified)
- Stale-while-revalidate: serve stale content while fetching fresh version in the background
- HTML caching: cache full HTML pages for public routes (docs, landing) using Cloudflare's Edge Cache HTML
- Cache purging API: internal endpoint for purging specific cache tags
- Cache hit ratio monitoring: track and report cache hit ratio per route pattern

### Non-functional requirements

- Static asset cache hit ratio: > 99%
- API edge cache hit ratio: > 80% for public endpoints
- Cache purge propagation: < 5 seconds globally
- Stale-while-revalidate TTL: 1 hour stale, 24 hours max age

### Performance requirements

- Origin offload: > 70% of requests served from CDN cache
- TTFB reduction: 50%+ for cached responses
- Cache purge latency: < 5s

### Security requirements

- Authenticated content must never be cached at the CDN
- Cache keys must not include session tokens or PII
- Cache purging API requires authentication
- HTTP Cache-Control headers must not be set on sensitive endpoints

### Accessibility requirements

- No accessibility impact — caching does not affect rendered content

## Package Boundaries

A new `@repo/cdn` package provides CDN caching, purging, and configuration utilities. It depends on `@repo/config` (Cloudflare API credentials), `@repo/logger`, and `@repo/inngest` (cache warming cron). The apps use the CDN package for cache tag management and purging.

---

## Architecture Constraints

- Cache tags follow the pattern: `{domain}-{entity}-{id}` (e.g., `marketplace-listing-slug`, `doc-document-id`)
- Cache purging is done via Cloudflare API `POST /zones/{zone_id}/purge_cache` with `tags` parameter
- Authenticated routes must set `Cache-Control: no-store, private` to prevent CDN caching
- Public routes must set `Cache-Control: public, max-age={ttl}, s-maxage={cdn_ttl}, stale-while-revalidate={swr_ttl}`
- HTML caching is configured through Cloudflare Page Rules (not application headers)
- Cache warming runs on a scheduled Inngest function to keep popular content hot

## Notes

CDN caching configuration requires Cloudflare access. Ensure the Cloudflare API token with Zone.Cache Purge permission is configured in environment variables. Cache tag purging is more expensive than purge by URL — use tags only for content that changes infrequently. For rapidly changing content, use shorter TTLs with stale-while-revalidate instead. Monitor cache hit ratio in the Cloudflare dashboard — a ratio below 80% indicates misconfigured caching that needs investigation.
