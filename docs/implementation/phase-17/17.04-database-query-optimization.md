## Goal

Systematically analyze and optimize all database queries across the platform. Identify slow queries, optimize indexing strategies, restructure inefficient aggregation pipelines, and improve query execution times.

---

## Business Context

Database performance directly impacts every user-facing feature. Slow queries cause page load delays, API timeouts, and poor user experience. As the platform has accumulated 16 phases of features, query patterns have grown without systematic optimization. This step brings all queries to production-grade performance.

---

## Technical Context

MongoDB Atlas provides built-in query profiling via `system.profile` collection and the Atlas Performance Advisor. The optimization process: (1) enable query profiling to capture slow queries, (2) analyze slow query logs to identify problematic patterns, (3) examine query execution plans using `explain()`, (4) add/optimize indexes, (5) restructure queries, (6) verify improvement with before/after execution stats.

## Requirements

### Functional requirements

- Enable MongoDB query profiling for development/staging environments
- Capture and analyze slow queries (execution time > 100ms)
- Examine execution plans for top 20 slowest queries
- Identify missing indexes: queries without `IXSCAN` (using `COLLSCAN`)
- Add compound indexes for common query patterns
- Optimize existing indexes: remove unused indexes, restructure compound indexes for selectivity
- Restructure aggregation pipelines: move `$match` stages earlier, limit documents before `$lookup`, use `$project` to reduce pipeline documents
- Optimize `$lookup` operations: add indexes on foreign fields, use `$lookup` with `pipeline` for filtered joins
- Implement query timeouts for long-running operations
- Configure MongoDB `maxTimeMS` for all queries
- Add query performance monitoring (slow query logging with Pino)

### Non-functional requirements

- Query p95 execution time: < 100ms for all indexed queries
- No queries should use COLLSCAN (full collection scan) in production
- Aggregation pipeline execution: < 500ms for all pipelines
- Indexes must be managed through schema definitions (not manual MongoDB commands)

### Performance requirements

- P95 query time reduction: 80%+ from baseline
- Zero COLLSCAN queries in production
- Aggregation pipeline optimization: 50%+ reduction in execution time

### Security requirements

- Query profiling must not capture sensitive data in queries
- Profiling data retention: 7 days maximum
- Index management must be idempotent

### Accessibility requirements

- No accessibility impact

## Architecture Constraints

- Indexes must be defined in Mongoose schema files using `schema.index()` — never create indexes manually in MongoDB
- Compound indexes must follow ESR (Equality, Sort, Range) rule for field ordering
- Aggregation pipelines must use `$match` as the first stage whenever possible
- `$lookup` must have indexes on the foreign field
- `$unwind` must be avoided where possible (use `$lookup` with `pipeline` instead)
- Query timeout (`maxTimeMS`) of 30 seconds must be set on all queries

## Notes

Use MongoDB Atlas Performance Advisor as a starting point — it automatically identifies slow queries and suggests indexes. However, rely on the application-level query analysis as the definitive source. The biggest gains typically come from: adding compound indexes for multi-field queries, restructuring aggregation pipelines to filter early, and optimizing `$lookup` with targeted indexes. Document each optimization with before/after execution time for the performance report.
