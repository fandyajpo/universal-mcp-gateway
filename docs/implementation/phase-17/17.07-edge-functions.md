## Goal

Identify compute-heavy operations that can be moved to edge functions (Cloudflare Workers) and implement them. Edge functions provide geo-distributed execution with near-zero cold starts, reducing latency for global users and offloading origin servers.

---

## Business Context

Edge functions execute close to the user, dramatically reducing latency for compute-heavy operations that do not require access to the origin database or application state. Common candidates: authentication token verification, request transformation, geolocation-based routing, A/B testing, rate limiting, and content personalization.

---

## Technical Context

The platform uses Next.js Edge Runtime (available in Middleware and Edge API Routes) and Cloudflare Workers for independent edge functions. Next.js Edge Runtime is limited in API availability (no Node.js APIs) but has access to the request/response pipeline. Cloudflare Workers provide a full edge compute platform with access to Cloudflare's network.

## Requirements

### Functional requirements

- Identify compute-heavy operations suitable for edge execution: session validation (already edge), rate limiting (already edge), geolocation-based routing, URL rewriting/redirects, A/B testing assignment, request authentication, content negotiation (language, format), simple API responses (health check, configuration), request transformation (headers, cookies)
- Move session validation fully to edge middleware (verify JWT/Redis at edge)
- Implement geolocation-aware routing: serve different content or redirect based on user country
- Implement A/B testing assignment at edge: assign variant, set cookie, route to appropriate handler
- Implement edge-based rate limiting using KV store or durable objects
- Implement edge-based health check endpoint (returns 200 without hitting origin)
- Implement edge API routes for: geolocation lookup, feature flag evaluation, public configuration
- Optimize Next.js Edge Middleware to minimize compute time (target: < 5ms)

### Non-functional requirements

- Edge function execution must complete within 50ms (including network)
- Edge middleware must not block request processing — fail open if edge function errors
- Edge cache must be used for deterministic edge responses
- Edge function must not have access to the primary database or secrets

### Performance requirements

- Edge middleware execution: < 5ms
- Edge API route response: < 20ms
- Geolocation lookup: < 10ms
- A/B test assignment: < 5ms

### Security requirements

- Edge functions must not access the primary database
- Edge functions must not have access to full secret keys (only derived tokens)
- Edge-authenticated requests must pass verification status to origin via headers
- Fail-open behavior must not bypass authentication

### Accessibility requirements

- No accessibility impact

## Package Boundaries

A new `@repo/edge` package provides edge-compatible utilities and helpers. It must not depend on any Node.js-specific packages — only edge-compatible APIs (Web APIs, TextEncoder, crypto, etc.). The `@repo/auth` package must export edge-compatible session validation functions.

---

## Architecture Constraints

- Edge middleware must run in Next.js Edge Runtime — no Node.js APIs, no `fs`, no `net`
- Cloudflare Workers must not import any package that requires Node.js APIs
- Edge functions pass data to origin via request headers (e.g., `x-geo-country`, `x-ab-variant`)
- Edge middleware must complete in under 5ms — minimize computation, cache aggressively
- Edge functions must have their own deployment pipeline separate from the main application
- Edge cache (Cloudflare KV or Cache API) must be used for configuration data

## Notes

Edge functions are a powerful optimization but limited in capabilities. The most impactful edge functions for this platform are: session validation (already at edge), geolocation-based content delivery, and A/B testing. Do not attempt to move database-dependent operations to the edge — the latency saving is outweighed by the complexity of managing distributed state. The `@repo/edge` package must be carefully maintained to avoid importing Node.js APIs.
