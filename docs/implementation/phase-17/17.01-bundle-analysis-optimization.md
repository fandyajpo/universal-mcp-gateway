## Goal

Analyze the JavaScript bundle sizes across all four Next.js applications, identify optimization opportunities, and implement code splitting, tree shaking, dynamic imports, and dependency optimization to reduce bundle sizes and improve load performance.

---

## Business Context

Large JavaScript bundles directly impact user experience: slower page loads, longer time-to-interactive, and higher bounce rates. As the platform has grown across 16 phases, dependency bloat and inefficient imports have accumulated. Bundle optimization is the highest-impact frontend performance improvement — reducing bundle size by 50% can cut load times in half.

---

## Technical Context

Next.js 15 with Turbopack provides built-in bundle analysis tools. The analysis uses `@next/bundle-analyzer` to generate interactive treemaps of bundle contents. Each app (web, admin, docs, landing) is analyzed separately. The optimization phase focuses on: code splitting by route, dynamic imports for heavy components, tree shaking unused exports, replacing large dependencies with lighter alternatives, and extracting common code into shared chunks.

## Requirements

### Functional requirements

- Run bundle analysis for all four apps (web, admin, docs, landing)
- Generate interactive bundle treemaps for analysis
- Identify largest dependencies and their contribution to bundle size
- Identify duplicate dependencies across packages
- Identify components that can be dynamically imported (loaded on demand)
- Identify opportunities for code splitting by route
- Identify unused exports (opportunities for tree shaking)
- Implement dynamic imports for: heavy UI components, chart libraries, markdown renderers, code editors, PDF viewers, large third-party dependencies
- Implement route-level code splitting using Next.js dynamic imports
- Remove or replace heavy dependencies with lighter alternatives
- Extract common code into shared chunks via Turborepo configuration
- Set up bundle size budgets and enforce in CI

### Non-functional requirements

- Critical path bundle size must be < 200KB per route
- Dynamic imports must use loading states (skeletons, spinners)
- Code splitting must not break route transitions (preload critical chunks)
- Bundle analysis must be reproducible (same results across runs)
- Bundle budgets must fail CI if exceeded

### Performance requirements

- Reduce web app initial JS bundle from baseline by 40%+
- Reduce admin app initial JS bundle from baseline by 30%+
- Time-to-Interactive improvement: 30%+
- First Contentful Paint improvement: 20%+

### Security requirements

- Dynamic imports must not introduce XSS vectors (lazy-loaded components must be trusted)
- Bundle analysis reports must not be publicly accessible

### Accessibility requirements

- Dynamic import loading states must be announced to screen readers
- Skeleton loaders must use `aria-hidden` appropriately

## Architecture Constraints

- Dynamic imports must use Next.js `next/dynamic` with `ssr: false` for client-only components
- Route-level code splitting is automatic in Next.js App Router — ensure all routes are properly separated
- Shared chunks are managed by Turborepo's build cache — configure properly to avoid duplicate chunks
- Bundle budgets are enforced in CI using `@next/bundle-analyzer` and custom threshold checks
- No optimization should break existing imports or types

## Notes

Use `@next/bundle-analyzer` with `ANALYZE=true` environment variable to generate treemaps. Analyze both production and development builds — production builds reveal the true bundle size with minification and tree shaking. Common heavy dependencies to investigate: moment.js (replace with date-fns), lodash (use per-method imports), chart libraries (lazy load), markdown renderers (lazy load), syntax highlighters (lazy load). Always measure before and after — the goal is 40%+ reduction in initial JS bundle.
