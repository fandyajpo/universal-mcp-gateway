## Goal

Review and optimize the platform's multi-level caching strategy across in-memory, Redis, and CDN layers. Implement cache stampede prevention, optimize TTLs, improve cache invalidation, and ensure tenant-isolated caching.

---

## Business Context

Caching is the most effective performance optimization after eliminating N+1 queries. A multi-level cache strategy reduces database load, improves response times, and provides resilience against traffic spikes. However, incorrect caching can serve stale data, cause cache stampedes (thundering herd), or leak data between tenants. This step reviews and optimizes all caching to ensure it is correct, efficient, and safe.

---

## Technical Context

The platform has three cache layers: (1) in-memory cache (request-scoped and process-scoped via `@repo/cache`), (2) Redis cache (distributed via Upstash, used by `@repo/cache`), (3) CDN cache (Cloudflare edge, configured in step 17.03). Each layer has different characteristics: speed, capacity, and consistency guarantees. The review covers: cache key design, TTL policies, invalidation strategies, cache stampede prevention, and tenant isolation.

## Requirements

### Functional requirements

- Audit all cache usages across the platform: what is cached, where, for how long
- Review cache key naming convention adherence: `tenant:{tenantId}:{domain}:{entity}:{id}:{field}`
- Optimize TTL values: data freshness requirements vs cache hit ratio
- Implement cache stampede prevention: probabilistic early expiration (X-PRE, also known as "jitter") or mutex-based regeneration
- Add cache miss metrics: track which keys are requested but not found (miss ratio)
- Implement write-through cache for high-read, low-write data (pricing plans, feature flags)
- Implement cache-aside for user-specific data (session, preferences)
- Add cache warming for critical data on application startup
- Review and fix cache invalidation: ensure mutations invalidate related caches
- Add cache health monitoring: hit ratio, latency, memory usage, eviction rate

### Non-functional requirements

- Cache hit ratio must exceed 90% for hot data
- Cache invalidation propagation must complete within 5 seconds
- Cache stampede prevention must add less than 5ms overhead per cache check

### Performance requirements

- In-memory cache: < 1µs read
- Redis cache: < 5ms read
- Cache hit ratio: > 90% for hot data
- Cache miss penalty (regenerate + populate): < 200ms

### Security requirements

- Cache keys must not contain sensitive data (PII, tokens, secrets)
- Cache values must not contain sensitive data — only store pre-rendered safe data
- Tenant isolation verified: keys must always include tenantId
- Cache purging must not allow cross-tenant data leaks

### Accessibility requirements

- No accessibility impact

## Architecture Constraints

- Cache stampede prevention uses probabilistic early expiration: when a key is within 10% of its TTL expiration, randomly expire it early with probability proportional to the load (prevents all requests from detecting expiration simultaneously)
- In-memory cache TTL must be shorter than Redis TTL (e.g., in-memory 30s, Redis 300s)
- Cache keys must strictly follow the convention `tenant:{tenantId}:{domain}:{entity}:{id}:{field}`
- Cache invalidation must cascade: updating a parent entity invalidates children's caches
- Cache warming loads critical data on application startup — do not block startup

## Notes

Cache stampede (also called "thundering herd") is a critical concern for high-traffic endpoints. When a popular cache key expires and hundreds of requests simultaneously try to regenerate it, the database is overwhelmed. Probabilistic early expiration is the most practical solution for this platform — it randomly refreshes cache entries before they expire, smoothing out the regeneration load. The Redis `SET NX` (set if not exists) with TTL is a simpler alternative for single-key stampedes but does not work well for hot keys that regenerate slowly.
