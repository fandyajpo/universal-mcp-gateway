## Goal

Establish a comprehensive performance testing capability for the platform. Implement Lighthouse CI audits, k6 load testing scenarios, performance budgets, and regression detection to ensure performance is measured, maintained, and continuously improved.

---

## Business Context

Performance optimization is only as good as the measurement system. Without performance testing, optimizations cannot be validated and regressions go undetected. A comprehensive performance testing suite provides: (1) Lighthouse scores for frontend performance, (2) load testing for backend capacity, (3) performance budgets enforced in CI, (4) regression detection on every PR, (5) performance dashboards for ongoing monitoring.

---

## Technical Context

The performance testing suite uses: Lighthouse CI for frontend audits (FCP, LCP, TTI, TBT, CLS), k6 for backend load testing (RPS, latency, error rate, saturation point), and custom performance budget checks in CI. Tests run in the CI pipeline on every PR and on a schedule against the production environment.

## Requirements

### Functional requirements

- Lighthouse CI integration: run Lighthouse audits for key pages (landing, login, workspace, chat, admin dashboard, marketplace)
- Lighthouse performance budgets: FCP < 1.5s, LCP < 2.5s, TTI < 3.5s, TBT < 200ms, CLS < 0.1
- Lighthouse scores: performance > 95, accessibility > 95, best practices > 95, SEO > 95
- k6 load testing scenarios: chat API streaming, RAG query API, MCP tool execution, marketplace listing API, workspace CRUD API, authentication login flow
- k6 performance targets: p95 API latency < 200ms, error rate < 0.1%, throughput baseline (100 RPS for chat, 500 RPS for marketplace)
- Performance budget enforcement in CI: fail PR if any budget is exceeded
- Regression detection: compare Lighthouse scores and k6 results against baseline
- Performance dashboard: historical view of Lighthouse scores, API latencies, throughput
- Alerting: notify team when performance degrades beyond thresholds

### Non-functional requirements

- Lighthouse CI must run against a production-like build (not dev)
- k6 tests must run against a staging environment (not production initially)
- Performance budget changes must be reviewed and approved (not silently updated)
- Historical data must be retained for 90 days

### Performance requirements

- Lighthouse CI run: < 5 minutes per app
- k6 load test run: < 10 minutes per scenario
- Performance budget check: < 1 minute

### Security requirements

- k6 tests must use test API keys (not production keys)
- Performance data must not expose sensitive information
- Load testing must not impact production users
- Load testing should be rate-limited to prevent accidental DoS

### Accessibility requirements

- Lighthouse accessibility scores must be included in performance budgets
- Accessibility regression must fail CI

## Package Boundaries

---

## Architecture Constraints

- Lighthouse CI must use the `lhci` CLI with `--collect` against a production build served via `next start`
- k6 tests must be parameterized by environment (staging URL, API keys) via environment variables
- Performance budgets are stored in `performance-budgets.json` and version-controlled
- Baselines are stored in a cloud bucket (R2) keyed by commit SHA and test name
- Regression detection compares the current run against the baseline from the last successful main branch run
- k6 test scenarios: smoke (1 iteration, verify no errors), load (target throughput, measure latency), stress (ramp up to find breaking point), soak (sustained load over 30 minutes)

## Notes

k6 is a powerful load testing tool — invest time in writing realistic test scripts that mimic actual user behavior. The most important scenarios are chat streaming (the primary user interaction) and marketplace listing queries (the most common API call). Performance budgets should be aggressive but achievable — start with the current measured values as baseline and tighten over time. The soak test (30 minutes at sustained load) is critical for detecting memory leaks and connection pool exhaustion.
