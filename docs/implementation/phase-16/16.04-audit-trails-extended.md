## Goal

Extend the platform's audit logging capabilities for enterprise compliance requirements. Implement immutable log storage with cryptographic verification, event streaming to external SIEM systems, compliance-focused query APIs, and comprehensive coverage of all enterprise-relevant events.

---

## Business Context

Enterprise customers require comprehensive audit trails for compliance, security investigations, and operational visibility. The basic audit log from Phase 13 (step 13.05) covers admin-viewable events. Extended audit trails add: cryptographic immutability guarantees, real-time event streaming to SIEM tools (Splunk, Datadog, Sumo Logic), fine-grained event coverage across all platform domains, and a query API designed for compliance workflows.

---

## Technical Context

The extended audit system builds on the existing `audit_logs` collection and `AuditLogRepository` (step 00.09). New features include: (1) cryptographic hash chaining — each log entry includes the hash of the previous entry, forming a tamper-evident chain, (2) event streaming — audit events are published to an event bus (Inngest) for SIEM forwarding, (3) expanded event coverage — all platform mutations across all phases generate audit events, (4) compliance API — query endpoint with filtering, export, and integrity verification.

## Requirements

### Functional requirements

- Cryptographic hash chain: each audit entry contains `previousHash` (SHA-256 of previous entry), current `hash` (SHA-256 of current entry data + previous hash), and `chainTip` (current chain tip for fast verification)
- Hash chain initialization: seed the first entry with a genesis hash
- Hash verification: verify the entire chain integrity on demand, or verify a range of entries
- Expanded event coverage: add audit events for all platform operations across auth, workspace, storage, PDF, embedding, RAG, AI gateway, MCP gateway, connectors, chat, billing, marketplace, admin, enterprise features
- Event streaming: publish audit events to Inngest event bus for consumption by SIEM forwarder
- SIEM integrations: build event forwarders for Splunk HEC, Datadog, and generic webhook
- Compliance API: query audit logs with compliance-specific filters (user ID, date range, event types, workspace), export in JSON/CSV formats, verify chain integrity for a range
- Event retention: configurable retention per event type (some events retained longer for compliance)
- Real-time alerting: configurable rules that trigger alerts on specific event patterns (e.g., 5 failed logins in 5 minutes)

### Non-functional requirements

- Hash computation must not add more than 5ms to audit log writes
- Chain verification of 1M entries must complete in under 60 seconds
- Event streaming latency must be under 5 seconds (event written → SIEM receives)
- Compliance API queries must return results in under 10 seconds for 3-month range

### Performance requirements

- Audit log write with hash: < 10ms
- Chain tip query: < 5ms
- Compliance query (90 days): < 10s
- Chain integrity verification (1M entries): < 60s

### Security requirements

- Hash chain must be stored separately from the log entries (dual-write with different credentials)
- Chain verification must detect any modification, deletion, or insertion
- SIEM forwarding must use TLS and API key authentication
- Compliance API requires `admin:audit` permission
- Alert rules must not reveal sensitive information in notifications

### Accessibility requirements

- Compliance API responses must follow standard formats
- Verification reports must be accessible

## Package Boundaries

It depends on `@repo/database` (audit log repository), `@repo/crypto` (SHA-256 hashing), `@repo/cache` (chain tip caching), `@repo/logger`, and `@repo/inngest` (event streaming, retention cron). The existing `AuditLogRepository` is modified to use the hash chain.

---

## Architecture Constraints

- Hash chain is implemented as a separate collection `audit_hash_chain` with `{ entryId, previousHash, hash, timestamp, chainTip }` — the chain is stored independently from the log entries
- Each log entry's hash = `SHA-256(entryId + tenantId + action + timestamp + JSON(details) + previousHash)`
- Chain tip is cached in Redis for fast access — on verification, the in-database chain tip is compared with Redis
- SIEM forwarding is asynchronous via Inngest — the event publisher writes to Inngest, which forwards to configured SIEM destinations
- Compliance API queries use the audit log collection (not the hash chain) — the hash chain is only for integrity verification

## Notes

The cryptographic hash chain is the most important feature for compliance. It provides tamper-evident audit logs that can be presented to auditors. The chain is stored separately from the log entries to prevent an attacker who compromises the database from modifying both. Consider implementing periodic chain integrity checks (Inngest cron) that verify the chain and alert on any inconsistencies. SIEM forwarding should be opt-in per workspace — enterprise workspaces can configure their own SIEM destinations.
