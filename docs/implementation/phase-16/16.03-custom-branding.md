## Goal

Implement custom branding capabilities for enterprise workspaces. Workspace admins can customize the platform's appearance with their organization's logo, colors, custom domain, email templates, and terms of service.

---

## Business Context

Enterprise customers need the platform to reflect their brand identity. Custom branding includes the workspace logo and favicon, color scheme (primary, secondary, accent colors), custom domain (workspace.app.com instead of app.platform.com), branded email notifications, and custom terms of service. Branding builds trust with end users and creates a seamless experience.

---

## Technical Context

Branding configuration is stored in the workspace settings document. The branding engine applies customizations at multiple levels: CSS custom properties for colors, Next.js middleware for domain detection, email service for template customization. The branding is workspace-scoped — each workspace has independent branding.

## Requirements

### Functional requirements

- Custom logo: upload logo image (PNG/SVG, max 2MB), display in app header and login page
- Custom favicon: upload favicon (ICO/PNG, max 1MB), apply to browser tab
- Custom colors: configure primary, secondary, accent colors via color picker or hex input
- Dark mode override: force light mode, force dark mode, or follow system preference
- Custom domain: configure custom domain with CNAME verification and SSL certificate provisioning
- Custom email sender: configure custom "from" name and email address for notifications
- Custom email templates: upload custom email template header/footer HTML
- Custom terms of service: upload custom ToS document (markdown or HTML), link in sign-up flow
- Custom privacy policy: upload custom privacy policy document
- Branding preview: live preview of branding changes before saving
- Branding reset: reset all branding to platform defaults

### Non-functional requirements

- Branding must apply globally across all pages (not just workspace-scoped pages)
- Custom domain must support HTTPS via automatic Let's Encrypt certificate provisioning
- Branding changes must take effect immediately (no deployment required)
- Logo and favicon must be served via CDN with caching

### Performance requirements

- Branding CSS injection: < 50ms (inline, no blocking render)
- Logo load: < 2s (CDN-cached)
- Color changes: immediate via CSS custom properties
- Custom domain DNS verification: < 60s

### Security requirements

- Logo and favicon uploads must be validated as actual images (magic byte check)
- Custom domain must verify domain ownership via DNS TXT record
- Custom email sender must verify domain ownership (SPF/DKIM)
- Custom HTML in email templates must be sanitized (no script tags)
- Custom ToS must be sanitized (no active content)

### Accessibility requirements

- Color picker must support keyboard input (hex values)
- Logo must have configurable alt text
- Custom color combinations must be validated for WCAG 2.1 AA contrast

## Package Boundaries

A new `@repo/branding` package provides the branding engine. It depends on `@repo/database` (branding repository), `@repo/storage` (logo/favicon storage), `@repo/cache` (branding config cache), and `@repo/ui` (ColorPicker, FileUpload components). The web app uses a `BrandingProvider` client component to apply branding dynamically.

---

## Architecture Constraints

- Branding CSS custom properties are injected via a `<style>` tag in the document `<head>` — no runtime CSS-in-JS
- Custom domain resolution happens in Next.js middleware — the middleware looks up the domain in Redis/mapped collection and resolves to the workspace
- Logo and favicon are stored at `r2://branding/{workspaceId}/{type}.{ext}` with CDN URL
- DNS verification for custom domains uses a TXT record with a verification token: `umg-verify={token}`
- Email templates are stored as Handlebars/Mustache templates with configurable header, footer, and CSS

## Notes

Custom domain configuration is the most complex part of branding. It requires the workspace admin to add a CNAME record pointing to the platform's CDN, plus a TXT verification record. The middleware must handle the domain → workspace lookup efficiently — use a Redis hash with domain as key and workspace ID as value, populated from the database on branding updates. Color contrast validation is important for accessibility — prevent admins from choosing color combinations that fail WCAG AA contrast ratios.
