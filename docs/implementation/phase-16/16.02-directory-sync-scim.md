## Goal

Implement SCIM (System for Cross-domain Identity Management) protocol for directory synchronization. Support both push (IdP → platform) and pull (platform → IdP) sync modes for users and groups, with delta sync and webhook-based provisioning.

---

## Business Context

SCIM is the standard protocol for synchronizing user identities and group memberships between an organization's identity provider and downstream applications. Enterprise customers need SCIM to automate user provisioning and de-provisioning, manage group memberships, and enforce access control policies. Without SCIM, enterprise IT admins must manually manage user accounts, which is error-prone and does not scale.

---

## Technical Context

SCIM 2.0 is implemented as a REST API with standard endpoints for Users and Groups resources. The platform acts as a SCIM Service Provider, exposing endpoints that IdPs (Okta, Azure AD) call to push user and group data. The implementation handles: user creation, update, deactivation (soft-delete), group creation, membership updates, and delta synchronization.

The SCIM endpoints are protected by a SCIM authentication token (not the regular session auth) that is configured during IdP setup. SCIM events trigger user provisioning in the platform's database and optionally send invitations to new users.

## Requirements

### Functional requirements

- SCIM 2.0 User endpoints: `POST /Users` (create), `GET /Users` (list/search), `GET /Users/{id}` (get), `PUT /Users/{id}` (update), `PATCH /Users/{id}` (partial update), `DELETE /Users/{id}` (deactivate)
- SCIM 2.0 Group endpoints: `POST /Groups` (create), `GET /Groups` (list/search), `GET /Groups/{id}` (get), `PUT /Groups/{id}` (update), `PATCH /Groups/{id}` (partial update), `DELETE /Groups/{id}` (delete)
- Support SCIM filter queries: `filter=userName eq "user@example.com"`, `filter=active eq true`
- Support SCIM attribute mapping: map SCIM standard attributes to platform user fields (userName → email, name.givenName → firstName, name.familyName → lastName, emails[primary] → email, groups → roles)
- User provisioning: create platform user account on SCIM user creation
- User deactivation: set user as inactive (suspended) on SCIM user deletion
- Group provisioning: create platform groups from SCIM groups
- Group membership sync: add/remove users from groups based on SCIM group memberships
- Delta sync: process only changed users/groups since last sync
- Webhook-based sync: subscribe to IdP SCIM events for real-time provisioning
- Sync logging: record all SCIM operations with status for troubleshooting
- Admin sync dashboard: view sync status, recent operations, and error logs

### Non-functional requirements

- SCIM API must respond within 2 seconds for single resource operations
- Bulk operations (list/search) must support pagination with configurable page size
- SCIM authentication token must be configurable per workspace
- Sync operations must be idempotent — processing the same SCIM request multiple times produces the same result

### Performance requirements

- Single user/group operation: < 2s
- List/search with 10k users: < 5s (paginated, 100 per page)
- Full directory sync (10k users): < 30 minutes (batched)

### Security requirements

- SCIM endpoints must use HTTP Basic Auth or Bearer token authentication (separate from user auth)
- SCIM token must have `scim:write` permission scope
- SCIM operations must be logged to audit trail
- User deactivation via SCIM must not delete user data immediately (soft-delete with retention period)
- SCIM token must be rotatable without downtime

### Accessibility requirements

- Admin sync dashboard must have accessible data tables

## Package Boundaries

SCIM is part of `@repo/auth`. It depends on `@repo/database` (user and group repositories), `@repo/validation` (SCIM schemas), `@repo/logger` (sync logging), and `@repo/config` (SCIM configuration). The SCIM API routes in `apps/web` are public (token-authenticated) — they bypass the normal session auth.

---

## Architecture Constraints

- SCIM endpoints must follow RFC 7643 and RFC 7644 specifications exactly
- SCIM authentication uses a pre-shared bearer token stored in the workspace's SSO configuration
- All SCIM operations are logged to a `scim_events` collection for audit
- User deactivation via SCIM sets `active: false` and `suspendedAt: now()` — data is preserved
- Group membership changes must be reflected in the platform's RBAC system (users inherit permissions from SCIM groups mapped to platform roles)
- SCIM pagination uses `startIndex` and `count` parameters per RFC 7644

## Notes

SCIM implementation requires strict adherence to the RFC specifications. Test against Okta and Azure AD SCIM implementations specifically, as they are the most widely used and have known quirks. The filter parser is the most complex part — SCIM filters support attribute expressions, logical operators, and parenthesized groups. Consider using a PEG parser library if the filter grammar becomes too complex to implement manually.
