## Goal

Implement Single Sign-On (SSO) via SAML 2.0 protocol. Support both IdP-initiated and SP-initiated SSO flows, multiple identity providers (Okta, Azure AD, Google Workspace, OneLogin), Just-In-Time user provisioning, and IdP configuration management through an admin UI.

---

## Business Context

SSO is the most requested enterprise feature. Organizations require their employees to authenticate through the corporate identity provider (Okta, Azure AD, Google Workspace). SSO simplifies user management, enforces corporate security policies, and provides a seamless login experience. Without SSO, enterprise adoption is blocked.

---

## Technical Context

SAML 2.0 is the standard protocol for enterprise SSO. The implementation uses a SAML library (e.g., `samlify` or `@boxyhq/saml20`) to handle SAML protocol details: XML parsing, signature verification, assertion processing. Better Auth (step 02.01) provides the auth provider abstraction — the SAML integration plugs into this as a custom provider.

The flow: (1) Admin configures IdP metadata in workspace settings, (2) User clicks "Sign in with SSO" on login page, (3) Platform generates SAML AuthnRequest and redirects to IdP, (4) IdP authenticates user and POSTs SAML Response back, (5) Platform validates signature, extracts user attributes, and creates/updates user session.

## Requirements

### Functional requirements

- Support SP-initiated SSO: user clicks "Sign in with SSO", enters email domain, gets redirected to IdP
- Support IdP-initiated SSO: user starts from IdP app dashboard and lands in the platform
- Support multiple identity providers: Okta, Azure AD, Google Workspace, OneLogin, generic SAML 2.0 IdP
- Admin configuration UI: upload IdP metadata XML, configure attribute mapping (email, name, groups), set SSO domain
- SAML assertion validation: verify signature, validate audience, check NotBefore/NotOnOrAfter conditions
- Just-In-Time (JIT) provisioning: create user account on first SSO login if not exists
- Attribute mapping: map IdP attributes to platform user fields (email, firstName, lastName, groups/roles)
- Session creation: after successful SAML assertion, create platform session
- IdP logout: support Single Logout (SLO) protocol
- Fallback to password login: users can still use email/password if SSO fails
- Domain enforcement: restrict SSO to specific email domains
- Certificate rotation: support IdP certificate rotation without downtime

### Non-functional requirements

- SAML response processing must complete in under 3 seconds
- IdP configuration must support multiple certificates for rotation
- Attribute mapping must be configurable per workspace
- SSO login flow must complete within 5 seconds total

### Performance requirements

- SAML assertion validation: < 1s
- JIT provisioning: < 2s (including user creation)
- Full SSO login flow: < 5s

### Security requirements

- SAML assertions must use signed responses (never accept unsigned assertions)
- Assertion signature must be verified using IdP's public certificate
- Audience restriction must be enforced (assertion is valid only for specific platform ACS URL)
- NotBefore/NotOnOrAfter time conditions must be validated with clock skew tolerance (5 minutes)
- Replay attack prevention: store assertion ID for 5 minutes to detect duplicates
- IdP metadata must be validated before storage (well-formed XML, valid certificate)
- SSO bypass must not be possible (cannot skip IdP authentication)

### Accessibility requirements

- SSO login button must have clear label
- IdP configuration form must have accessible field labels

## Package Boundaries

SSO is part of `@repo/auth`. It depends on a SAML library (`samlify` or `@boxyhq/saml20`), `@repo/database` (IdP config repository), `@repo/config` (SP entity ID, ACS URL configuration), `@repo/cache` (assertion ID deduplication), and `@repo/logger`. The admin UI depends on `@repo/auth` and `@repo/ui`.

---

## Architecture Constraints

- SP entity ID must be configurable via environment variable (default: platform URL)
- ACS URL must be a fixed endpoint: `/api/auth/saml/callback`
- IdP metadata XML must be parsed and stored as structured data (not raw XML)
- Certificate validation must support multiple active certificates for rotation
- Assertion ID must be stored in Redis with 5-minute TTL for replay detection
- JIT provisioning creates users with default role (member) — workspace admin assigns roles later

## Notes

SAML is notoriously complex and error-prone. Test against multiple identity providers (Okta, Azure AD, Google Workspace, OneLogin) to ensure compatibility. The clock skew tolerance (5 minutes) prevents issues with time-synchronized IdPs but must not be too large to avoid replay attacks. Store the IdP metadata validation rules in a well-documented format. Consider implementing a SAML debug mode that logs assertion details (without sensitive data) for troubleshooting. This is the most technically challenging step in Phase 16 — allocate extra time for IdP compatibility testing.
