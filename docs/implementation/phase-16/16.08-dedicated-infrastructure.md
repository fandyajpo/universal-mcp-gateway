## Goal

Implement dedicated infrastructure provisioning for enterprise tenants. Support isolated compute, database, storage, and networking resources for workspaces that require maximum isolation, performance, and compliance.

---

## Business Context

Top-tier enterprise customers require dedicated infrastructure due to compliance requirements, performance guarantees, or data sovereignty needs. Dedicated infrastructure provides a completely isolated deployment of the platform for a single tenant. This includes dedicated compute instances, isolated database clusters, dedicated storage buckets, and separate networking. This is the highest tier of service and commands premium pricing.

---

## Technical Context

Dedicated infrastructure provisioning is a complex DevOps automation task. The implementation focuses on the provisioning orchestration and management layer — the actual infrastructure resources are provisioned using Infrastructure as Code (Terraform/Pulumi) templates. The platform's dedicated infrastructure service: (1) accepts provisioning requests from the admin UI, (2) triggers IaC pipeline to provision resources, (3) monitors provisioning progress, (4) configures the platform to use the dedicated resources, (5) manages decommissioning.

Given the scope, this step focuses on the automation layer and assumes cloud infrastructure APIs are available. The dedicated infrastructure includes: isolated Next.js compute (Kubernetes namespace or separate ECS service), dedicated MongoDB Atlas cluster, dedicated Redis instance, dedicated R2 bucket path, and isolated networking (VPC, security groups).

## Requirements

### Functional requirements

- Dedicated infrastructure request: admin initiates provisioning for an enterprise workspace
- Infrastructure template selection: choose provision template (medium, large, x-large) with predefined resource specs
- Provisioning orchestration: trigger IaC pipeline, track provisioning status (provisioning, ready, failed, decommissioning)
- Resource configuration: after provisioning, configure platform services to use dedicated resources (separate database URL, Redis URL, R2 bucket)
- Database provisioning: provision dedicated MongoDB Atlas cluster (M20+, dedicated, multi-region)
- Redis provisioning: provision dedicated Upstash Redis instance
- Storage provisioning: configure dedicated R2 bucket with isolated paths
- Compute isolation: configure dedicated Kubernetes namespace or ECS service for workspace traffic
- Network isolation: configure VPC, security groups, and network policies
- Monitoring integration: connect dedicated resources to platform monitoring and SLA tracking
- Decommissioning: trigger decommission when workspace downgrades or leaves — data migration, backup, resource teardown
- Provisioning dashboard: view provisioning status, resource details, connection status, and health of dedicated infrastructure

### Non-functional requirements

- Provisioning must be fully asynchronous — admin UI shows progress, does not block
- Resource provisioning must complete within 60 minutes (typical cloud provisioning time)
- Decommissioning must include data export/backup before resource teardown
- Configuration changes must be applied without downtime (rolling update)

### Performance requirements

- Provisioning trigger response: < 2s (IaC pipeline is queued asynchronously)
- Provisioning status check: < 500ms
- Configuration switch to dedicated resources: < 5 minutes (zero-downtime)

### Security requirements

- Dedicated resources must be fully isolated at the network level (no cross-tenant network access)
- Database credentials for dedicated clusters must be generated per workspace
- Encryption keys must be unique per dedicated deployment
- Decommissioning must include secure data destruction (secure wipe of storage)
- Access to provisioning APIs requires `admin:infrastructure` permission

### Accessibility requirements

- Provisioning status indicators must include text and progress bars

## Package Boundaries

A new `@repo/infrastructure` package provides dedicated infrastructure management. It depends on `@repo/database` (provisioning repository), `@repo/config` (cloud API credentials), `@repo/logger`, and `@repo/inngest` (provisioning monitors). Cloud provider SDKs (MongoDB Atlas API, Upstash API, AWS SDK for R2) are dependencies.

---

## Architecture Constraints

- All infrastructure provisioning goes through the IaC pipeline — no manual resource creation
- Provisioning status transitions: `requested → provisioning → ready → active → decommissioning → decommissioned`
- Failed provisioning transitions to `failed` with error details for debugging
- Resource configuration is stored in the workspace settings document and referenced by the config package
- Decommissioning includes: backup workspace data → notify workspace admins → decommission database → decommission Redis → decommission storage → decommission compute → mark as decommissioned
- Configuration switch to dedicated resources uses feature flags and environment variable overrides

## Notes

Dedicated infrastructure is the most operationally complex feature. This implementation provides the automation layer — actual cloud infrastructure templates must be maintained separately by the DevOps team. The provisioners use cloud provider APIs (MongoDB Atlas API, Upstash API, AWS SDK) to provision resources. In a development environment, provisioners should use mock implementations to avoid actual cloud resource creation. Consider implementing a "dry run" mode that validates the provisioning request without creating resources.
