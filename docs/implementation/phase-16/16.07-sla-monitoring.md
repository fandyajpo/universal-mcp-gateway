## Goal

Implement SLA monitoring that tracks platform uptime and performance against committed service level agreements. Build external health probes, uptime calculation with exclusion windows, SLA breach detection and notification, and a customer-facing SLA dashboard.

---

## Business Context

Enterprise contracts include Service Level Agreements (SLAs) that define committed uptime percentages (e.g., 99.9% uptime monthly), response time guarantees, and credits for breaches. The SLA monitoring system must independently verify platform availability, calculate uptime accurately (excluding scheduled maintenance), detect breaches promptly, and provide evidence for SLA credit calculations.

---

## Technical Context

The SLA monitoring system extends the health check infrastructure from step 13.06. External monitoring probes (running outside the platform's infrastructure) check platform endpoints from multiple geographic locations. Probes record availability and response time. Uptime is calculated per month with configurable exclusion windows (scheduled maintenance). Breach detection triggers notifications and creates SLA incident records.

## Requirements

### Functional requirements

- External monitoring probes: run from separate infrastructure (Cloudflare Workers or simple cron) to check platform endpoints — `/api/health`, `/api/auth/health`, login page availability, API response time
- Multi-region probes: check from at least 3 geographic regions (US, EU, APAC)
- Probe results: HTTP status code, response time (ms), timestamp, region, probe ID
- Uptime calculation: monthly uptime percentage = (total checks - failed checks) / total checks * 100, with configurable exclusion windows for scheduled maintenance
- Exclusion windows: schedule maintenance windows in advance, exclude from uptime calculations
- SLA definition: per-workspace SLA targets (uptime %, max response time) configured in workspace settings
- Breach detection: compare calculated uptime against SLA target, trigger breach alert when threshold is crossed
- Breach notifications: email notification to workspace admins, in-app notification, webhook to PagerDuty or similar
- SLA dashboard: current month uptime, historical uptime (12 months), breach history, exclusion window management
- SLA report: generate monthly SLA report with uptime calculations, breach details, and credit calculations
- SLA credit calculator: compute service credits based on breach severity and contract terms

### Non-functional requirements

- External probes must be independent of platform infrastructure
- Probe frequency: every 60 seconds from each region
- Uptime calculations must be accurate to 3 decimal places (99.9%)
- Breach detection must trigger within 5 minutes of threshold crossing

### Performance requirements

- Probe execution: < 5s per region (including network latency)
- Uptime calculation for a month: < 10s
- SLA dashboard load: < 3s

### Security requirements

- External probes must use API key authentication (not session auth)
- Probe endpoints must be rate-limited to prevent abuse
- SLA data must be workspace-scoped — workspaces see only their own SLA
- Exclusion window management requires `admin:workspace` permission

### Accessibility requirements

- SLA dashboard must have accessible data tables and charts
- Uptime percentage must be displayed with clear text

## Package Boundaries

A new `@repo/sla` package provides SLA monitoring functionality. It depends on `@repo/database` (SLA repositories), `@repo/health` (probe infrastructure), `@repo/logger`, `@repo/inngest` (scheduled checks), and `@repo/notifications` (breach alerts). External probe scripts are separate from the main package.

---

## Architecture Constraints

- External probes must not depend on the platform's infrastructure — deploy as separate Cloudflare Workers or simple Node.js scripts on a cron schedule
- Probe results are submitted via API: `POST /api/sla/probes/{probeId}/results` with API key auth
- Uptime is calculated per calendar month, reported with 3 decimal places
- Exclusion windows are defined in advance with start/end time and reason — only pre-approved windows are excluded
- Breach thresholds are evaluated daily — if the projected end-of-month uptime would fall below the SLA target, a breach warning is issued
- SLA reports are generated on the 1st of each month for the previous month

## Notes

External probes are critical for credible SLA monitoring — the platform cannot verify its own uptime. Deploy probes as Cloudflare Workers in US, EU, and APAC regions. The probe script should be minimal: HTTP GET to the health endpoint, record status and timing, POST result back. The probe API key should be different from platform API keys and restricted to the probe endpoint only. SLA credit calculations are contract-specific — implement a configurable credit formula that can be customized per enterprise contract.
