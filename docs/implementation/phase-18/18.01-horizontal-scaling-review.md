## Goal

Conduct a comprehensive horizontal scaling review of the entire platform. Identify architectural bottlenecks, verify statelessness of all services, analyze database scaling limits, review caching and queuing infrastructure, assess CI/CD pipeline scalability, and produce a detailed scaling report with prioritized recommendations.

---

## Business Context

The platform must handle 10x growth in users, workspaces, and API traffic. Horizontal scaling — adding more instances rather than vertically scaling a single instance — is the only cost-effective way to achieve this. Every component must be designed for horizontal scaling: stateless services, distributed caching, sharded databases, and partitioned queues. This review identifies the gaps that must be addressed before scaling to 10x.

---

## Technical Context

The review methodology: (1) analyze request flow for each major feature (chat, RAG, AI gateway, MCP gateway, marketplace, admin), (2) identify stateful components that prevent horizontal scaling, (3) review database query patterns for sharding readiness, (4) assess Redis usage for distributed caching correctness, (5) evaluate Inngest queue configuration for worker scalability, (6) review session management, (7) analyze file storage architecture for multi-region access.

## Requirements

### Functional requirements

- Audit every service for statefulness: identify any in-memory state, local file system usage, or sticky session requirements
- Review session management: ensure sessions are stored in Redis (not local memory)
- Review WebSocket connections: ensure they can be routed to any instance (not tied to specific instance)
- Review file uploads: ensure uploaded files are stored in R2 (not local disk)
- Review background job processing: ensure jobs are idempotent and can run on any worker
- Review database connection pooling: ensure connection pool configuration scales with instances
- Review caching: ensure cache invalidation works across instances
- Review rate limiting: ensure rate limits are enforced globally (not per-instance)
- Review logging: ensure logs are centralized (not per-instance files)
- Review CI/CD: ensure deployment pipeline supports blue-green or rolling deployment
- Produce scaling report with: current capacity estimate, bottleneck list, priority-ranked recommendations, estimated effort for each recommendation

### Non-functional requirements

- Review must be reproducible — document methodology and criteria
- Each finding must include: component, issue description, impact (users affected, severity), recommendation, estimated effort
- Recommendations must be prioritized: P0 (blocking), P1 (high), P2 (medium), P3 (low)

### Security requirements

- Statefulness findings that affect security (e.g., session data in memory) must be P0 priority
- Cross-instance cache invalidation must guarantee tenant isolation

## Architecture Constraints

- Services must be stateless: no in-memory session storage, no local file system, no instance-affinity
- Database scaling: MongoDB Atlas supports horizontal scaling via sharding — the review must confirm sharding readiness
- Cache distribution: Redis (Upstash) is the centralized cache — the review must verify no local caching of critical data
- Queue distribution: Inngest handles job distribution — the review must verify worker configuration supports scaling
- File storage: R2 is the centralized object store — the review must verify no local file storage

## Notes

The scaling review is primarily an analysis and documentation step. It sets the direction for the remaining steps in Phase 18. Key areas to focus on: (1) session storage — must be Redis, not local memory, (2) WebSocket connections — must use Redis pub/sub or a separate WebSocket broker for cross-instance communication, (3) database — MongoDB Atlas sharding readiness, (4) file uploads — direct-to-R2 uploads to avoid instance-boundary bottlenecks. The review should also assess the deployment architecture: can the platform run in multiple Kubernetes clusters across regions?
