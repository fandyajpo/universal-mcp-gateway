## Goal

Configure MongoDB Atlas read replica set for analytics and reporting queries. Implement read preference routing, query classification (read vs write), and replica-aware repository methods.

---

## Business Context

As query volume grows, the primary database node becomes a bottleneck, especially for read-heavy workloads like analytics, reporting, and marketplace browsing. Read replicas offload read queries from the primary, improving both read performance (more capacity) and write performance (primary has less contention). Read replicas also provide high availability and can serve as disaster recovery targets.

---

## Technical Context

MongoDB Atlas supports read replica sets with automated failover. The application must configure read preference per query: primary for writes and consistent reads, secondaryPreferred for analytics and batch jobs, nearest for low-latency reads. The repository layer is modified to accept read preference hints.

## Requirements

### Functional requirements

- Configure MongoDB Atlas replica set with primary + 2 secondaries (minimum)
- Implement read preference options in the connection manager: `primary`, `primaryPreferred`, `secondary`, `secondaryPreferred`, `nearest`
- Add read preference parameter to all repository query methods
- Classify queries by read preference: primary (user-facing queries requiring consistency), secondaryPreferred (analytics, reporting, background jobs), nearest (geo-distributed low-latency reads)
- Implement replica lag monitoring: track replication delay, alert if > 10 seconds
- Add replica-aware health checks for monitoring
- Configure secondary read routing for: marketplace listing queries, installation metrics queries, billing dashboard queries, audit log queries, compliance report queries, analytics and aggregation pipelines, batch processing jobs

### Non-functional requirements

- Read replicas must not serve stale data for user-facing queries (use `primary` or `primaryPreferred`)
- Replication lag must be < 2 seconds for user-facing secondary reads
- Secondary reads must not impact primary write performance

### Performance requirements

- Read replica query latency: < 50ms p95 (local region)
- Replication lag: < 2 seconds p95
- Primary write performance improvement: 40%+ reduction in write contention

### Security requirements

- Replica set communication must use TLS
- Read replicas in different regions must comply with data residency requirements

## Architecture Constraints

- User-facing read queries use `primaryPreferred` (read from primary if secondary is lagging)
- Analytics and reporting queries use `secondaryPreferred` (read from secondary if available)
- Background jobs use `secondary` (explicitly read from secondary, fail if no secondary available)
- Dashboard and admin pages use `secondaryPreferred`
- Real-time features (chat, streaming) must use `primary` for consistency
- Replica lag monitoring polls `rs.printSlaveReplicationInfo()` or `db.adminCommand({ replSetGetStatus: 1 })` every 10 seconds

## Notes

Read replicas are most impactful for the analytics-heavy pages: marketplace browsing, installation metrics, billing dashboards, and audit log queries. The most important consideration is replication lag — a secondary that is too far behind the primary will serve stale data. Set `secondaryPreferred` (not `secondary`) as the default for analytics queries so they fall back to primary if the secondary is lagging. MongoDB Atlas replica sets with auto-failover provide high availability — test the failover scenario to verify the application handles it gracefully.
