## Goal

Implement autoscaling for background queue workers based on queue depth and processing latency. Workers handle PDF processing, embedding generation, connector syncs, audit event streaming, and billing tasks — all must scale with demand while minimizing idle resource costs.

---

## Business Context

Background queue workers process asynchronous tasks: PDF ingestion, embedding generation, connector synchronization, audit event streaming, billing computations, and compliance report generation. During peak usage (e.g., many users uploading documents simultaneously), the queue depth can spike. Workers must scale up to handle the spike and scale down when demand subsides. Manual scaling is too slow for traffic spikes and wasteful during low-traffic periods.

---

## Technical Context

The platform uses Inngest for background job orchestration. Inngest manages the queue and dispatches jobs to workers. Autoscaling is implemented at the worker infrastructure level (Kubernetes HPA or ECS Service Auto Scaling) based on queue depth metrics from Inngest. The Inngest API provides queue depth and processing latency metrics that can be exported to CloudWatch or Prometheus for autoscaling decisions.

## Requirements

### Functional requirements

- Export Inngest queue metrics: queue depth per function, processing latency p50/p95/p99, success/failure rate, worker utilization
- Configure autoscaling metric: queue depth-based scaling (scale up when queue depth > 100 per worker) and processing latency-based scaling (scale up when p95 latency > 30s)
- Implement Kubernetes HPA (Horizontal Pod Autoscaler) for worker pods: min 2, max 20, scale up within 60 seconds, scale down within 5 minutes (cooldown period)
- Implement ECS Service Auto Scaling as alternative: same parameters
- Configure worker concurrency: max 10 concurrent jobs per worker instance
- Implement graceful shutdown: workers complete in-flight jobs before terminating
- Add worker health checks: liveness (is worker running), readiness (is worker accepting jobs)
- Add queue depth alerting: alert when queue depth > 1000 for more than 5 minutes
- Implement cost optimization: schedule scale-to-zero during known low-traffic periods (configurable)

### Non-functional requirements

- Scale-up latency: < 60 seconds from spike to new worker ready
- Scale-down cooldown: 5 minutes minimum between scale-down operations
- Worker must handle at least 10 concurrent jobs without degradation
- Autoscaling must not cause thrashing (rapid scale up/down)

### Performance requirements

- Queue processing throughput: linear scaling with worker count
- Job processing latency p95: < 30s during normal load, < 60s during peak
- Queue depth recovery: drain 1000-job backlog within 5 minutes at max worker count

### Security requirements

- Worker must have minimal IAM permissions (least privilege)
- Worker instances must not be publicly accessible
- Autoscaling must not expose internal metrics publicly

### Accessibility requirements

- No accessibility impact

## Architecture Constraints

- Worker pods must be stateless — no local state between jobs
- Inngest handles job distribution — workers only need to register with Inngest
- Autoscaling metrics must come from Inngest API (queue depth, processing latency)
- Scale-down cooldown prevents thrashing: once a scale-down decision is made, wait 5 minutes before another
- Graceful shutdown: workers stop accepting new jobs, complete in-flight jobs (max 30s wait), then terminate
- Worker concurrency is configured via Inngest client's `maxConcurrency` option

## Definition of Done

---

## Notes

Worker autoscaling is critical for cost-effective operations. The key metric is queue depth — it directly indicates demand. Processing latency is a secondary metric that catches cases where jobs are being picked up but taking too long. The scale-down cooldown (5 minutes) prevents thrashing when queue depth fluctuates at the threshold boundary. Consider implementing predictive scaling based on historical patterns (e.g., more document processing during business hours in US timezone) for even more efficient scaling.
