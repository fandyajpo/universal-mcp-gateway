## Goal

Create a detailed microservice extraction plan based on the horizontal scaling review findings. Identify bounded contexts, define service boundaries, design inter-service communication, and produce a phased migration roadmap from monolith to microservices.

---

## Business Context

The platform is currently deployed as a monorepo monolith (Next.js applications with co-located packages). While the monolith provides fast iteration and simple deployment, it will eventually limit independent scaling, team autonomy, and deployment velocity. The extraction plan defines how to decompose the monolith into independently deployable microservices based on Domain-Driven Design bounded contexts, without disrupting ongoing development.

---

## Technical Context

The architecture document (ARCHITECTURE.md section 14) already defines the long-term vision for microservice extraction. This step operationalizes that vision with a concrete, phased plan. Each bounded context (AI Gateway, MCP Gateway, RAG Engine, Auth, Billing, Marketplace, Chat, Admin) is evaluated for extraction readiness using criteria: independent scalability, team ownership, deployment frequency, data isolation requirements, and failure isolation needs.

## Requirements

### Functional requirements

- Identify bounded contexts using Domain-Driven Design: analyze the domain model, identify aggregates, define context boundaries based on language and data ownership
- Map each bounded context to a potential microservice: AI Gateway (`@repo/ai`), MCP Gateway (`@repo/mcp`), RAG Engine (`@repo/rag`), Auth Service (`@repo/auth`), Billing Service (`@repo/billing`), Marketplace Service (`@repo/marketplace`), Chat Service (`@repo/rag` + chat logic), Connector Registry (`@repo/connectors`), Sync Service (`@repo/connector-sdk` + sync engine)
- Design inter-service communication: synchronous (HTTP/REST, gRPC) for request-response, asynchronous (Inngest events) for event-driven integration
- Define API contracts between services: request/response schemas, error handling, pagination, rate limiting
- Design data ownership per service: each service owns its database collections, accessed only through its API
- Design shared infrastructure: shared `@repo/types` as published npm package, shared monitoring/observability, shared CI/CD templates
- Create phased migration roadmap: Phase 0 (preparation — shared types, API contracts), Phase 1 (AI Gateway extraction), Phase 2 (Auth Service extraction), Phase 3 (MCP Gateway extraction), Phase 4 (RAG Engine extraction), Phase 5 (Billing/Marketplace extraction), Phase 6 (Chat/Connector extraction)
- Document migration risks and mitigation strategies
- Estimate effort per phase
- Define success criteria per phase (rollback criteria, performance validation)

### Non-functional requirements

- Each microservice must be independently deployable
- Service-to-service communication must have < 10ms overhead (local network)
- Event-driven integration must have at-least-once delivery guarantee
- Migration must be reversible per phase (rollback capability)

### Performance requirements

- Inter-service communication overhead: < 10ms p95
- Extracted service performance: within 5% of monolith baseline
- Migration zero-downtime: no user-facing impact during extraction

### Security requirements

- Service-to-service communication must use mTLS
- Each service must have its own authentication/authorization
- Service mesh (Istio/Linkerd) must enforce network policies
- Secrets must be per-service (not shared)

### Accessibility requirements

- No accessibility impact

## Package Boundaries

The extraction plan is a documentation deliverable. No code changes are made in this step. The plan covers all `@repo/*` packages and their potential extraction into microservices.

---

## Architecture Constraints

- Extraction must follow strangler fig pattern: new functionality is built as microservices, existing monolith functionality is migrated incrementally
- Shared types (`@repo/types`) becomes a published npm package consumed by all microservices
- Each microservice gets its own database (or database namespace) — no shared databases across services
- Inngest event bus is the backbone for asynchronous integration
- Service mesh (Istio) handles mTLS, traffic routing, and canary deployments
- API gateway (Kong/Envoy) fronts all microservices for routing, rate limiting, and authentication

## Notes

The microservice extraction plan is a strategic document that guides the platform's evolution over 6-12 months. It is not an implementation task but a roadmap for future work. The key insight from the horizontal scaling review (step 18.01) is that the monolith is well-structured for extraction — the package boundaries already align with bounded contexts. The first extraction (AI Gateway) is the lowest risk and highest value — it requires independent scaling for LLM traffic. Do not attempt all extractions at once — follow the phased approach with validation gates between each phase.
