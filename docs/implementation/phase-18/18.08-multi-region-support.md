## Goal

Implement active-active multi-region deployment with data synchronization, conflict resolution, and seamless failover. Ensure the platform operates correctly with users distributed across global regions, each region processing reads and writes independently.

---

## Business Context

Active-active multi-region deployment is the highest level of platform availability and performance. Every region serves both reads and writes independently, with data synchronized between regions. Users get sub-100ms latency from their nearest region, and the platform survives any single-region outage without downtime. This is the target architecture for enterprise customers with global operations.

---

## Technical Context

Active-active multi-region requires: (1) MongoDB Atlas multi-region cluster with primary in one region and readable secondaries in others, writes go to primary (cross-region), reads go to local replica, (2) Upstash Global Database with active-active Redis replication, (3) R2 with cross-region replication and read-from-region, (4) application instances in each region with region-aware routing. Conflict resolution is handled at the application layer using last-writer-wins with vector clocks for critical data.

## Requirements

### Functional requirements

- MongoDB Atlas multi-region cluster: primary region (US) handles writes, secondary regions (EU, APAC) serve reads from local replicas, automated failover on primary region failure
- Cross-region write routing: all writes route to the primary region, reads use local secondaries, application handles latency for cross-region writes transparently
- Upstash Global Database: active-active Redis across all regions, local reads and writes, conflict resolution via last-writer-wins
- R2 global replication: objects replicated to all regions, reads from nearest region, writes to primary region with async replication
- Application region-awareness: region cookie set by Cloudflare, application uses region context for routing decisions
- Cross-region user sessions: JWT-based sessions (stored in Redis with cross-region replication) — users can switch regions without re-authentication
- Cross-region workspace consistency: workspace data is eventually consistent across regions — critical updates (subscription changes, feature flags) use synchronous cross-region replication
- Conflict resolution: last-writer-wins with timestamp comparison for most entities, manual resolution for complex conflicts (identified and queued)
- Region-aware health checks: each region monitors its own health and reports to global health dashboard
- Region failover: automated failover when a region becomes unhealthy, traffic rerouted to next-nearest healthy region

### Non-functional requirements

- Cross-region write latency: < 300ms (US primary, writes from EU/APAC)
- Cross-region read-after-write consistency: reads from any region see the write within 2 seconds
- Active-active Redis: < 50ms cross-region replication latency
- Region failover: < 30 seconds for DNS + < 60 seconds for application warm-up

### Performance requirements

- Local region read latency: < 30ms
- Cross-region write latency (EU → US): < 200ms
- Cross-region write latency (APAC → US): < 300ms
- Read-after-write consistency: < 2 seconds

### Security requirements

- Cross-region replication must use TLS
- Data residency: EU user data must not leave EU region (use EU primary for EU-tenants)
- Write routing must respect data residency constraints
- Cross-region secrets must be unique per region

### Accessibility requirements

- No accessibility impact

## Architecture Constraints

- Writes go to MongoDB Atlas primary region (US) — secondary regions read from local replicas
- Cross-region write latency is accepted — the platform is optimized for read performance
- Sessions use JWT tokens stored in Redis with Global Database replication — no local session state
- Conflict resolution uses last-writer-wins with the region ID as tiebreaker (US region wins ties)
- Data residency: enterprise workspaces with EU data residency requirements route all writes to the EU region's write-capable replica
- Region failover: health checks detect region failure → DNS routing updated → traffic rerouted to next-nearest region

## Notes

Multi-region active-active is the most complex operational architecture. MongoDB Atlas multi-region clusters require careful planning — the write latency from secondary regions to the primary can be significant. Consider whether true active-active is necessary or if active-passive (DR) is sufficient for your use case. For this platform, the chat and RAG workloads are read-heavy and benefit greatly from active-active. Write-heavy operations (document upload, subscription changes) are less frequent and can tolerate the cross-region latency. Upstash Global Database simplifies the Redis multi-region story significantly.
