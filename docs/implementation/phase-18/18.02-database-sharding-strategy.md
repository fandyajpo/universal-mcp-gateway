## Goal

Design and implement the database sharding strategy for MongoDB Atlas. Select the shard key, configure sharding, implement the migration strategy, and verify query performance with sharded collections.

---

## Business Context

As the platform grows to millions of users and billions of documents, a single MongoDB replica set will hit capacity limits for storage, write throughput, and query performance. Sharding distributes data across multiple shards (independent replica sets) based on a shard key, enabling horizontal scaling of the database layer. The sharding strategy must maintain tenant isolation, support efficient queries, and minimize migration complexity.

---

## Technical Context

MongoDB sharding uses a shard key to distribute documents across shards. The shard key selection is critical: it must have high cardinality, support the most common query patterns, and provide even data distribution. For a multi-tenant platform, `workspaceId` is the natural shard key — it has high cardinality (many workspaces), supports tenant-scoped queries (all queries filter by `workspaceId`), and enables zone-based sharding for data locality.

## Requirements

### Functional requirements

- Select shard key for each collection: primary shard key = `workspaceId` (hashed), secondary shard key = `_id` for uniqueness
- Configure MongoDB Atlas sharded cluster (minimum 3 shards for production)
- Implement hashed sharding for even distribution (avoid range-based hot spots)
- Configure zone-based sharding for data locality (workspaces from US region on US shards)
- Verify all queries include `workspaceId` in the query filter (mandatory for targeted queries)
- Implement query routing: queries without `workspaceId` must use `$broadcast` (scatter-gather)
- Add `shardKey` fields to all Mongoose schemas
- Implement shard key validation: documents without shard key must be rejected
- Create shard-aware connection manager
- Document the sharding strategy with migration plan

### Non-functional requirements

- Shard key must be immutable after document creation
- All queries must include `workspaceId` for targeted shard access
- Hashed sharding must provide even distribution across shards
- Zone-based sharding must support data residency requirements

### Performance requirements

- Targeted query (with shard key): < 50ms p95
- Scatter-gather query (without shard key): < 500ms p95 (rare, for admin queries only)
- Shard rebalancing impact: < 5% performance degradation during rebalancing
- Write throughput: linear scaling with shard count

### Security requirements

- Shard key must not contain sensitive data
- Cross-shard queries must respect tenant isolation
- Shard zones must not allow data to move across regulatory boundaries

### Accessibility requirements

- No accessibility impact

## Architecture Constraints

- Shard key must be `workspaceId` (hashed) for all tenant-scoped collections
- Global collections (pricing plans, system config) use `_id` hashed shard key
- Shard key must be present in every document and query filter
- Hashed sharding uses MongoDB `hashed` index type on `workspaceId`
- Zone sharding aligns with the multi-region deployment (step 18.08)
- Migration to sharded cluster requires a maintenance window or online migration via MongoDB Atlas Live Migration

## Notes

Sharding is a complex database operation with significant operational implications. Engage MongoDB Atlas support for the initial sharded cluster setup. The most critical decision is the shard key — changing the shard key after data is inserted is extremely difficult. For this platform, `workspaceId` hashed is the correct choice: high cardinality, even distribution, and matches the primary query pattern. The migration to a sharded cluster can be done online using MongoDB Atlas Live Migration, but have a rollback plan. Test the sharded cluster with production-scale data before the actual migration.
