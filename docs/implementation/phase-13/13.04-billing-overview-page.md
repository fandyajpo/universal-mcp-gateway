## Goal

Build the billing overview page in the admin app that provides a read-only dashboard showing aggregate billing metrics: Monthly Recurring Revenue (MRR), churn rate, active subscriptions, failed payments, revenue by plan tier, and revenue trends over time.

---

## Business Context

The billing overview page gives platform operators and business stakeholders a real-time view of the platform's financial health. It answers critical questions: How much revenue is the platform generating? How many customers are on each plan? What is the churn rate? Are there customers with failed payments that need attention? This page does not allow modifying billing data — it is a monitoring and analysis tool only.

---

## Technical Context

This step depends on step 14.03 (Subscription Management) which provides the subscription data structures and query methods. The page is read-only — billing mutations happen through the billing portal (step 14.08) and Stripe webhooks (step 14.07). Data is aggregated from the subscriptions and usage records collections in MongoDB. The aggregation pipeline should be optimized through MongoDB aggregation framework with pre-computed materialized views for time-series metrics.

## Requirements

### Functional requirements

- Display key metric cards: MRR (current), Churn Rate (30-day), Active Subscriptions, Failed Payments (7-day), Average Revenue Per User (ARPU)
- Show revenue breakdown by plan tier (Free, Pro, Business, Enterprise) with counts
- Display a revenue trend chart (last 12 months) with monthly MRR
- Show subscription status distribution: active, past_due, canceled, trialing
- Display a list of recently failed payments with workspace name, amount, date, and failure reason
- Show top 10 workspaces by MRR
- Support date range selector for all metrics (7d, 30d, 90d, 1y, custom)
- Auto-refresh metrics every 5 minutes with a manual refresh button

### Non-functional requirements

- All data is read-only — no mutation endpoints exposed
- Metrics must be computed server-side, not in the browser
- Trend data may be stale by up to 15 minutes (acceptable for analytics)
- Page must render meaningful content even when some metrics are unavailable

### Performance requirements

- Initial page load: < 3s (aggregation queries on potentially large collections)
- Date range change: < 2s
- Refresh: < 3s

### Security requirements

- Only users with `admin:billing` permission can access this page
- Financial data must not be exposed in client-side bundle
- API responses must use signed responses to prevent tampering

### Accessibility requirements

- Chart data must be available in tabular form for screen readers
- Metric cards must use semantic HTML
- Color-coded metrics must include text labels

## Architecture Constraints

- Aggregation queries should use MongoDB's `$lookup` and `$group` pipeline stages — no application-level aggregation
- For performance, create a materialized view collection `billing_daily_snapshots` that is updated by a cron job via Inngest
- The metric card component must handle loading, error, and empty states explicitly
- Failed payments data comes from Stripe's webhook events stored in the `payment_events` collection

## Notes

The billing overview page is intentionally read-only. All billing mutations (plan changes, cancellations, refunds) must go through the Stripe dashboard or the billing portal. The materialized view approach (`billing_daily_snapshots`) is critical for performance — the aggregation queries run against the full subscription collection can be expensive on large datasets. Schedule the materialized view update via Inngest cron: `0 */6 * * *` (every 6 hours).
