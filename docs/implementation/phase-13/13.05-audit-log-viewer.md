## Goal

Build the audit log viewer in the admin app that allows platform operators to browse, search, and filter all platform audit events. The viewer must support filtering by entity type, action, user, date range, and workspace, with real-time event streaming and CSV export.

---

## Business Context

Audit logs are the immutable record of all security-relevant events on the platform. They are essential for security investigations, compliance reporting (SOC 2, GDPR), customer support (determining what happened and when), and internal monitoring. The audit log viewer must make it easy for operators to find specific events across potentially millions of log entries. Audit logs have legal significance — they must be tamper-proof and retain their integrity.

---

## Technical Context

Audit logs are stored in a dedicated `audit_logs` collection in MongoDB (created in step 00.09 as part of the repositories). The collection uses an append-only pattern — documents are never updated or deleted. Each log entry contains: `id`, `tenantId`, `userId` (who performed the action), `entityType` (what was acted upon), `entityId`, `action` (what was done), `details` (JSON blob with action-specific data), `ip` (request IP), `userAgent`, and `timestamp`.

The viewer must handle large collections efficiently using MongoDB indexes on `timestamp`, `entityType`, `action`, `userId`, and `tenantId`. The default view should show the last 7 days of events.

## Requirements

### Functional requirements

- Display paginated table of audit events with columns: timestamp, user (name + email), entity type, action, entity identifier, summary
- Support filtering by entity type (user, workspace, subscription, api_key, connector, tool, document), action (created, updated, deleted, suspended, exported, login, logout, failed_login), user (searchable dropdown), date range (calendar picker with presets), and workspace (searchable dropdown)
- Provide a detail view for each event showing the full JSON details blob
- Support full-text search across event details
- Show real-time event stream (new events appear at the top via SSE)
- Support CSV export of filtered results (limited to 10k rows)
- Display event count for current filter set
- Show daily event volume sparkline chart

### Non-functional requirements

- Logs are append-only — no delete or edit UI or API
- Default view: last 7 days, newest first
- CSV export runs asynchronously for large result sets
- Events must include a cryptographic hash chain for tamper detection

### Performance requirements

- Initial load (7 days): < 2s
- Filtered query: < 1s with appropriate indexes
- CSV export: < 30s for 10k rows
- Real-time events: < 1s latency from write to display

### Security requirements

- Only users with `admin:audit` permission can access the viewer
- CSV export must be restricted to 10k rows regardless of filter results
- Audit log integrity hash must be verified on each query
- Event details must not include sensitive data (passwords, tokens, API keys)
- Export URL must expire after 1 hour

### Accessibility requirements

- Event details panel must be keyboard accessible
- Real-time events must be announced to screen readers
- Color-coded action types must include text labels

## Notes

The audit log integrity hash is implemented as a SHA-256 chain where each event's hash includes the previous event's hash. This allows detection of tampering — if an event in the middle is modified, all subsequent hashes will not match. Display a green "Integrity Verified" badge when the chain is valid, and a red "Integrity Breach Detected" badge if any hash is invalid. The hash verification should happen at query time and may be computationally expensive — consider caching the chain tip hash.
