## Goal

Implement an admin authentication guard that restricts access to all admin routes to users with the `admin:access` permission. The guard must verify session validity, check role elevation, log access attempts, and provide clear error responses for unauthorized access.

---

## Business Context

The admin dashboard is the control center for platform operators. It exposes sensitive operations: user suspension, billing data access, audit log viewing, and system configuration. Every request to the admin app must be authenticated and authorized at the edge. Unauthorized access attempts must be logged for security auditing. This guard is the foundational security layer that all subsequent admin features build upon.

---

## Technical Context

The admin app (`apps/admin`) is a Next.js 15 application. It shares the same Better Auth session system as `apps/web` but requires an elevated permission check. The RBAC framework from Phase 2 (`@repo/auth`) provides `requirePermission()` helpers. The guard must work at multiple layers: Edge Middleware for route protection, Server Component level for data access, and API route level for mutations.

The existing auth middleware from step 02.07 validates sessions generically — it checks that the user is authenticated but does not enforce admin-specific permissions. Step 13.01 builds on top of that by adding the `admin:access` permission check. Users who attempt to access admin routes without the proper role should receive a 403 response, and the attempt must be recorded in the audit log.

## Requirements

### Functional requirements

- Must intercept all requests to `/admin/*` routes
- Must validate the session token on every request
- Must check that the authenticated user has `admin:access` permission
- Must return 401 for unauthenticated requests (no valid session)
- Must return 403 for authenticated requests lacking admin permission
- Must log all denied access attempts to the audit log with user ID, IP, and timestamp
- Must allow a configurable allowlist of public admin routes (e.g., admin login page)
- Must support silent session refresh for expired tokens
- Must expose a `requireAdmin()` helper for Server Components and Route Handlers

### Non-functional requirements

- Guard must execute in Edge Runtime — no database calls on the hot path for session validation
- Session validation must complete in under 50ms p95
- Denied access log writes must be asynchronous (fire-and-forget)
- Guard must not add measurable overhead to authorized requests

### Performance requirements

- Session validation: < 10ms (Redis read)
- Permission check: < 5ms (in-memory)
- Total guard overhead: < 25ms per request

### Security requirements

- Log all denied attempts with full request context
- Do not reveal why access was denied in the response body
- Set `Cache-Control: no-store` on all admin responses
- Validate that the session token has not been tampered with (cryptographic verification)

### Accessibility requirements

- Error pages must use semantic HTML and ARIA landmarks
- 401 and 403 pages must be navigable by keyboard

## Package Boundaries

Admin auth guard must only depend on `@repo/auth` (session validation, RBAC), `@repo/cache` (Redis session lookup), and `@repo/logger` (audit logging). It must NOT import from `@repo/database` directly — audit log writes go through the repository layer.

---

## Architecture Constraints

- Middleware must use Edge Runtime compatible code (no Node.js-specific modules)
- Session validation must be stateless — no session state stored in memory
- Audit log writes must use the `AuditLogRepository` from `@repo/database`
- The guard must not modify the request or response unless denying access
- Admin routes must be explicitly listed as protected — no implicit protection

## Notes

The guard must handle the edge case where a user's session is valid but their role has been changed since the session was created. This requires a role check against the database rather than relying solely on the session claim. Use a Redis cache with a 60-second TTL for role lookups to avoid a database hit on every request. For the first implementation, the session claim check is sufficient — the database check can be added in a follow-up optimization.
