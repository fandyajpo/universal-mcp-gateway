## Goal

Build the announcements system that allows platform operators to create, schedule, target, and manage in-app announcements. The system includes an admin creation UI and a client-side display component in the web app.

---

## Business Context

Announcements are the primary way platform operators communicate with users about new features, scheduled maintenance, policy changes, and important updates. The system must support targeted announcements (specific users, workspaces, or plan tiers), scheduled publishing, expiration, and dismissible display. This replaces ad-hoc communication methods and ensures users see important messages at the right time.

---

## Technical Context

Announcements are stored in an `announcements` collection in MongoDB. Each announcement has: `title`, `body` (markdown), `type` (info, warning, critical, feature), `targeting` (all users, by plan tier, by workspace, by user role), `scheduledAt`, `expiresAt`, `isDismissible`, `createdBy`, `createdAt`, `updatedAt`. The client-side component polls or receives announcements via an API route that respects targeting and dismissal state.

Dismissed announcements are tracked per-user in a `dismissed_announcements` collection or as a sub-array on the user document. The display component appears as a banner at the top of the web app or as a toast notification depending on the announcement type.

## Requirements

### Functional requirements

- Admin UI to create announcements with title, body (markdown editor), type, and targeting configuration
- Schedule announcements for future publishing with timezone support
- Set automatic expiration date/time
- Target announcements to: all users, users on specific plan tiers (free/pro/enterprise), users in specific workspaces, users with specific roles (admin/member/viewer)
- Preview announcement as users will see it
- Edit draft announcements before publishing
- List all announcements with status (draft, scheduled, active, expired), type, targeting summary, and creation date
- Cancel or delete scheduled announcements
- Client-side display component in web app: banners for critical/warning, toasts for info/feature announcements
- Dismissible announcements with per-user dismissal tracking
- Mark announcement as read for read receipts tracking

### Non-functional requirements

- Announcement polling: every 5 minutes, or push via SSE for instant delivery
- Dismissal state persists across sessions
- Expired announcements are automatically hidden
- Only active announcements (current time between scheduledAt and expiresAt) are returned by the API
- Announcement body supports limited markdown (headings, links, bold, italic, lists)

### Performance requirements

- Announcement API response: < 50ms (cached with 5-minute TTL)
- Client-side display: zero layout shift on announcement load
- Banner animation: < 300ms

### Security requirements

- Announcement creation requires `admin:access` permission
- Announcement body must be sanitized to prevent XSS (strip script tags, event handlers)
- Targeting logic must not reveal the existence of other users or workspaces
- Dismissal API must verify user authentication

### Accessibility requirements

- Announcement banners must be focusable and announced by screen readers
- Dismiss button must have accessible label
- Color-coded types must include icon and text labels
- Markdown content must maintain heading hierarchy

## Architecture Constraints

- Announcements collection must have a TTL index on `expiresAt` for automatic cleanup
- Targeting must be evaluated server-side — the API returns announcements filtered by user context
- Dismissal is stored per-user in a separate `dismissed_announcements` collection with `userId + announcementId` unique compound index
- Announcement body markdown must be sanitized using DOMPurify on the client
- Active announcement query: `scheduledAt <= now() AND expiresAt > now() AND enabled = true`
