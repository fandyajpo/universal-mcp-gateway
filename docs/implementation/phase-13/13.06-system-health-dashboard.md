## Goal

Build the system health dashboard in the admin app that displays real-time status of all platform services, uptime history, incident records, and resource utilization metrics.

---

## Business Context

Platform operators need at-a-glance visibility into the health of all system components. When a customer reports an issue, the health dashboard is the first place operators look to determine if there is a platform-wide problem. The dashboard must show the status of MongoDB, Redis, R2, OpenRouter, Better Auth, Inngest, Sentry, and the Next.js applications themselves. It must also track historical uptime and record incidents for SLA reporting.

---

## Technical Context

The health dashboard aggregates data from multiple sources: health check endpoints on each service, monitoring probes that run on a schedule via Inngest, and incident records stored in MongoDB. Health checks are implemented as lightweight HTTP requests or database ping commands that verify connectivity and basic functionality. The dashboard itself is a client-side React application that polls a health API route every 30 seconds.

## Requirements

### Functional requirements

- Display service status cards for each platform service with: name, status (healthy/degraded/down), response time (ms), last checked timestamp, and uptime percentage (30-day).
- Color-code status: green (healthy), yellow (degraded), red (down).
- Show detailed status page for each service with response time history chart, error rate, and recent incidents.
- Display uptime history as a 90-day calendar heatmap.
- List recent incidents with severity (critical, major, minor), duration, affected services, and resolution notes.
- Show resource utilization: API request rate (req/s), average response time, error rate (5xx percentage), and active connections.
- Provide a manual "Run Health Check" button that triggers an immediate check of all services.
- Show Last Incident, Total Incidents (30d), Current Uptime (30d), and Average Response Time as summary metric cards.

### Non-functional requirements

- Health checks must not depend on the services they are checking (no cascading failures)
- Dashboard must render even when some services are down
- Service status is cached for 30 seconds to prevent thundering herd
- Health check probes run from the application — do not require external monitoring infrastructure

### Performance requirements

- Dashboard initial load: < 2s
- Status polling: < 500ms per check
- Manual health check: complete within 5s for all services

### Security requirements

- Health endpoints must be admin-only
- Health responses must not expose internal connection details
- Only aggregate response times — never individual query details

### Accessibility requirements

- Status indicators must use icons and text labels (not color alone)
- Incident list must be navigable by keyboard
- Charts must include aria-labels

## Package Boundaries

A new `@repo/health` package should be created for the health check service and probes. This keeps health check logic separate from the admin UI. The admin app depends on `@repo/health`, `@repo/database`, `@repo/cache`, `@repo/ui`, and `@repo/auth`. The health package depends on `@repo/logger` and `@repo/config`.

---

## Architecture Constraints

- Health checks must be idempotent — running them multiple times has no side effects
- Each service probe implements a `Probe` interface: `check(): Promise<ProbeResult>` where `ProbeResult` includes `status`, `responseTime`, `error?`, and `timestamp`
- Health check results are stored in a MongoDB collection `health_check_results` with a TTL of 7 days
- The Inngest cron job runs every 60 seconds and writes results to the collection
- Dashboard reads from the collection — it does not trigger checks on read

Create the `@repo/health` package in this step. It will contain the `HealthChecker` class, service probes, and result types. The probes should use a timeout of 5 seconds per service. Consider adding a webhook notification integration for critical status changes (optional — can be added in Phase 16).
