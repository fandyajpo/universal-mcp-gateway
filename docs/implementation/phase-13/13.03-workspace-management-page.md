## Goal

Build the workspace management page in the admin app that allows platform operators to list, search, filter, view details, suspend, and trigger data export for all workspaces on the platform.

---

## Business Context

Platform operators need visibility into all workspaces to monitor usage, enforce terms of service, and provide customer support. The workspace management page is the central place to investigate workspace-level issues, manage workspace lifecycle, and perform data exports when customers request their data. This page complements the user management page (13.02) to give operators full coverage of the platform's tenant landscape.

---

## Technical Context

This page uses the `WorkspaceRepository` from step 03.02 which provides tenant-scoped workspace CRUD. However, since this is an admin view, it must bypass the automatic tenant scoping to see across all workspaces. A separate `AdminWorkspaceRepository` or an admin-specific method on the existing repository will be needed. The pattern established in step 13.02 (data table, filtering, detail panel) should be reused for consistency.

Workspace data export requires asynchronous processing through Inngest (step 00.12) to handle potentially large datasets. The export includes workspace settings, member list, document metadata, chat history, and connector configurations. Exported data is packaged as a ZIP file and stored in R2 with a signed URL for download.

## Requirements

### Functional requirements

- Display a paginated table of all workspaces with columns: name, slug, owner, member count, plan tier, storage used, status (active/suspended), created date
- Support server-side search by name or slug
- Support filtering by plan tier (free/pro/enterprise), status, and date range
- Provide a workspace detail panel showing settings, member roster, usage metrics, billing info, and recent activity
- Implement workspace suspension with confirmation and reason
- Implement workspace data export with progress tracking and email notification on completion
- Support bulk selection for batch operations (suspend, export metadata)
- Show aggregate workspace metrics: total workspaces, active today, avg members per workspace

### Non-functional requirements

- Admin repository methods must explicitly bypass tenant scoping
- Workspace detail panel must lazy-load member list for large workspaces (500+ members)
- Data export must not block the admin UI — triggered asynchronously via Inngest
- Export progress must be polled or pushed via SSE

### Performance requirements

- Workspace list load: < 2s for 10k workspaces
- Search response: < 500ms for 10k workspaces
- Detail panel initial load: < 1s (core info), member list loads after

### Security requirements

- Suspended workspaces must immediately block all API access for all members
- Workspace export must respect data isolation — only include data from the target workspace
- Export download URL must expire after 24 hours
- Only super-admins can trigger workspace deletion (not just suspension)

### Accessibility requirements

- Same as step 13.02 — keyboard navigable table, accessible controls

## Package Boundaries

Workspace management depends on `@repo/database` (workspace repository), `@repo/auth` (admin permission check), `@repo/storage` (export file storage), `@repo/inngest` (async export job), `@repo/ui` (shared components), and `@repo/validation` (input schemas). The export Inngest function must live in a package that has access to all data repositories needed for the export.

---

## Architecture Constraints

- Workspace suspension must cascade: set `suspendedAt` on workspace, invalidate all member sessions, reject all API requests with 403
- Export must be implemented as a multi-step Inngest function with progress events
- Export ZIP structure: `{workspace-slug}/settings.json`, `members.csv`, `documents.json`, `chat-history.json`, `connectors.json`
- Admin workspace repository must extend BaseRepository with an explicit `adminBypassTenant` flag

## Notes

The admin workspace repository should use the same MongoDB collection as the regular workspace repository but without the `tenantId` filter. This is a deliberate architecture violation of the multi-tenant pattern — it is acceptable because admins operate outside the tenant model. The export job is the first Inngest function that needs access to multiple repositories — design it carefully to avoid dependency issues.
