## Goal

Build the feature flags management page in the admin app that allows platform operators to create, toggle, configure, and monitor feature flags with percentage-based rollout and user/workspace targeting.

---

## Business Context

Feature flags enable safe, gradual rollouts of new functionality. They allow the team to release features to a subset of users or workspaces before full deployment, enabling canary testing, A/B experimentation, and emergency kill-switches. The feature flags UI gives operators and product managers self-service control over feature releases without requiring code deployments. Every flag change is logged for audit.

---

## Technical Context

Feature flags are stored in a `feature_flags` collection in MongoDB and cached in Redis for low-latency checks. Each flag has: `key` (unique identifier), `name`, `description`, `enabled` (global toggle), `rolloutPercentage` (0-100), `targetingRules` (array of user/workspace/group matchers), `environmentOverrides` (dev, staging, production), `owner`, `createdAt`, `updatedAt`. The feature flag evaluation logic lives in `@repo/config` or a new `@repo/feature-flags` package and is used by application code to conditionally enable features.

The evaluation order is: explicit user override > explicit workspace override > group membership > percentage rollout > global enabled flag.

## Requirements

### Functional requirements

- Display a list of all feature flags with: key, name, status (on/off/partial), rollout percentage, environment overrides, last modified, owner
- Create new feature flags with key, name, description, default state
- Toggle individual flags on/off globally
- Configure rollout percentage (0-100%) with a slider
- Add targeting rules: specific users (email/ID), specific workspaces (name/ID), user attributes (plan tier, role, creation date), workspace attributes (plan tier, size, region)
- Add environment overrides (force on/off for dev/staging/production)
- View flag evaluation for a specific user or workspace (test evaluation)
- Display change history for each flag (who changed what and when)
- Search and filter flags by name, key, status, owner

### Non-functional requirements

- Flag evaluation logic must be efficient enough to run on every request
- Redis cache must be invalidated on flag changes
- Admin UI must show stale cache indicator if Redis is out of sync
- Flag changes must propagate within 60 seconds (eventual consistency)

### Performance requirements

- Flag evaluation: < 5ms (Redis read, in-memory for hot path)
- Admin page load: < 1s
- Flag change propagation: < 60s

### Security requirements

- Only users with `admin:access` can manage feature flags
- Critical flags (auth, billing, security) must require a second confirmation
- All flag changes must be logged to the audit trail
- Flag keys must follow a naming convention: `scope.feature_name` (e.g., `chat.streaming_v2`)

### Accessibility requirements

- Switch controls must have accessible labels
- Rollout slider must be keyboard adjustable
- Color indicators must include text labels

## Package Boundaries

A new `@repo/feature-flags` package provides the evaluation engine and Redis client. The admin app depends on `@repo/feature-flags`, `@repo/database` (flag storage), `@repo/auth` (permission check), `@repo/ui` (Switch, Slider, Dialog, Select, Input), and `@repo/validation` (flag schema). The feature flags package depends on `@repo/cache` (Redis) and `@repo/logger`.

---

## Architecture Constraints

- Flag keys must use reverse-domain notation: `com.umg.chat.streaming_v2`
- Targeting rules are evaluated in order — first match wins
- Redis cache key pattern: `feature_flag:{key}` with TTL of 60 seconds
- On flag update, immediately invalidate the Redis cache entry
- The `evaluator.evaluate(key, context)` function is the single entry point for all flag checks
- Evaluation context includes: `userId`, `workspaceId`, `planTier`, `userRole`, `environment`

## Notes

The feature flags system is a powerful tool but can introduce complexity. Establish a naming convention early and enforce it with validation. Critical flags (those affecting auth, billing, or security) should be marked with `isCritical: true` which triggers additional confirmation in the UI and slower rollout requirements (max 10% per day increase). The evaluation tester is invaluable for debugging — build it early.
