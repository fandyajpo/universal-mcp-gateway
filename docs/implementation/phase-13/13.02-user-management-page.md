## Goal

Build the user management page in the admin app that allows platform operators to list, search, filter, view details, suspend, delete, and impersonate users. The page must handle large user populations efficiently and provide bulk action capabilities.

---

## Business Context

Platform operators need a centralized view of all users across all workspaces. This page is the primary tool for customer support, abuse prevention, and account management. Operators must be able to find specific users quickly, understand their activity and billing status, and take corrective actions like suspension or impersonation for troubleshooting. The page must handle thousands of users without performance degradation.

---

## Technical Context

The user management page uses the `UserRepository` from `@repo/database` (step 00.09) which provides `findAll`, `findById`, `search`, `update`, and `softDelete` methods. The admin app uses shadcn/ui components, TanStack Query for server state, and TailwindCSS for styling. The data table pattern established in Phase 3 (workspace member list) serves as the UI template.

Impersonation is a sensitive feature. When an admin impersonates a user, a special session token must be created that logs every action. The impersonation session must be clearly distinguishable in the UI with a visible banner. Impersonation must be logged to the audit trail.

## Requirements

### Functional requirements

- Display a paginated table of all users with columns: name, email, role, status (active/suspended), workspace count, last login, created date
- Support server-side search by name or email
- Support filtering by status (active, suspended, all), role (admin, member, viewer), and date range
- Provide a user detail panel (slide-over or modal) showing full user profile, workspace memberships, recent activity, billing history, and API keys
- Implement user suspension with confirmation dialog and reason input
- Implement user soft-delete with cascading workspace membership cleanup
- Implement admin impersonation with prominent visual indicator and full audit logging
- Support bulk selection for batch operations (suspend, delete, export)
- Provide CSV export of user list with selected filters
- Show real-time user count and active user metrics

### Non-functional requirements

- Table must render 100+ rows without virtual scrolling performance issues
- Search must debounce at 300ms and cancel in-flight requests
- API responses must paginate with cursor-based or offset-based pagination
- Suspension/delete actions must be immediately reflected in the table
- Impersonation session must auto-terminate after 15 minutes of inactivity

### Performance requirements

- Initial page load: < 2s to first paint
- Search response: < 500ms for 10k users
- Pagination: < 200ms per page transition
- Bulk actions: < 5s for 100 users

### Security requirements

- Impersonation requires a second confirmation dialog with reason
- All suspension/deletion actions must be logged with admin ID, target user ID, timestamp, and reason
- Suspended users must be immediately unable to authenticate
- Impersonation actions must be logged with a special `impersonation:` prefix in the audit event type
- Soft-deleted users must be restorable within 30 days

### Accessibility requirements

- Data table must be fully keyboard navigable
- Action buttons must have accessible labels
- Status indicators must use both color and text (not color alone)
- Sort controls must announce current sort direction to screen readers

## Package Boundaries

The user management page depends on `@repo/database` (UserRepository), `@repo/auth` (impersonation session creation), `@repo/validation` (input schemas for actions), `@repo/ui` (DataTable, Dialog, Button components), and `@repo/logger` (audit logging). It must not call the database directly — all data access goes through repositories.

---

## Architecture Constraints

- Impersonation must create a separate session that does not replace the admin's own session
- User deletion is soft — the record remains in the database with `deletedAt` set
- Suspension sets a `suspendedAt` and `suspensionReason` field on the user record
- Bulk actions must process in batches of 50 with progress indication
- All mutations must validate input Zod schemas before processing
